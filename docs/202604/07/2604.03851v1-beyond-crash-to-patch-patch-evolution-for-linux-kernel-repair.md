## 摘要
Linux 内核漏洞修复通常被视为从崩溃报告到代码补丁的直接映射。然而在实践中，内核修复往往在邮件列表上经历多次迭代修订，接受者的反馈会影响修复的正确性、并发处理以及 API 合规性。这一迭代优化过程蕴含了宝贵的修复知识，而现有的自动化方法往往忽略了这一点。我们开展了一项大规模的内核补丁演化研究，重建了 6946 个与 syzbot 相关的漏洞修复生命周期，连接了崩溃报告、重现器、邮件列表讨论、修订历史以及最终合并的修复。分析结果证实，最终被接受的修复往往涉及非局部修改，并且受审阅者强制约束的影响，这些约束在漏洞报告中并未体现。基于这些洞察，我们开发了 PatchAdvisor——一个将基于检索的记忆模块与经过微调的诊断顾问相结合的修复框架，用以引导代码生成代理产出符合审阅者要求的补丁。在对时间上留出测试的 syzbot 案例进行评估时，我们发现利用补丁演化历史能在审阅者一致性信号和端到端修复质量两方面均显著优于无引导和仅检索的基线方法。

## Abstract
Linux kernel bug repair is typically approached as a direct mapping from crash reports to code patches. In practice, however, kernel fixes undergo iterative revision on mailing lists before acceptance, with reviewer feedback shaping correctness, concurrency handling, and API compliance. This iterative refinement process encodes valuable repair knowledge that existing automated approaches overlook.   We present a large-scale study of kernel patch evolution, reconstructing 6946 syzbot-linked bug-fix lifecycles that connect crash reports, reproducers, mailing-list discussions, revision histories, and merged fixes. Our analysis confirms that accepted repairs are frequently non-local and governed by reviewer-enforced constraints not present in bug reports. Building on these insights, we develop PatchAdvisor, a repair framework that integrates retrieval-based memory with a fine-tuned diagnostic advisor to guide a coding agent toward reviewer-aligned patches. Evaluation on temporally held-out syzbot cases demonstrates that leveraging patch-evolution history yields measurable gains in both reviewer-aligned refinement signals and end-to-end repair quality compared to unguided and retrieval-only baselines.

## 1. 核心问题与研究背景

- **研究动机**：  
  当前自动化 Linux 内核修复主要被视为“从崩溃报告到代码补丁”的单步映射。然而在现实开发中，补丁经常在邮件列表（LKML）上经历多次修订，开发者与评审者的反复反馈才形成最终可被合并的“正确”补丁。  
  现有自动程序修复（APR）和大模型修复方法忽略了这种**迭代演化过程**，因此生成的补丁常无法满足维护者的隐含约束（如锁定规则、API 一致性、并发与回滚安全性），被拒绝或引入新的隐患。

- **问题本质**：  
  自动修复系统仅依据崩溃报告或最终提交记录，缺乏关于“为什么早期补丁被拒”“评审者如何指导修订”的上下文监督信号，导致模型难以学会“可被接受的修复”。

- **研究目标**：  
  作者希望系统化地重建**Linux 内核补丁演化的全生命周期**，并提出一种结合修复演化历史的系统性自动修复架构——**PatchAdvisor**，以提高自动生成补丁的可接受性和质量。

## 2. 方法论

### 2.1 核心思想

- 将 Linux 内核补丁视作一个**演化过程（evolutionary process）**而非单步修复。
- 将此过程封装为可学习、可检索的知识，以指导 LLM 修复模型生成更符合评审者要求的补丁。
- 提出包含两大核心组件的系统：
  1. **检索记忆模块（Evolution Memory）**：从历史补丁演化中提取可复用模式和评审反馈；
  2. **诊断顾问模块（Diagnostic Advisor）**：基于轻量微调模型，生成面向修复方向与约束的诊断总结，辅助代码生成代理（coder）进行有约束的修补。

### 2.2 技术细节与流程

整个流程（见论文 Figure 2）分四阶段：

1. **数据收集与整合**  
   - 从多个公开源（syzbot API、git.kernel.org、lore.kernel.org、patchwork.kernel.org）整合补丁链路；
   - 对 6946 个 syzbot 报告的 bug 构建完整生命周期实例。

2. **分析与知识抽取**  
   - 设计 12 个分析器（analyzers），识别 bug 类型、修复模式、局部性、复杂度、评审反馈等；
   - 构建 bug–revision–review 对齐结构，为学习提供高质量标签。

3. **多层记忆系统**  
   - **实例记忆**：存储每个 bug 的描述、修复模式、版本变更与评审意见；
   - **模式记忆**：汇总常见 bug–fix 模式及评审经验；
   - **嵌入记忆**：通过向量化检索寻找语义相似的历史修复。

4. **微调诊断顾问模型**  
   - 输入内容：崩溃报告、候选补丁、检索上下文；
   - 训练目标：生成“修复方向、约束、策略、注意事项”的诊断总结；
   - 优化目标为条件语言模型概率最小化：  
     \(L_{advisor} = -\sum_k \log P(z_k | X, z_{<k})\)。

5. **修复生成阶段**  
   - 使用通用代码生成模型（如 Claude 或 LLM coder），在接收诊断顾问输出后生成候选补丁；
   - 采用“补丁编译反馈循环”自动检测并修订无法编译的候选修复。

## 3. 实验设计

### 3.1 数据与场景

- **核心数据集**：  
  - 以 syzbot 平台为锚点，构建 **6946 个已修复内核漏洞的全生命周期样本**；
  - 包含崩溃报告、C/syz reproducer、修订邮件、版本历史、最终补丁；
  - 覆盖约 5000 个具备完整讨论的实例；
  - 数据量约 **13 GB（11GB JSON + 1.5GB 原始邮件 + 210MB 训练数据）**。

- **分析范围**：  
  Bug 分类（20类）、修复模式（10类）、修复局部性、复杂度分级、版本迭代原因等。

### 3.2 评估基准与对比方法

1. **诊断质量评估**（100个时间留出样本）  
   - 衡量模型是否能正确预测评审反馈类别（12类）。  
   - **指标**：Precision、Recall、F1。  
   - **对比**：无引导（baseline） vs. 仅记忆检索（retrieval-only） vs. 诊断顾问指引。

2. **端到端修复评估**（6个新修复实例）  
   - 选择 2026 年 2 月后、非训练集的 syzbot bugs。  
   - **条件**：具有 reproducer、≥2 次修订历史、时间独立。  
   - **对比配置**：
     - Baseline（仅崩溃报告）
     - Memory-guided（使用记忆检索）
     - Advisor-guided（加入诊断顾问）
     - Base/FT 两种版本（Gemma-3-12B, LLaMA-3.2-11B）

- **指标**：
  - Diagnostic Coverage Rate (DCR)
  - CodeBERTScore（语义相似度）
  - 编译成功率
  - AI 质量评审平均排名（由 Claude 评审）

- **环境验证**：  
  在真实内核环境（Ubuntu 22.04 + QEMU 6.2.0，Intel Xeon 80核 125GB RAM）通过600秒运行 reproducer 确认可修复性。

## 6. 主要结论与发现

1. **内核修复高度迭代和协作化：**  
   78.3% 的补丁版本迭代存在实质性评审反馈，多数修改并非自我修订。

2. **被接受补丁往往非本地：**  
   仅 3.1% 的修复发生在与崩溃函数相同的位置，显示症状与根因的显著脱节。

3. **最终补丁与修复难度不成比例：**  
   很多“微小”补丁背后是长达数百天、数轮讨论的复杂诊断过程。

4. **评审反馈揭示隐性约束：**  
   包括并发、API使用、内存安全、性能等，全为修复成功的关键因素。

5. **PatchAdvisor 的效果：**
   - 诊断阶段：F1 提升约 **21.8%**；
   - 端到端修复：Gemma-3-12B（微调版）**5/6 成功修复**，CodeBERTScore 提升至 0.91；
   - 对比基线 (1/6) 和仅检索 (4/6)，显著优越。
