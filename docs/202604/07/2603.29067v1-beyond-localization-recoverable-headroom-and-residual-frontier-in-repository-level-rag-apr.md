## 摘要
在仓库级自动程序修复（APR）研究中，越来越多的工作将更强的定位能力视为实现更佳修复的主要途径。本文提出一个更有针对性的问题：一旦定位能力得到强化，哪些定位后的杠杆仍能带来可恢复的改进，哪些在我们的协议范围内受限，以及还存在哪些残余前沿？我们在 SWE-bench Lite 上针对三种具有代表性的仓库级 RAG-APR 范式——Agentless、KGCompass 和 ExpeRepair——探讨这一问题。我们设计的实验协议结合了 Oracle Localization、池内 Best-of-K、固定接口的附加上下文探针（具有每种条件下相同 token 的填充控制）以及同仓库的强负样本，并进行了统一封装的 oracle 检查。Oracle Localization 可提升三个系统的性能，但 Oracle 成功率仍低于 50%。在抽样的 10 个补丁池中，额外的候选多样性仍然有助于改进，但这种提升很快趋于饱和。在两个固定接口下，大多数信息量丰富的附加上下文条件仍然优于其各自匹配的控制条件。统一封装检查显示，不同系统对该操作的响应不同：在统一封装下，KGCompass 和 ExpeRepair 的提升仍较为显著，而 Agentless 的变化更多取决于构建器的选择。提示级融合仍留存较大的残余前沿：最佳固定探针仅在原生三系统 Solved@10 并集之外多解决 6 个实例。总体而言，更强的定位、受限的搜索、证据质量与接口设计共同决定了仓库级修复的结果。

## Abstract
Repository-level automated program repair (APR) increasingly treats stronger localization as the main path to better repair. We ask a more targeted question: once localization is strengthened, which post-localization levers still provide recoverable gains, which are bounded within our protocol, and what residual frontier remains? We study this question on SWE-bench Lite with three representative repository-level RAG-APR paradigms, Agentless, KGCompass, and ExpeRepair. Our protocol combines Oracle Localization, within-pool Best-of-K, fixed-interface added context probes with per-condition same-token filler controls and same-repository hard negatives, and a common-wrapper oracle check. Oracle Localization improves all three systems, but Oracle success still stays below 50%. Extra candidate diversity still helps inside the sampled 10-patch pools, but that headroom saturates quickly. Under the two fixed interfaces, most informative added context conditions still outperform their own matched controls. The common-wrapper check shows different system responses: under a common wrapper, gains remain large for KGCompass and ExpeRepair, while Agentless changes more with builder choice. Prompt-level fusion still leaves a large residual frontier: the best fixed probe adds only 6 solved instances beyond the native three-system Solved@10 union. Overall, stronger localization, bounded search, evidence quality, and interface design all shape repository-level repair outcomes.

## 1. 核心问题与整体含义

### 研究动机
- **背景**：在仓库级自动程序修复（Repository-Level APR）中，研究者普遍认为“更好的定位（Localization）”是提高修复成功率的主要途径。  
- **问题**：然而，当定位能力已显著增强后，后续环节（如上下文构造、修复生成、验证等）是否仍有“可恢复的提升空间（recoverable headroom）”？提升定位后遗留的“残余前沿（residual frontier）”究竟有多大？
- **意义**：该问题关乎 APR 系统未来优化的方向——即继续改善定位，还是改进上下文利用、搜索策略或接口设计。

## 2. 方法论与研究设计

### 核心思想
论文提出一种**控制性后定位实验框架（post-localization controlled study）**，旨在定量评估定位之后的改进空间和残余难点。

### 关键技术与步骤
包含四个主要干预模块：
1. **Oracle Localization（预言定位）**  
   - 利用真实修复差异（gold diff）提取故障文件与行范围作为定位信息，插入各系统的候选集合中。  
   - 仅暴露文件与行位置信息，不泄露修复补丁内容。
   - 能评估“若定位完美，系统还会出错多少”。

2. **Best-of-K（候选采样与理想选择）**  
   - 对每个实例生成 10 个补丁样本；求其中“至少一个解决”的比例（Solved@K）。  
   - 相当于计算候选空间内部的可恢复上限（within-pool upper bound）。

3. **Fixed-Interface Added Context（固定接口附加上下文）**  
   - 在保持原有 Oracle 提示结构的前提下，添加来自其他系统的“跨范式上下文”，同时设置：
     - **同 token 填充控制（same-token filler）**  
     - **同仓库强负样本控制（hard negative）**
   - 用以验证提升是否来源于内容，而非提示长度。

4. **Common-Wrapper Oracle Check（统一封装检查）**  
   - 将不同系统置于相同的输出框架中，以隔离“界面差异”带来的影响。

### 评价指标
- 主指标：**Success**（修复后全部测试通过比例）。  
- 辅助指标：Completion Rate、Solved@K、File Hit、Recall@1@5。  
- 统计方法：Wilson 置信区间、McNemar 检验、Bootstrap 置信带。

### 数据集与 Benchmark

### 对比系统

## 6. 主要结论与发现

1. **Oracle 定位提升显著但有限**  
   - 三系统成功率由 28–33% 提升到约 39–43%，但仍低于 50%。  
   - 主要提升来自更高的“完成率”与“执行成功率”，而非彻底解决新增案例。

2. **候选多样性仍有价值，但收益饱和**  
   - Best-of-10 带来约 10–13 个百分点提升，但在 K=5 时已回收 86%–87% 全部可恢复空间。  
   - 说明增采样效益递减。

3. **附加上下文确实有效**  
   - 几乎所有信息性上下文（尤其来自 KGCompass 或融合提示）都显著优于长度匹配的 filler/hard-negative。  
   - 提升来自语义结构，而非 token 数。

4. **接口设计影响敏感**  
   - Agentless 的改进依赖 prompt builder，统一封装下变化显著。  
   - KGCompass 和 ExpeRepair 则相对稳定。

5. **Prompt 级融合仍留大残余前沿**  
   - 最优固定探针仅在三系统并集之外多解决 6 个实例（共 300）。  
   - 残余前沿（jointly unsolved）仍有 101 案例，占基准 1/3 以上。

6. **残余 frontier 的特征**  
   - 集中于 sympy 与 django 两大仓库。  
   - 主要失败原因为 API 误用、修复不完全、导入缺失与断言错误。  
   - 即便组合所有探针，也仅修复 14/101（≈14%）。

## 7. 方法优点与创新点

- **体系性诊断框架**：首次在仓库级 RAG-APR 中分离“定位效果”与“后定位瓶颈”。
- **严格控制实验**：same-token filler 与 hard-negative 控制有效排除提示长度干扰。
- **多角度量化分析**：包括 paired 统计、置信区间、重采样置信带、前缀饱和曲线。
- **统一封装检验**：揭示接口实现对评测结果的偏倚影响。
- **细粒度残余前沿分析**：以仓库和错误类型两个维度刻画未解决难题。
