## 摘要
与代码补全不同，调试需要定位故障并进行针对性的修改。我们观察到，前沿的大型语言模型（LLM）在调试时常常会重新生成正确但过度修改的解决方案。为了评估 LLM 在精确调试方面的表现，我们提出了 Precise Debugging Benchmark（PDB）框架，它能够自动将任何编程数据集转换为具备精度感知评估的调试基准。PDB 通过综合经过验证的原子级错误并将其组合成多错误程序来生成有缺陷的程序。我们定义了两个新的指标：编辑级精度和错误级召回率，分别衡量执行的必要修改数量以及解决的错误数量。我们发布了两个评估基准：针对单行错误的 PDB-Single-Hard，以及针对多行错误的 PDB-Multi。实验表明，前沿模型（如 GPT-5.1-Codex 和 DeepSeek-V3.2-Thinking）在单元测试通过率超过 76% 的情况下，精度仍低于 45%，即便明确要求执行最小化调试。最后，我们展示了迭代式和代理式调试策略并未显著提高精度或召回率，这强调了重新思考编程模型后训练流程的必要性。

## Abstract
Unlike code completion, debugging requires localizing faults and applying targeted edits. We observe that frontier LLMs often regenerate correct but over-edited solutions during debugging. To evaluate how far LLMs are from precise debugging, we introduce the Precise Debugging Benchmark (PDB) framework, which automatically converts any coding dataset into a debugging benchmark with precision-aware evaluation. PDB generates buggy programs by synthesizing verified atomic bugs and composing them into multi-bug programs. We define two novel metrics, edit-level precision and bug-level recall, which measures how many necessary edits are made and how many bugs are resolved. We release two evaluation benchmarks: PDB-Single-Hard on single-line bugs, and PDB-Multi on multi-line bugs. Experiments show that frontier models, such as GPT-5.1-Codex and DeepSeek-V3.2-Thinking, achieve unit-test pass rates above 76% but exhibit precision below 45%, even when explicitly instructed to perform minimal debugging. Finally, we show that iterative and agentic debugging strategies do not substantially improve precision or recall, highlighting the need to rethink post-training pipelines for coding models.

## 1. 核心问题与研究背景

- **问题起点**：当前大语言模型（LLMs）在自动编程中显示出强大的代码生成能力，但其在**代码调试（debugging）**场景下存在显著偏差——倾向于**完整重写程序（regeneration）**而非**精准定位并修复错误（precise debugging）**。  
- **主要困境**：传统的单元测试（unit test）评估方式无法区分“通过测试的整体重写”和“精确修改的真实调试”，导致模型调试能力被高估。  
- **研究动机**：为真实评估 LLM 是否具备类人调试能力，作者提出一种能够衡量模型修改“必要性”和“局部性”的评估框架，探索模型从代码生成转向**精确代码修复**的距离。

## 2. 方法论：Precise Debugging Benchmark (PDB)

### 2.1 核心思想
PDB 是一个**自动化调试基准生成与精度评估框架**，旨在：
- 将任意代码生成数据集自动转换为调试任务；
- 通过**精度感知指标**衡量模型的调试行为；
- 明确区分“功能正确”与“编辑精准”之间的差异。

### 2.2 技术流程

1. **Bug 合成阶段（Benchmark Generation）**
   - 从现有编程任务数据集（如 BigCodeBench、LiveCodeBench）中选取正确代码；
   - 通过大型语言模型（如 Claude、Gemini、GPT）自动生成**原子级（atomic）可验证的单行错误**；
   - 组合多个原子错误以形成多错误（multi-bug）程序；
   - 严格验证每个错误的**独立性与原子性**——即修复一个错误不会影响另一个。

2. **评估阶段（Evaluation Pipeline）**
   - PDB 在模型输出的修复结果上计算三类指标：
     - **单元测试通过率（Unit Score）**：传统的功能性评价；
     - **编辑级精度（Edit-Level Precision）**：衡量修改的必要性，即模型的修改中有多少是真正必需的；
     - **错误级召回率（Bug-Level Recall）**：衡量修复的错误比例。
   - 通过“ϵ-松弛（ϵ-relaxed）精度”允许合理的多余行修改，以容忍代码风格或实现差异；
   - 引入 *essentialU* 函数自动提取最小必要修改，用于消除过度修复的影响。

3. **评测指标公式（文字描述）**
   - 精度 `precision = (正确修复的编辑数) / (总编辑数)`
   - 召回 `recall = (已修复的错误数) / (总错误数)`
   - 并通过测试函数 `FU(C)` 自动评判修复后代码是否通过单元测试。

## 6. 主要结论与发现

- **调试精准度普遍低**：
  - GPT-5.1-Codex 与 DeepSeek-V3.2-Thinking 虽单元测试通过率高于 76%，但编辑精度低于 45%；
  - Claude-Sonnet-4.5 与 Gemini-2.5-Pro 在所有指标间表现较平衡（精度与召回均在 70% 以上）。
- **精度–功能性背离**：模型通过率高≠真实调试好；多数“正确答案”来自重新生成。
- **迭代与代理机制无效**：虽提高测试通过率，但不提高精度，反而放大过度修改（regeneration）倾向。
- **提示词影响显著**：最小化调试提示显著提升精度（↑40%），自由式调试精度普遍低于 60%。
- **跨模型生成易–难差异**：Claude 生成的 bug 最难修复，GPT 生成的最易。
- **错误类型差异**：模型在 Build/Package/Merge 类错误上召回最高，说明其偏向于修复较表层的依赖问题。
