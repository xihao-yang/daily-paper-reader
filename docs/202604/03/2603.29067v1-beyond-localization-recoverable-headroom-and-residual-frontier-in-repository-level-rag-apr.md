## 摘要
仓库级自动程序修复（APR）越来越倾向于将强化定位视为获得更好修复效果的主要途径。本文提出一个更具针对性的问题：当定位能力得到增强后，哪些定位后的调控手段仍然能带来可恢复的收益，哪些在我们设定的协议范围内受到限制，以及仍然残留的潜在改进边界是什么？我们在 SWE-bench Lite 数据集上，对三种具有代表性的仓库级 RAG-APR 范式——Agentless、KGCompass 和 ExpeRepair——进行了研究。我们的实验协议结合了 Oracle Localization、池内 Best-of-K 选择、带有按条件相同 token 填充控制的固定接口上下文探测、同仓库困难负样本以及统一包装器的 oracle 检查。Oracle Localization 提升了这三种系统的表现，但 Oracle 成功率仍低于 50%。在采样的 10 个补丁池中，额外的候选多样性仍有助益，但该增益很快趋于饱和。在两个固定接口下，大多数信息量较高的附加上下文条件仍然优于其匹配的控制条件。统一包装器检查显示不同系统的响应差异：在统一包装器下，KGCompass 和 ExpeRepair 的收益依然显著，而 Agentless 对构建方式更为敏感。提示级融合后仍存在显著的剩余改进空间：最佳固定探针仅比原生的三系统 Solved@10 联合结果多解决 6 个实例。总体而言，更强的定位、受限的搜索、证据质量以及接口设计共同影响了仓库级修复的结果。

## Abstract
Repository-level automated program repair (APR) increasingly treats stronger localization as the main path to better repair. We ask a more targeted question: once localization is strengthened, which post-localization levers still provide recoverable gains, which are bounded within our protocol, and what residual frontier remains? We study this question on SWE-bench Lite with three representative repository-level RAG-APR paradigms, Agentless, KGCompass, and ExpeRepair. Our protocol combines Oracle Localization, within-pool Best-of-K, fixed-interface added context probes with per-condition same-token filler controls and same-repository hard negatives, and a common-wrapper oracle check. Oracle Localization improves all three systems, but Oracle success still stays below 50%. Extra candidate diversity still helps inside the sampled 10-patch pools, but that headroom saturates quickly. Under the two fixed interfaces, most informative added context conditions still outperform their own matched controls. The common-wrapper check shows different system responses: under a common wrapper, gains remain large for KGCompass and ExpeRepair, while Agentless changes more with builder choice. Prompt-level fusion still leaves a large residual frontier: the best fixed probe adds only 6 solved instances beyond the native three-system Solved@10 union. Overall, stronger localization, bounded search, evidence quality, and interface design all shape repository-level repair outcomes.

## 一、核心问题与研究背景

- **研究动机**：  
  现有的仓库级自动程序修复（Repository-level APR）通常将“强化定位”视为提高性能的主要途径。然而，强化定位后依然存在部分未解决的修复实例，说明性能瓶颈不仅在定位环节。本文旨在问：  
  > 当定位能力得到增强后，程序修复系统还能从哪些后续阶段获得可恢复的性能提升？哪些收益受限？哪些问题仍然构成长久的未解前沿？

- **背景意义**：  
  随着 LLM 参与代码修复任务的增多，检索增强生成（RAG）成为主流方法。研究者普遍认为若检索到正确位置，修复效果必显著提升，但这类评估往往无法区分定位本身和后续阶段（上下文构建、补丁生成、验证）的贡献。  
  本文通过系统性分解分析，揭示定位之后仍存在的“可恢复空间”和“剩余前沿”，为后续研发方向提供依据。

## 二、方法论与核心思想

- **总体思路**：  
  构建一个“后定位控制实验框架”，在相同 benchmark 上系统比较三种典型仓库级 RAG-APR 系统，通过预言机（Oracle）干预分离定位影响，从而量化后定位阶段的性能空间。

- **主要技术组件**：
  1. **Oracle Localization（预言机定位）**  
     - 从金标准补丁提取故障文件和代码行范围（不暴露修复内容）。  
     - 在系统的功能管线中注入这些真实故障位置，测试如果定位完美，系统性能能提高多少。  
     - 每个系统保留自身上下文构建逻辑，以揭示接口敏感度差异。
  2. **Best-of-K 候选多样性分析**  
     - 每个实例采样 10 个候选补丁，计算其中任一个成功时的 Solved@K 指标。  
     - 通过贪婪排序求理论上限，考察搜索空间和采样多样性带来的额外提升。
  3. **Fixed-Interface Context Probes（固定接口上下文探针）**  
     - 在保持接口固定情况下增加跨系统的上下文信息，如知识图（KGCompass）、经验记忆（ExpeRepair）等。  
     - 配套对照组包括 “相同长度填充”（same-token filler）和 “同仓库硬负样本”，确保比较不受输入长度干扰。
  4. **Common-Wrapper Oracle Check（统一包装器 Oracle 检查）**  
     - 通过共享接口重构，检验系统性能提升是否来自定位信息或接口设计本身。

- **研究问题（RQ）划分**：  
  - RQ1：定位增强带来的直接增益与失效模式。  
  - RQ2：系统内部候选搜索的剩余潜力。  
  - RQ3：固定接口背景下上下文增益归因。  
  - RQ4：在多系统联合后仍存在的剩余前沿。

## 六、主要结论与发现

1. **Oracle 定位显著提升但仍低于 50% 成功率**  
   - Agentless: 28% → 40.3%  
   - KGCompass: 29.3% → 43.0%  
   - ExpeRepair: 32.7% → 39.0%  
   定位完美后仍失败约半数实例，说明后定位环节仍具显著瓶颈。

2. **候选搜索空间有限且早期饱和**  
   - 额外采样 K=10 补丁仅提升 9–13 个百分点。  
   - 前五个候选即可覆盖 86% 以上可恢复增益，后续收益很少。

3. **跨范式上下文增强有效但非线性**  
   - 加入 KGCompass 上下文可提升至 60.7%，而简单联合多个源（UnionContext）反而略有下降。  
   - 表明“更多内容”≠“更好结构”，关键在于信息相关性。

4. **接口设计影响显著**  
   - Agentless 在统一包装器下增益几乎消失，表现对 Builder 高度敏感。  
   - KGCompass / ExpeRepair 的性能提升则较稳定。

5. **剩余前沿广而深**  
   - 三系统联合仅解决 199/300 实例，仍有 101 未解。  
   - 最佳探针仅再解 6 个实例（约 6%）。  
   - 残留错误主要集中在 API 错误、未完成修复、导入缺失等复杂逻辑问题。
