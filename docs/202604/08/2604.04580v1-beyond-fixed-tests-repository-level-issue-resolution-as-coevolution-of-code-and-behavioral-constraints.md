## 摘要
软件工程师在解决仓库级问题时，并不会将现有测试视为不可更改的正确性权威。相反，他们会在迭代过程中同时改进代码和用于表征预期行为的测试，因为新的修改往往会暴露出原有假设的缺失或对失败条件的误解。相比之下，现有的大多数基于大型语言模型（LLM）的修复系统采用线性流程，其中测试或其他验证信号主要充当事后过滤器，在修复过程中将行为约束视为固定不变。这种设定将修复简化为在静态且可能不匹配的约束下进行代码优化，从而导致约束不足的搜索以及脆弱或过拟合的修复结果。我们认为，仓库级问题的解决本质上不是在固定测试下的优化，而是针对不断演化的行为约束进行搜索。为实现这一观点，我们提出了 Agent-CoEvo——一种协同进化的多智能体框架，其中候选代码补丁和测试补丁被联合探索并不断迭代优化。该框架不再将测试视为不可更改的权威，而是将其建模为能够在修复过程中引导并被修订的动态约束。通过相互评估与语义重组，代码与测试候选项逐步收敛到与问题描述一致的行为空间。在 SWE-bench Lite 和 SWT-bench Lite 上的实验表明，Agent-CoEvo 在修复成功率和测试复现质量上均持续超越当前最先进的基于智能体和非智能体的基线方法。我们的研究结果表明，使修复智能体能够在搜索过程中修订行为约束对于可靠的问题解决至关重要，这意味着研究方向应从仅优化代码转向实现实现与规范的协同进化。

## Abstract
Software engineers resolving repository-level issues do not treat existing tests as immutable correctness oracles. Instead, they iteratively refine both code and the tests used to characterize intended behavior, as new modifications expose missing assumptions or misinterpreted failure conditions. In contrast, most existing large language model (LLM)-based repair systems adopt a linear pipeline in which tests or other validation signals act mostly as post-hoc filters, treating behavioral constraints as fixed during repair. This formulation reduces repair to optimizing code under static and potentially misaligned constraints, leading to under-constrained search and brittle or overfitted fixes. We argue that repository-level issue resolution is fundamentally not optimization under fixed tests, but search over evolving behavioral constraints. To operationalize this view, we propose Agent-CoEvo, a coevolutionary multi-agent framework in which candidate code patches and test patches are jointly explored and iteratively refined. Rather than treating tests as immutable oracles, our framework models them as dynamic constraints that both guide and are revised by the repair process. Through mutual evaluation and semantic recombination, code and test candidates progressively narrow the space of behavior consistent with the issue description. Evaluated on SWE-bench Lite and SWT-bench Lite, Agent-CoEvo consistently outperforms state-of-the-art agent-based and agentless baselines in both repair success and test reproduction quality. Our findings suggest that enabling repair agents to revise behavioral constraints during search is critical for reliable issue resolution, pointing toward a shift from code-only optimization to coevolution of implementation and specification.

## 一、研究背景与核心问题

- **研究动机：**  
  当前基于大型语言模型（LLM）的自动程序修复系统普遍将测试视为静态、固定的正确性标准，仅在修复生成后用作验证或过滤。这种“固定约束优化”范式忽视了软件开发中的现实过程——实际工程师会在修复过程中同时修改代码与测试，以反映新发现的行为假设与约束。
- **核心问题：**  
  传统方法面临两类问题：
  1. **约束不足（Under-constraining）**：不完整或错误的测试使修复搜索过于宽松，导致生成的补丁虽能通过测试，但并非真正修复了问题。  
  2. **过拟合与脆弱修复**：代码被迫满足错误或不充分的测试约束，从而产生不鲁棒的修复结果。
- **论文目标：**  
  作者提出将仓库级问题解决视为**“代码实现（implementation）与行为约束（specification）协同进化”**的过程，而非仅在固定测试下优化代码。

## 二、方法论：Agent-CoEvo 协同进化框架

### 1. 核心思想

- 将代码补丁与测试补丁视为相互依赖的两个种群，**在演化过程中交替评估与共生优化**。
- 测试不再是固定的判断标准，而成为可演化的行为约束；代码在满足这些约束的同时，反过来也帮助改进测试的充分性。

### 2. 系统总体结构

**三类智能体：**
1. **LocationAgent：** 根据自然语言 issue 描述执行“问题定位”，利用动态执行轨迹识别出潜在故障位置。  
2. **CodeAgent：** 维护代码补丁种群，通过LLM生成和语义交叉；基于适应度函数进行进化。  
3. **TestAgent：** 维护测试补丁种群，生成能重现错误行为的测试，并在演化过程中与 CodeAgent 交互反馈。

### 3. 演化过程（Algorithm 1 & 2 概述）

1. **初始化阶段：**  
   - 生成初始的代码与测试补丁集（P<sub>code</sub>，P<sub>test</sub>）。  
   - 仅保留能在原始仓库上失败的测试用例（满足故障相关性）。
2. **交叉评估（CrossEvaluation）：**  
   - 构建执行矩阵 M(|P<sub>code</sub>|×|P<sub>test</sub>|)，元素为测试是否通过（1 表 PASS，0 表 FAIL）。  
   - 计算适应度：  
     - **代码补丁适应度：** 取通过测试数量与一致性群体大小之积；  
       \( F_{code}[c] = |S_{consensus}| × |S_{pass}| \)。  
     - **测试补丁适应度：** 按被高质量代码通过的程度加权求和；  
       \( F_{test}[t] = \sum_c M[c,t]·F_{code}[c] \)。
3. **选择与交叉：**  
   - 基于锦标赛选择（Tournament Selection）挑选父代。  
   - **语义交叉（Semantic Crossover）：** LLM综合父代逻辑语义生成新个体，而非文本拼接。
4. **精英保留：**  
   - 每轮保留最佳个体，确保迭代过程中适应度不下降。

### 4. 环境保障
- 使用**Docker隔离运行环境**执行真实测试，防止补丁副作用影响系统状态，确保结果可复现。

## 三、实验设计

### 1. 数据集与任务
- **SWE-bench Lite：**  
  - 300个源自GitHub真实问题的Python项目实例，用于评估修复正确性。  
  - 指标：修复成功率（Resolved %）。
- **SWT-bench Lite：**  
  - 276个与SWE-bench配对的实例，用于评估错误复现测试生成能力。  
  - 指标：测试复现成功率、ΔC（修改区域覆盖率增量）。

### 2. 对比基线
分为三类：
1. **代码修复代理（固定测试范式）：** Agentless、SpecRover、Moatless Tools、KGCompass、DARS、SWE-Search；
2. **测试生成代理：** SWE-Agent+、AssertFlip、AEGIS；
3. **通用软件智能体：** AutoCodeRover、SWE-Agent、OpenHands。

### 3. 实验配置
- 使用 **DeepSeek-V3-0324** 模型（通过 OpenRouter API 访问）。  
- 种群大小 = 10，最大迭代 = 5。  
- 位置定位温度 = 0.2，代码/测试生成温度 = 0.5。  
- 评价指标统一遵循 SWE-bench 官方标准。
