## 摘要
随着多核处理器和分布式系统的兴起，并发编程变得既重要又充满挑战，主要原因在于线程执行的非确定性。手动修复并发错误既耗时又容易出错。自动化程序修复技术提供了一种前景广阔的解决方案。然而，开发一个端到端的并发错误修复工具尤其困难。现有的大多数工具依赖于假设，即错误相关信息是现成可得的，或者并发错误上下文可以被理想地提取，但这在现实场景中往往并不实际。本文介绍了一种名为 ConFixAgent 的基于大型语言模型（LLM）的代理，它能够以端到端的方式修复多种类型的并发错误，而无需任何先验的错误相关信息。我们提出了一种新的用于并发错误修复的上下文提取方法，基于静态 Happens-Before 图（Static Happens-Before Graph, SHB）来识别与错误相关的代码片段。我们实现了 ConFixAgent 并在多个基准数据集上进行了评估。大量实验结果表明，ConFixAgent 在处理多种类型的并发错误方面显著优于现有的最新工具，其上下文提取方法显著提高了 LLM 生成修复方案的准确性。

## Abstract
With the rise of multi-core processors and distributed systems, concurrent programming has become essential yet challenging, primarily due to the non-deterministic nature of thread execution. Manually addressing concurrency bugs is time-consuming and error-prone. Automated Program Repair techniques provide a promising solution. However, developing an end-to-end concurrency bug repair tool is particularly challenging. Most existing tools rely on the assumption that bug-related information is readily available or that concurrency bug contexts are ideally extracted, which is often impractical in real-world scenarios. This paper introduces ConFixAgent, an LLM-driven agent capable of fixing various types of concurrency bugs in an end-to-end manner, eliminating the need for any prior bug-related information. Specifically, we propose a novel context extraction approach designed for concurrency bug repair, utilizing Static Happens-Before Graphs to identify bug-relevant sections.We implemented ConFixAgent and evaluated it across multiple benchmark sets. Our extensive experiments demonstrate that ConFixAgent significantly outperforms state-of-the-art tools in addressing diverse types of concurrency bugs, with its context extraction method markedly enhancing the accuracy of LLM-generated repair solutions.

## 一、研究背景与核心问题

- **研究动机**：  
  随着多核处理器与分布式系统的普及，并发编程成为现代软件开发的核心。然而，由于线程执行具有非确定性特征，并发程序极易产生复杂的**并发错误（concurrency bugs）**，如数据竞争、原子性违规、死锁等。这类错误往往难以检测与修复，导致软件故障频发、维护成本高昂。

- **核心挑战**：  
  当前的自动化程序修复（Automated Program Repair，APR）方法在并发场景下仍有三大难题：
  1. **错误信息难获取**：现有工具通常要求人工提供错误定位或上下文。
  2. **类型覆盖有限**：多数工具只针对特定类型的并发错误（如数据竞争）。
  3. **修复正确性不足**：难以保证在各种线程调度下均能保持正确性，易引入新的错误。

- **研究意义**：  
  本文提出一种端到端的自动化并发错误修复系统 **ConFixAgent**，不仅可自主检测错误，还能定位并生成修复补丁，完成从检测到修复的闭环。

## 二、方法论与技术方案

### 1. 核心思想
ConFixAgent 是一个基于 **大型语言模型（LLM）驱动的智能代理**，通过结合静态分析与动态检测，实现端到端并发错误修复，无需任何手动输入或先验错误信息。

### 2. 关键模块

- **(1) 静态 Happens-Before 图（Static Happens-Before Graph, SHBG）**  
  构建线程间事件的静态因果关系图，用来分析可能的内存访问顺序和潜在冲突：
  - 节点：代表内存访问事件或同步操作。
  - 边：表示 `happens-before` 关系（事件的潜在执行顺序）。
  - 并行事件 (`e1 ∥hb e2`) 表示可能发生并发执行，为并发错误检测重点。

- **(2) 上下文提取算法（Algorithm 1）**  
  核心目标是过滤掉与并发错误无关的代码，保留 “bug-relevant methods”。算法主要步骤：
  1. 构建 SHBG。
  2. 检测可能的死锁事件（采用 [51],[52] 的算法）。
  3. 标记语义相关方法（涉及线程创建或通信的函数）。
  4. 标识与 bug 相关的事件及其方法。
  5. 在调用图上进行闭包计算，保留调用链相关函数。
  6. 输出经过过滤的代码片段传递给 LLM。

  时间复杂度约为 **O(N²)**，其中 N 为方法数量。

- **(3) LLM 修复模块**
  - 利用 GPT 系列模型（主要使用 **GPT-4-turbo-2024-04-09**）。
  - 生成结构化 Prompt（含 Prelude、Code、Bug Report、Instructions 和 Format Guidelines 五部分）。
  - 自动生成补丁并反复验证（最多五次迭代），结合 JPF 动态检测反馈逐步优化。
  - 采用单步策略 (**Gone step**) 把定位与修复融合为一次性任务，效果优于分步 CoT 方案。

## 五、主要结论与发现

- **总体表现**：
  - ConFixAgent 在非死锁错误上成功修复 29/33 个实例，超过 PFIX（22）和 HIPPODROME（21）。
  - 死锁错误方面成功修复 4 个，是唯一具备此能力的工具。
  - 修复成功率（GPT-4）达 **76%**，远超 SWE-agent（40%）。

- **上下文提取效果**：
  - 过滤代码超过 90% 的无关 token，修复准确率提升约 **14%**。
  - 实践中达到几乎完全的完整性，没有出现因过滤导致修复失败的案例。

- **修复质量**：
  - 平均新增锁数量为 2 个（PFIX 为 1.5，HIPPODROME 为 8.42），更高效且更少引入死锁。

## 六、方法与设计亮点

- **端到端闭环体系**：从检测到修复完全自动，无需开发者输入。
- **SHB 图创新使用**：将原并发分析的 `Happens-Before` 关系用于上下文选择，显著提升 LLM 理解力。
- **高效上下文精简**：过滤超 90% token，在保持修复率同时降低模型干扰。
- **LLM 调用优化策略**：采用单步合并（Gone step）代替长推理链（CoT），效率更高。
- **健全评测体系**：覆盖多基准、多策略、重复试验与统计验证，结果可靠。

## 七、不足与局限

- **理论完备性不足**：上下文提取算法仅具实证完整性，无法保证理论完备。
- **并发检测瓶颈**：使用 JPF 时面临状态空间爆炸，影响大程序可扩展性。
- **模型随机性**：LLM 输出存在概率性，导致少量补丁语义偏差（约 5.4%）。
- **资源限制**：仅使用 CPU 环境，无 GPU 加速；对大规模工业项目的性能未知。
- **训练数据污染风险**：作者指出可能存在模型训练数据中含有部分相似代码片段，影响结果纯度。
- **语言与平台限制**：当前仅支持 **Java** 程序，不适用于 C/C++ 或多语言系统。
