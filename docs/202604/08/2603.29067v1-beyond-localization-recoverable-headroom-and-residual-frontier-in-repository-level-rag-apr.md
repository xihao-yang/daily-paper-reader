## 摘要
在仓库级自动程序修复（APR）中，研究者越来越多地把更强的定位能力视为提升修复性能的主要途径。本文提出一个更具针对性的问题：当定位能力得到增强后，定位之后的哪些操作仍能带来可恢复的增益？这些增益在我们的协议框架内受到哪些限制？又剩下怎样的残余前沿？我们在 SWE-bench Lite 上使用三种具有代表性的仓库级 RAG-APR 范式——Agentless、KGCompass 和 ExpeRepair——研究这一问题。我们的协议结合了 Oracle Localization、池内 Best-of-K、固定接口下添加上下文探针（为每种条件设置相同 token 的填充控制）与同仓库的强负样本，以及通用包装器下的 oracle 检查。Oracle Localization 改善了三个系统的性能，但 Oracle 成功率仍低于 50%。在采样的 10 个补丁池中，额外的候选多样性仍有帮助，但这种提升空间很快达到饱和。在两个固定接口下，大多数信息量较高的附加上下文条件仍显著优于其匹配控制条件。通用包装器检查显示各系统的不同响应：在通用包装器下，KGCompass 和 ExpeRepair 的增益仍然较大，而 Agentless 对构建器选择更加敏感。提示级融合仍留下较大的残余前沿：最优固定探针仅比三系统原生的 Solved@10 联合结果多解决 6 个实例。总体而言，更强的定位能力、受限的搜索范围、证据质量以及接口设计共同塑造了仓库级修复的结果。

## Abstract
Repository-level automated program repair (APR) increasingly treats stronger localization as the main path to better repair. We ask a more targeted question: once localization is strengthened, which post-localization levers still provide recoverable gains, which are bounded within our protocol, and what residual frontier remains? We study this question on SWE-bench Lite with three representative repository-level RAG-APR paradigms, Agentless, KGCompass, and ExpeRepair. Our protocol combines Oracle Localization, within-pool Best-of-K, fixed-interface added context probes with per-condition same-token filler controls and same-repository hard negatives, and a common-wrapper oracle check. Oracle Localization improves all three systems, but Oracle success still stays below 50%. Extra candidate diversity still helps inside the sampled 10-patch pools, but that headroom saturates quickly. Under the two fixed interfaces, most informative added context conditions still outperform their own matched controls. The common-wrapper check shows different system responses: under a common wrapper, gains remain large for KGCompass and ExpeRepair, while Agentless changes more with builder choice. Prompt-level fusion still leaves a large residual frontier: the best fixed probe adds only 6 solved instances beyond the native three-system Solved@10 union. Overall, stronger localization, bounded search, evidence quality, and interface design all shape repository-level repair outcomes.

## 一、核心问题与研究动机

- **背景：**  
  仓库级自动程序修复（Repository-level APR）是让大型语言模型（LLMs）在整个代码库范围内自动定位并修复缺陷。然而，仓库规模远超单个 Prompt 窗口，必须通过**检索增强生成（RAG）**技术构建有限上下文来辅助修复。  
  现有研究普遍认为，提高「**缺陷定位（localization）**」精度能最显著提升修复性能。

- **核心问题：**  
  本文并非继续改进定位算法，而是提出关键问题：  
  👉 **当定位已足够精准后，后续流程还能恢复多少性能？这种增益是否存在上限，剩下的瓶颈是什么？**

- **研究意义：**  
  以往研究往往将性能提升混淆为定位改进导致，未能区分「定位质量、上下文融合、补丁搜索」等因素的贡献。本文通过系统化的「后定位实验设计」来明确这些环节的作用与天花板。

## 二、方法论与研究框架

### 1. 核心思想
- 设计一套**“定位后协议（post-localization protocol）”**，控制变量地分析 RAG-APR 系统在增强定位后可恢复的性能空间与残留瓶颈。
- 实现方式是引入多级对照实验与 oracle 控制，将「定位准确性」独立于后续 repair 流程。

### 2. 四个核心技术模块
1. **Oracle Localization（预言机定位）：**  
   向系统注入由官方修复补丁（gold diff）解析出的真实故障位置 *G = {(file, [start, end])}*，仅暴露行号与文件，不泄露修复内容。  
   → 观察若定位完美，系统性能提升上限为何。
2. **Best-of-K 多样性搜索：**  
   对每个样例生成 10 个候选补丁，统计「Solved@K」随 K 增长的曲线，评估系统在已有定位下的**内部搜索潜力与饱和趋势**。
3. **Fixed-Interface Added Context Probes（固定接口上下文探针）：**  
   固定不同系统的提示接口（Agentless 或 ExpeRepair），在最后生成前**注入额外跨系统上下文证据**（来自 KGCompass 或 ExpeRepair）；
   对比：
   - 同 token 长度的填充控制（filler control）；
   - 同仓库的负样本（hard negative）。  
   → 判断性能改进是否源于信息质量，而非提示字数。
4. **Common-Wrapper Oracle Check（通用包装检查）：**  
   用统一的补丁生成包装器重放 Oracle 实验，拆解接口差异（wrapper/builder）对结果的影响。

### 3. 诊断指标与流程逻辑
- 定义三种输出状态：**已解决（resolved）**、**提交未解决（completed-unresolved）**、**未提交（no patch）**；
- 对比阶段变化：基线 → Oracle → Best-of-K；
- 交叉系统比较与残余前沿分析（residual frontier），量化三系统在联合后仍未解决的样例比例。

## 六、主要结论与发现

1. **增强定位显著提升性能但仍有限**  
   - Oracle 定位将成功率提至约 40–43%，明显高于基线（28–33%），但仍不到一半；
   - 定位改进主要减少 “未提交补丁” 错误，而非彻底解决问题。
2. **候选多样性（Best-of-K）有助但饱和快**  
   - 从 Oracle 到 Best-of-10 仅再提升约 10–13 个百分点；
   - 大部分可恢复增益在 K≤5 即已达到 86%+。
3. **添加上下文有正效应**  
   - 信息性上下文显著优于同 token 填充与错位负样本；
   - 表明增益源于上下文结构质量非提示长度；
   - 但多源拼接（UnionContext）不总有效，部分情况下还稀释单源增益。
4. **不同系统对包装依赖差异明显**  
   - Agentless 对 wrapper/builder 更敏感；
   - KGCompass 与 ExpeRepair 在共享包装下仍保持高幅度增益。
5. **残余前沿仍广泛存在**  
   - 三系统结合后仍有 101/300 样例未解；
   - 最优单探针仅额外解 6 个案例；六探针联合最多恢复 14 个；
   - 主要瓶颈集中于 **API错误、修复不完整、断言不匹配** 等高层逻辑失误。
