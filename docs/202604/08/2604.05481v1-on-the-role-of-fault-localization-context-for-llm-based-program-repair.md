## 摘要
故障定位（Fault Localization，FL）是基于大型语言模型（LLM）的自动程序修复（Automated Program Repair，APR）的关键组成部分，但其影响仍缺乏充分研究。特别地，目前尚不明确需要多少定位信息，预测出错误位置之外的额外上下文是否有益，以及应如何获取此类上下文。我们在使用 GPT-5-mini 的 500 个 SWE-bench Verified 实例上进行了一项大规模实证研究，评估了 61 种配置，这些配置在文件级、元素级和行级上下文上有所不同。结果表明，更多上下文并不总能提高修复性能。文件级定位是主导因素，相比于无文件基线表现提升达 15–17 倍。扩展文件上下文通常与性能提升相关，最成功的修复多出现在约 6–10 个相关文件的配置中。元素级上下文扩展提供了依赖于文件上下文质量的条件性收益，而行级上下文扩展则常因噪声放大而导致性能下降。基于 LLM 的检索在使用更少文件和 token 的情况下通常优于结构启发式方法。总体而言，最有效的 FL 上下文策略通常结合高层抽象上的广义语义理解与精确的行级定位。这些发现挑战了“增加定位上下文可均匀提升 APR 性能”的假设，并为设计基于 LLM 的 FL 策略提供了实践性指导。

## Abstract
Fault Localization (FL) is a key component of Large Language Model (LLM)-based Automated Program Repair (APR), yet its impact remains underexplored. In particular, it is unclear how much localization is needed, whether additional context beyond the predicted buggy location is beneficial, and how such context should be retrieved. We conduct a large-scale empirical study on 500 SWE-bench Verified instances using GPT-5-mini, evaluating 61 configurations that vary file-level, element-level, and line-level context. Our results show that more context does not consistently improve repair performance. File-level localization is the dominant factor, yielding a 15-17x improvement over a no-file baseline. Expanding file context is often associated with improved performance, with successful repairs most commonly observed in configurations with approximately 6-10 relevant files. Element-level context expansion provides conditional gains that depend strongly on the file context quality, while line-level context expansion frequently degrades performance due to noise amplification. LLM-based retrieval generally outperforms structural heuristics while using fewer files and tokens. Overall, the most effective FL context strategy typically combines a broad semantic understanding at higher abstraction levels with precise line-level localization. These findings challenge our assumption that increasing the localization context uniformly improves APR, and provide practical guidance for designing LLM-based FL strategies.

## 1. 研究背景与核心问题

- **研究动机：**  
  传统自动程序修复（Automated Program Repair, APR）系统依赖故障定位（Fault Localization, FL）来确定可疑代码区域，而在基于大语言模型（LLM）的APR中，模型具有强大的上下文理解能力，这使得“应该给多少上下文”成为一个关键问题。  
  现有研究多聚焦于提升定位精度，却忽视了**不同层次（文件 / 函数 / 行）上下文的影响机制**，导致人们普遍假设“更多上下文总是更好”，但缺乏实证依据。  

- **研究目标：**  
  探明在 LLM 驱动的 APR 中：
  1. 不同层次（文件级、元素级、行级）上下文对修复性能的具体影响；
  2. 扩展额外上下文（beyond buggy location）是否有利；
  3. LLM 语义检索与传统结构性启发式（如静态分析）的效果对比。

## 2. 方法论与技术设计

- **核心思想：**  
  设计一个多维度的上下文因子实验（factorial design），系统评估文件级、元素级、行级三种上下文的组合效应，分析不同粒度定位的独立贡献与交互作用。

- **三层上下文定义：**
  - **文件级（File-level）**：  
    无文件 / 仅错误文件（Buggy） / 基于导入关系的相关文件（Rule-based） / 由 LLM 按语义检索的相关文件（LLM-based）。  
  - **元素级（Element-level）**：  
    无元素、错误元素、调用图扩展（Call Graph）、LLM 检索相关函数或类。

    无行、错误行、±10 行窗口（Context Window）、静态切片（Code Slicing）、LLM 检索行。

- **实验变量：**
  - 共 61 种上下文组合（3 层 × 多方式因子全交），每种配置独立运行一次。
  - 为排除定位误差，使用 **真实开发者补丁（Ground Truth, GT）** 提取精确的错误文件、函数、行。

- **技术框架：**

    - 系统提示：设定为“专家软件工程师”；

## 6. 主要发现与结论

- **整体趋势：**
  - 仅提供错误文件即可相对无上下文基线提升 **约15–17倍** 的修复率；
  - File-level 上下文是最关键的因素；

- **分层特性：**
  - **文件级：** 扩展有益，尤其 LLM 检索优于基于导入的规则扩展；最优文件数约 6–10；且成本（tokens）更低、更稳定。
  - **元素级：** 仅当文件上下文精确时才显著提升性能；LLM 语义检索优于调用图扩展。
  - **行级：** 过多邻行或代码切片**反而降低表现**，因引入噪声；适度且精确的行位置信息最可靠。
  - **最佳组合：**  
    「LLM 文件 + LLM 元素 + 精确错误行」 → 63.4% 修复成功率（最佳配置）。

- **交互关系：**
  - 高层（文件、函数）需语义广泛；低层（行级）需定位精确；
  - 统一扩大上下文并不一定提高性能。
