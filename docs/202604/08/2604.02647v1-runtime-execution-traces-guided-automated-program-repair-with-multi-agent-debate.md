## 摘要
自动程序修复（Automated Program Repair, APR）在处理复杂逻辑错误和隐性故障时面临挑战。当前基于大型语言模型（LLM）的APR方法主要是静态的，依赖源代码和基础测试输出，无法准确捕捉复杂的运行时行为和动态数据依赖。尽管引入执行轨迹等运行时证据可以揭示具体的状态转变，但单个LLM独立解释这些信息时往往会过拟合于特定假设，从而生成仅凭巧合通过测试而非基于正确逻辑的补丁。因此，运行时证据应作为客观约束，而不仅仅是额外输入。我们提出TraceRepair，一个利用运行时事实作为共享约束进行补丁验证的多智能体框架。一个探测智能体用于捕获关键变量的执行快照，以形成客观的修复基础；同时，一组专门的评审智能体交叉验证候选补丁，揭示不一致并迭代改进。我们在Defects4J基准上评估了TraceRepair，正确修复了392个缺陷，显著优于现有的基于LLM的方法。大量实验表明，该方法在新构建的近期漏洞数据集上表现出更高的效率和强泛化能力，证明性能提升源自动态推理而非记忆。

## Abstract
Automated Program Repair (APR) struggles with complex logic errors and silent failures. Current LLM-based APR methods are mostly static, relying on source code and basic test outputs, which fail to accurately capture complex runtime behaviors and dynamic data dependencies. While incorporating runtime evidence like execution traces exposes concrete state transitions, a single LLM interpreting this in isolation often overfits to specific hypotheses, producing patches that satisfy tests by coincidence rather than correct logic. Therefore, runtime evidence should act as objective constraints rather than mere additional input. We propose TraceRepair, a multi-agent framework that leverages runtime facts as shared constraints for patch validation. A probe agent captures execution snapshots of critical variables to form an objective repair basis. Meanwhile, a committee of specialized agents cross-verifies candidate patches to expose inconsistencies and iteratively refine them. Evaluated on the Defects4J benchmark, TraceRepair correctly fixes 392 defects, substantially outperforming existing LLM-based approaches. Extensive experiments demonstrate improved efficiency and strong generalization on a newly constructed dataset of recent bugs, confirming that performance gains arise from dynamic reasoning rather than memorization.

## 二、方法论与技术框架

### 1. 核心思想

提出 **TraceRepair** 框架——一种结合**运行时执行轨迹（Runtime Execution Trace）**与**多智能体辩论机制（Multi-Agent Debate）**的自动程序修复系统：
- 利用运行时探测（Probe Agent）生成执行快照；
- 多个修复智能体（Repair Agents）形成辩论组，对候选补丁进行交叉审查；
- 通过Judge Agent统合结果，生成最终修复补丁。

### 2. 三阶段工作流程

1. **阶段 1：诊断探测（Diagnostic Probing）**
   - 定位可疑代码行与关键变量；
   - 通过插桩（AST修改）注入日志捕获运行时轨迹；
   - 采用防御式策略（try-catch封装、自动回滚）保证探测安全；
   - 定义运行时轨迹 `τ = {(t, v, val) | t ∈ 时间步, v ∈ 关键变量, val ∈ 运行值}`；
   - 实行**尾部优先截断策略**以控制上下文规模（保留错误发生前后状态）。

2. **阶段 2：多策略辩论（Multi-Strategy Debate）**
   - 初始化三类智能体：
     - **Defensive**：注重边界与鲁棒性；
     - **Causal**：追踪错误传播路径；
     - **Semantic**：保持API语义与规范一致。
   - 每个智能体生成多个候选补丁，并进入循环：
     - “生成→互评→更新”。
   - **辩论公式**：
     - 若防御补丁 `h_def` 与运行轨迹不一致，则由攻击者 `h_atk` 产生批评  
       `k = F_crit(h_atk, h_def, τ)`；
     - 防御者据此更新补丁：`h_new = F_refine(h_def, K, τ)`；
   - 每轮迭代受运行时证据τ约束，防止逻辑幻觉与错误假设。

3. **阶段 3：仲裁与综合（Arbitration and Synthesis）**
   - 当辩论未收敛时引入Judge Agent对全程辩论历史进行回顾；
   - 综合多策略的局部修正为完整补丁；
   - 使用 runtime trace 校验一致性，丢弃任何违反执行事实的修复结果。

算法1（TraceRepair过程）包括三个阶段共约30步：

## 六、方法与设计亮点

- **运行时指导原则**：把执行轨迹视为逻辑约束而非文本辅助，使模型推理“以事实为准”。  
- **多智能体辩论机制**：不同策略（防御/因果/语义）相互批评，减少幻觉与错误自洽。  
- **Probe Agent安全插桩**：利用自动回滚策略确保插桩不破坏原编译性。  
- **Tail-Biased Trace处理策略**：针对LLM上下文长度限制，保留最关键的临近失败快照。  
- **可扩展性强**：框架与模型解耦，可兼容任意LLM后端。
