## 摘要
多智能体框架广泛应用于自主代码生成，并在复杂算法问题求解中具有重要用途。近期的研究通过引入基于模拟驱动的规划与调试，解决了生成功能正确代码的挑战，其中语言模型会追踪执行步骤以验证逻辑。然而，这些方法依赖人工提供的公共测试用例来支撑调试与模拟循环。手动编写全面的输入输出示例是软件开发周期中的劳动密集型瓶颈。由于在真实的软件工程环境中，真实的输入输出示例在实现之前极少可得，这种依赖性限制了方法只能应用于精心整理的竞赛编程基准。此外，我们发现对这些公共测试的依赖会引发“过度自信差距”，导致框架对简单示例过拟合，并在隐藏评测中失败。相反，我们观察到外部样本输入并非代码生成的必要条件。我们展示了大型语言模型能够自主生成有效输入并模拟执行轨迹以自我修正。据此，我们开发了 DryRUN 框架，该框架通过允许 LLM 迭代规划、自主生成输入并模拟执行，无需依赖真实样本，从而减轻算法性过度自信。在 LiveCodeBench v6（2025 年 3 月之后）数据集上的评估表明，DryRUN 在完全不使用公共测试用例或外部执行反馈的情况下，其性能与依赖公共测试的最新框架 CodeSIM 相当，同时减少了输出 token 的消耗。

## Abstract
Multi-agent frameworks are widely used in autonomous code generation and have applications in complex algorithmic problem-solving. Recent work has addressed the challenge of generating functionally correct code by incorporating simulation-driven planning and debugging, where language models trace execution steps to verify logic. However, these approaches depend on human-provided public test cases to ground the debugging and simulation loop. Manually authoring comprehensive input-output examples is a labor-intensive bottleneck in the software development lifecycle. Because ground-truth input-output examples are rarely available prior to implementation in real-world software engineering, this dependency restricts methods to curated competitive programming benchmarks. Furthermore, we identify that reliance on these public tests induces an ``overconfidence gap,'' causing frameworks to overfit to simplistic examples and fail on hidden evaluations. In contrast, we observe that external sample inputs are not strictly necessary for code generation. We demonstrate that large language models can autonomously generate valid inputs and simulate execution traces to self-correct. Consequently, we develop DryRUN, a framework that eliminates the need for ground-truth samples by allowing the LLM to iteratively plan, autonomously generate its own inputs and simulate execution, mitigating algorithmic overconfidence. Evaluations on the LiveCodeBench v6 dataset (post-March 2025) demonstrate that DryRUN matches performance against CodeSIM, a state-of-the-art and public-test-dependent framework, while operating entirely without public test cases or external execution feedback while reducing output token consumption.

## 一、研究问题与背景  

- **研究动机**：  
  当前的 LLM 代码生成框架（如 CodeSIM、MapCoder 等）普遍依赖于**人工编写的公共测试样例（public tests）**来进行调试和模拟。这种依赖具有两大问题：

  2. **过拟合性**：公共测试往往只覆盖简单场景，使得模型高估其性能（通过公共测试但在隐藏私有测试中失败），这一现象被作者定义为“**过度自信差距（Overconfidence Gap）**”。  

- **核心问题**：  
  论文旨在探索——是否可以完全去除公共测试，用纯语言模型的“内在推理与模拟能力”完成代码验证与调试？  
  作者提出的答案是肯定的，并基于此提出了新的框架 **DryRUN（Debugging and Refinement Under Non-execution）**。

## 二、方法论：DryRUN 框架  

### 核心思想  
- 让 LLM 在**没有任何真实输入输出样例**的情况下，通过自我规划、自行生成输入、进行“心理执行（mental simulation）”与自我修正，实现代码的自我调试。  
- 彻底消除对外部执行环境（sandbox）和公共测试的依赖。

### 技术流程（Algorithm 1 概述）  

DryRUN 包含多个阶段：  

1. **规划生成（Planning）**  
   - LLM 阅读问题说明 → 先生成一个详细的解决计划（Plan），而非立即生成代码。  
   - 进行若干轮（𝑁ₚₗₐₙ）计划迭代，系统性完善逻辑和考虑边界情况。  

2. **代码生成（Code Synthesis）**  
   - 根据最终计划生成初始代码。  

3. **心理执行与轨迹驱动修正（Mental Simulation & Trace-Driven Refinement）**  
   - LLM 自主生成符合约束的有效输入；  
   - 在脑内“执行”代码（不会真的运行）并追踪变量变化；  
   - 根据模拟轨迹的逻辑错误更新计划并重生代码；  
   - 迭代多轮（𝑁ₛᵢₘ）。  

4. **最终润色（Polish）**  
   - 对最终代码进行语法与逻辑的自检与微调。  

**特点对比 CodeSIM：**  
- 无外部反馈、依赖自主输入。  
- 不执行代码，只进行心智模拟。  
- 更强调前置的“计划精化”而非后置修复。  
- 增加最终润色阶段，以弥补缺少执行反馈可能引入的语法与逻辑残差。

## 六、主要结论与发现  

1. **性能对比：**  
   - DryRUN 在无公共测试的零样本条件下，Pass@1 表现几乎等同甚至略高于 CodeSIM。  
     - 例如在 gpt-5-mini 上整体 67.5% vs CodeSIM 64.2%；  
     - 在 gemini-3-flash 上 69.6% vs 74.6%，差距在误差范围内。  

2. **资源效率：**  
   - DryRUN 仅消耗 CodeSIM 的约 **70% 总 token** 和 **50% 输出 token**。  

3. **消融表现：**  
   - 单独规划阶段提升 +15%，  
   - 计划双次迭代进一步稳定逻辑；  
   - 两轮心理模拟可显著降低结果波动；  
   - 最终润色阶段额外提升约 +3.6%。  

4. **过度自信差距分析（Overconfidence Gap）：**  
   - 传统依赖公共测试的方法（CodeSIM）在困难题中表现出明显的“错误自信”现象；  
   - DryRUN 通过自生成较难但可追踪的输入，有效缓解此问题。
