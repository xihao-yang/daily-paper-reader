## 摘要
在代码仓库级自动化程序修复（APR）领域，更强的定位（localization）越来越被视为提升修复效果的主要途径。本研究提出一个更具针对性的问题：在定位能力增强之后，哪些定位之后的调节因素仍能带来可恢复的收益，哪些收益受制于我们的实验协议，以及剩余的边界前沿在哪里？我们在 SWE-bench Lite 上针对三种具有代表性的仓库级 RAG-APR 范式——Agentless、KGCompass 和 ExpeRepair——进行研究。我们的实验协议结合了 Oracle Localization、池内 Best-of-K、固定接口附加上下文探针，并为不同条件设置相同 token 的填充控制和同一仓库的强负样本，以及统一的封装器（common-wrapper）oracle 校验。Oracle Localization 提升了所有三种系统的性能，但其 Oracle 成功率仍低于 50%。在采样的 10 个补丁池中，额外候选的多样性仍然有帮助，但这一改进空间很快达到饱和。在两个固定接口下，大多数信息量丰富的附加上下文条件仍优于其自身匹配的控制条件。统一封装器检查显示不同系统的响应：在共同封装器下，KGCompass 和 ExpeRepair 的收益仍然较大，而 Agentless 的变化更依赖于构建器选择。提示级融合（Prompt-level fusion）仍留下较大的剩余前沿：最佳固定探针仅比三个原生系统的 Solved@10 联合集增加 6 个已解决实例。总体而言，更强的定位、有限的搜索范围、证据质量以及接口设计，共同决定了仓库级修复的结果。

## Abstract
Repository-level automated program repair (APR) increasingly treats stronger localization as the main path to better repair. We ask a more targeted question: once localization is strengthened, which post-localization levers still provide recoverable gains, which are bounded within our protocol, and what residual frontier remains? We study this question on SWE-bench Lite with three representative repository-level RAG-APR paradigms, Agentless, KGCompass, and ExpeRepair. Our protocol combines Oracle Localization, within-pool Best-of-K, fixed-interface added context probes with per-condition same-token filler controls and same-repository hard negatives, and a common-wrapper oracle check. Oracle Localization improves all three systems, but Oracle success still stays below 50%. Extra candidate diversity still helps inside the sampled 10-patch pools, but that headroom saturates quickly. Under the two fixed interfaces, most informative added context conditions still outperform their own matched controls. The common-wrapper check shows different system responses: under a common wrapper, gains remain large for KGCompass and ExpeRepair, while Agentless changes more with builder choice. Prompt-level fusion still leaves a large residual frontier: the best fixed probe adds only 6 solved instances beyond the native three-system Solved@10 union. Overall, stronger localization, bounded search, evidence quality, and interface design all shape repository-level repair outcomes.

## 二、方法论与研究设计思路

### 1. 核心思想
作者并非提出一个新的修复系统，而是**构建一个对 RAG-APR 各阶段的“受控实验协议”**，以分离不同因素的贡献。研究的重点是识别强化定位后的“可恢复余量（Recoverable Headroom）”和“残余前沿（Residual Frontier）”。

### 2. 核心技术与流程
论文提出了一套有系统的分析协议，包括以下几个关键要素：

1. **Oracle Localization（预言机定位）**  
   - 从真实补丁（gold diff）中提取正确的故障文件和行号，仅暴露定位范围，不泄露修复内容。  
   - 在不改变各系统下游逻辑的前提下，以此模拟“理想定位”场景，衡量定位增强所能带来的上限效益。

2. **Best-of-K 多样性搜索**  
   - 每个任务采样多达 10 个候选补丁，评估在“理想补丁选择器”下能额外恢复的成功率，刻画搜索空间的**可恢复余量**与饱和点。

3. **固定接口上下文（Fixed-Interface Added Context Probes）**  
   - 为剖析跨系统上下文信息的可迁移性，在固定 prompt 接口下，加入来自其他系统（如 KGCompass、ExpeRepair）的额外上下文，对比同样 token 长度的“填充控制”和“难负样本”，判断提升是否源于“信息质量”而非“prompt 增长”。

4. **共同封装器（Common-Wrapper Oracle Check）**  
   - 将所有系统在相同包装器和相同 oracle 构建器下运行，以分析系统改进是否依赖接口设计而非算法本身。

5. **残余前沿分析（Residual Frontier Analysis）**  
   - 对比多系统联合后仍未修复的实例集合，识别这些“共同失败点”在仓库分布、错误类型和可恢复性上的特征。

## 六、主要发现与结论

### 1. Oracle 定位带来的增益有限
- 三个系统的成功率由约 28–33% 提升到约 39–43%，**仍低于 50%**，说明定位虽重要但并不足以解决主要瓶颈。  
- 约 60% 的收益来自已提交补丁的执行成功率提升，而非减少“无补丁”情况。  

### 2. 搜索多样性有益但饱和
- 在 10 个补丁采样下，**Best-of-5 就能恢复约 86% 的提升空间**，后续增加候选并无显著收益。  
- 系统早期样本顺序存在改进空间，说明排序与候选多样化仍是优化点。

### 3. 上下文附加信息确有价值
- 信息量丰富的跨系统上下文明显优于同 token 长度的填充与负样本控制。  
- 但“更多上下文”并不总是更好——长提示可能削弱匹配系统自身结构。  

### 4. 简单 Prompt 融合效果有限
- 将多系统上下文直接拼接（UnionContext）**并未超越最佳单源**，反而导致部分稳定修复下降。  
- 表明改进需要内容级融合，而非文本拼接。

### 5. 残余前沿依旧庞大
- 三系统共解 300 任务中的 199 个，留有 **101 个未解决实例**。  
- 最佳探针仅新增 6 个前沿实例；即使所有探针联合，也仅恢复 14/101。  
- 前沿主要集中在 **sympy** 与 **django** 仓库，主要失败类型包括：
  - 错误的 API/运行异常（25%）；
  - 不完整修复（21%）；
  - 缺少导入或名称错误（20%）。
