## 摘要
自动化程序修复（APR）在处理复杂逻辑错误和静默失败时表现困难。目前基于大型语言模型（LLM）的 APR 方法主要是静态的，依赖源代码和基本的测试输出，这些方法无法准确捕捉复杂的运行时行为和动态数据依赖。尽管引入诸如执行轨迹等运行时证据可以揭示具体的状态转换，但单个 LLM 在孤立地解释这些信息时经常会对特定假设过拟合，生成的补丁只是偶然满足测试，而非真正纠正逻辑。因此，运行时证据应当作为客观约束，而不是单纯的额外输入。我们提出了 TraceRepair，这是一种多智能体框架，利用运行时事实作为共享约束，用于补丁验证。一个探测智能体捕获关键变量的执行快照，以形成客观的修复基础。同时，一个由专门智能体组成的委员会交叉验证候选补丁，以暴露不一致性并进行迭代优化。在 Defects4J 基准上评估结果显示，TraceRepair 成功修复了 392 个缺陷，显著优于现有的基于 LLM 的方法。大量实验表明，该方法在新构建的最新漏洞数据集上表现出更高的效率和强泛化能力，并且性能提升来源于动态推理而非记忆。

## Abstract
Automated Program Repair (APR) struggles with complex logic errors and silent failures. Current LLM-based APR methods are mostly static, relying on source code and basic test outputs, which fail to accurately capture complex runtime behaviors and dynamic data dependencies. While incorporating runtime evidence like execution traces exposes concrete state transitions, a single LLM interpreting this in isolation often overfits to specific hypotheses, producing patches that satisfy tests by coincidence rather than correct logic. Therefore, runtime evidence should act as objective constraints rather than mere additional input. We propose TraceRepair, a multi-agent framework that leverages runtime facts as shared constraints for patch validation. A probe agent captures execution snapshots of critical variables to form an objective repair basis. Meanwhile, a committee of specialized agents cross-verifies candidate patches to expose inconsistencies and iteratively refine them. Evaluated on the Defects4J benchmark, TraceRepair correctly fixes 392 defects, substantially outperforming existing LLM-based approaches. Extensive experiments demonstrate improved efficiency and strong generalization on a newly constructed dataset of recent bugs, confirming that performance gains arise from dynamic reasoning rather than memorization.

- 如何将**运行时执行轨迹（execution traces）**作为可验证的约束，引导 LLM 做出符合程序真实逻辑的修复决策。
- 如何通过**多智能体协作机制（multi-agent debate）**减少单一 LLM 的推理偏误和幻觉（hallucination）。

## 二、方法论与核心技术细节

### 总体框架：TraceRepair
TraceRepair 是一种**基于运行时证据驱动的多智能体自动程序修复系统**，其核心思想是：
> 将运行时执行轨迹视为“客观事实约束”，并让多个智能体以辩论形式交叉验证补丁，从动态证据中推导真实错误逻辑。

### 三个阶段的算法流程
1. **诊断探测（Phase 1: Diagnostic Probing）**
   - “Probe Agent”自动在可疑区域注入日志语句。
   - 运行程序收集变量的运行时快照（state transitions），得到执行轨迹 τ。
   - 动态约束格式化为 `{(时间t, 变量v, 值val)}`。
   - 实施安全注入策略（try-catch 包裹、AST 回滚机制、防止编译失败）。
   - 采用“尾部偏置截断 (Tail-Biased Truncation)”确保在 LLM 上下文窗口限制内优先保留关键的终止阶段状态。

2. **多策略辩论（Phase 2: Multi-Strategy Debate）**
   - 初始化三个不同修复策略代理：
     - **Defensive Agent**：强调健壮性与边界防护。
     - **Causal Agent**：依据轨迹分歧点推断算法性错误。
     - **Semantic Agent**：校验语义与API规范一致性。
   - 各代理生成候选补丁集（Best-of-N selection），并根据运行时轨迹验证逻辑一致性。
   - 辩论过程包括“生成 → 互相批评 → 更新”循环。若某个补丁满足测试，则提前终止。
   - 公式建模：
     - 生成批判条件：`k = F_crit(h_atk, h_def, τ)`，当补丁与轨迹不一致时产生有效批评。
     - 更新规则：`h_new = F_refine(h_def, K, τ)`，基于批评集合修正补丁。

3. **归纳综合（Phase 3: Arbitration & Synthesis）**
   - “Judge Agent”作为仲裁者整合所有辩论信息（历史提案、批评记录、轨迹）。
   - 基于运行时事实剔除逻辑冲突，融合各策略的局部修复，生成最终补丁 `P_final`。
   - 通过沙箱执行验证最终补丁是否通过测试套件。
