## 摘要
在代码库级自动程序修复（APR）中，越来越多的方法将增强定位（localization）视为提升修复效果的主要途径。本文提出一个更具针对性的问题：当定位能力增强后，哪些定位之后的杠杆仍可带来可恢复的增益，哪些受制于我们的协议而界限分明，又存在哪些残余前沿？我们在 SWE-bench Lite 上，针对三种具有代表性的代码库级 RAG-APR 范式（Agentless、KGCompass 和 ExpeRepair）研究这一问题。我们的实验协议结合了 Oracle Localization、池内 Best-of-K、固定接口新增上下文探针（附带每种条件相同 token 填充控制和同库硬负样本）以及通用封装器 oracle 检查。Oracle Localization 可提升所有三个系统的性能，但 Oracle 的成功率仍低于 50%。在采样的 10 个补丁池中，额外候选的多样性仍能提供帮助，但这一空间很快饱和。在两个固定接口下，大多数信息量较高的新增上下文条件仍优于其相应的匹配控制。通用封装器检查显示系统响应差异：在使用相同封装器时，KGCompass 和 ExpeRepair 的增益仍显著，而 Agentless 随构建器选择变化更大。提示级融合（prompt-level fusion）仍保留较大的残余前沿：最佳固定探针仅比原生三系统的 Solved@10 联合总数多解决 6 个实例。总体而言，定位强度、受限搜索空间、证据质量和接口设计共同塑造了代码库级修复的结果。

## Abstract
Repository-level automated program repair (APR) increasingly treats stronger localization as the main path to better repair. We ask a more targeted question: once localization is strengthened, which post-localization levers still provide recoverable gains, which are bounded within our protocol, and what residual frontier remains? We study this question on SWE-bench Lite with three representative repository-level RAG-APR paradigms, Agentless, KGCompass, and ExpeRepair. Our protocol combines Oracle Localization, within-pool Best-of-K, fixed-interface added context probes with per-condition same-token filler controls and same-repository hard negatives, and a common-wrapper oracle check. Oracle Localization improves all three systems, but Oracle success still stays below 50%. Extra candidate diversity still helps inside the sampled 10-patch pools, but that headroom saturates quickly. Under the two fixed interfaces, most informative added context conditions still outperform their own matched controls. The common-wrapper check shows different system responses: under a common wrapper, gains remain large for KGCompass and ExpeRepair, while Agentless changes more with builder choice. Prompt-level fusion still leaves a large residual frontier: the best fixed probe adds only 6 solved instances beyond the native three-system Solved@10 union. Overall, stronger localization, bounded search, evidence quality, and interface design all shape repository-level repair outcomes.

## 一、核心问题与整体含义

- **研究动机：**  
  当前仓库级自动程序修复（Repository-level Automated Program Repair, APR）系统普遍认为，“强化故障定位（localization）”是提升修复效果的主要突破口。然而，定位准确后，修复性能仍存在明显上限。  
  作者提出新的研究问题：  
  > 当定位能力增强后，APR 系统还能从哪些“定位之后的因素”中获得可恢复的性能增益？这些增益的边界在哪里？还存在哪些残余的难以突破的前沿问题？

- **研究背景：**  
  代码库级 APR 面临的核心挑战在于：
  - 处理范围远超单文件，长上下文和复杂依赖使故障定位和补丁生成难度显著增加；
  - 当前 RAG（Retrieval-Augmented Generation）式修复系统将检索到的上下文作为主要提升手段；
  - 现有研究往往比较“端到端成功率”，却未细分定位、上下文构建、生成与验证等阶段的贡献。

- **研究目标：**
  - 建立一个**受控的后定位分析框架**；
  - 定量测量在“强化定位后”各种环节的增益来源、边界与残余瓶颈；
  - 揭示代码库级 RAG-APR 在当前设计下的真正性能限制。

## 二、方法论与技术设计

### 1. 总体思路
作者设计了一个**控制式实验协议**（controlled protocol），通过多层干预分离定位后的关键因素，包括：
- **Oracle Localization（预言定位）**：提供真实 fault span（文件+行号）；
- **Best-of-K**：测量在给定样本池内的潜在“搜索可恢复空间”；
- **固定接口上下文探针（Fixed-Interface Added Context Probes）**：在不改变系统结构的前提下，控制性加入额外上下文；
- **通用封装器检查（Common-wrapper oracle check）**：比较不同系统在共同接口下的表现。

### 2. 技术流程
实验流程基于三阶段 RAG-APR 模型框架：
1. **Localization（定位）**：缩小候选范围；
2. **Context Construction（上下文构建）**：将若干候选代码片段与任务描述组合；
3. **Repair（修复）**：模型根据上下文生成 patch。

核心分析步骤包括：
- 以 SWE-bench Lite 中的真实 patch 为 “Oracle Gold Span”，注入到各系统本地 pipeline；
- 对各系统维持相同 prompt 接口，仅在注入点前后干预；
- 通过多次采样和不同控制条件（填充控制、硬负样本）识别出真正有用的增益来源；
- 统计成对的性能变化、实例重叠与残余未解集合。

### 3. 三个对比系统
- **Agentless**：阶段化定位与压缩库视图，依赖测试反馈；
- **KGCompass**：构建知识图谱进行方法级关联推理；
- **ExpeRepair**：利用双记忆结构检索历史示例和语义经验。

## 六、主要结论与发现

1. **强化定位带来有限提升：**
   - Oracle 提升成功率约 10–13 个百分点，但最高仅达 43% 左右；
   - 表明仅靠定位强化不足以突破修复瓶颈。

2. **搜索多样性带来的“可恢复空间”有限：**
   - Best-of-10 下提升至 51%–56%，主要收益集中在前 5 个候选内；
   - 采样多样性快读饱和。

3. **上下文信息的“结构性”远比长度重要：**
   - 信息性上下文明显优于同 token 填充或硬负样本；
   - 简单的 prompt 拼接（UnionContext）并不总优于单一来源。

4. **系统响应依赖接口：**
   - Agentless 对构建器差异更敏感；
   - KGCompass、ExpeRepair 在共享封装器下依然保持增益。

5. **残余前沿依然庞大：**
   - 三系统合计 Best-of-K 仍有 101/300 未解；
   - 即便合并所有探针与 oracle 仍仅追回 14 个；
   - 主要集中于 `sympy`、`django`，错误类型以 API 误用、未完整修复、缺少 import 为主。
