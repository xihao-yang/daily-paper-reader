## 摘要
自动化程序修复（Automated Program Repair, APR）受益于大型语言模型（Large Language Models, LLMs）的代码理解与生成能力。现有的基于反馈的APR方法通过测试执行反馈迭代优化候选补丁，并已显示出良好的效果。然而，大多数方法依赖于结果层面的失败症状（如堆栈跟踪），这些信息仅展示了故障的外在表现，而无法揭示对根因分析至关重要的中间运行时状态。因此，LLM往往在缺乏充分运行时证据的情况下推断缺陷原因，导致错误补丁。为了解决这一局限，我们提出了DebugRepair——一种面向LLM的自导式调试APR框架。DebugRepair通过模拟调试收集的中间运行时证据增强补丁优化过程。该框架由三个组件构成：测试语义纯化、模拟插桩以及基于调试驱动的对话式修复。这三部分共同作用，减少测试上下文噪声，通过带有规则回退机制的目标化调试语句收集运行时跟踪，并利用以往尝试和新观测的运行时状态逐步优化候选补丁。我们在Java和Python的三个基准上评估了DebugRepair。实验结果表明，DebugRepair在15种方法中取得了当前最优性能。基于GPT-3.5，它在Defects4J上正确修复了224个缺陷，较先前最优的LLM方法提升了26.2%；基于DeepSeek-V3，它正确修复了295个Defects4J缺陷，比第二优秀基线多修复59个。在五个额外的主干LLM上，DebugRepair相较于基础设置的修复性能提升达51.3%。消融研究进一步验证了各组件的有效性。

## Abstract
Automated Program Repair (APR) has benefited from the code understanding and generation capabilities of Large Language Models (LLMs). Existing feedback-based APR methods iteratively refine candidate patches using test execution feedback and have shown promising results. However, most rely on outcome-level failure symptoms, such as stack traces, which show how failures are observed but fail to expose the intermediate runtime states critical for root-cause analysis. As a result, LLMs often infer bug causes without sufficient runtime evidence, leading to incorrect patches. To address this limitation, we propose DebugRepair, a self-directed debugging framework for LLM-based APR. DebugRepair enhances patch refinement with intermediate runtime evidence collected through simulated debugging. It consists of three components: test semantic purification, simulated instrumentation, and debugging-driven conversational repair. Together, they reduce noisy test context, collect runtime traces through targeted debugging statements with rule-based fallback, and progressively refine candidate patches using prior attempts and newly observed runtime states. We evaluate DebugRepair on three benchmarks across Java and Python. Experiments show that DebugRepair achieves state-of-the-art performance against 15 approaches. With GPT-3.5, it correctly fixes 224 bugs on Defects4J, outperforming prior SOTA LLM-based methods by 26.2%. With DeepSeek-V3, it correctly fixes 295 Defects4J bugs, surpassing the second-best baseline by 59 bugs. Across five additional backbone LLMs, DebugRepair improves repair performance by 51.3% over vanilla settings. Ablation studies further confirm the effectiveness of all components.

- **研究领域**：自动化程序修复（Automated Program Repair, APR），旨在自动生成补丁修复软件缺陷，减少人工调试工作量。

  - 当前主流的LLM-based APR框架多为**基于反馈修复（feedback-based APR）**，其通过读取测试用例执行的结果（如错误日志、堆栈跟踪）来逐步修复代码。
  - 但这些“结果层级（outcome-level）”的反馈仅揭示程序失败的表象，而**无法给出中间运行时状态（runtime states）**，LLM只能基于表征性错误信息推断缺陷原因，易导致补丁错误。
- **研究动机**：
  - 模拟人类开发者的调试思维：人类在修复错误时往往会主动插入打印语句以观察运行中变量状态，从而定位根因。
  - 因此，论文希望让LLM具备“自我调试能力”，在修复阶段主动生成并利用运行时信息进行补丁优化。

## 二、方法论：DebugRepair 框架

### 1. 核心思想
> **DebugRepair = 自导调试（self-directed debugging） + LLM驱动修复**

- 通过模拟调试过程，让LLM不仅依靠异常输出，还能采集和理解中间运行状态。
- 包含三大核心组件，形成一个**贯穿式自循环修复流程**：
  1. **测试语义净化（Test Semantic Purification）**  
     - 通过静态程序切片，从失败测试用例中提取最小语义相关子集，保留触发错误的最小语境。  
     - 过滤无关测试场景以减少调试阶段的噪声日志。  
     - 核心算法：基于AST分析变量依赖关系、数据流依赖与隐式状态修改，多轮反向遍历确保捕获到隐式别名影响。
  2. **模拟插桩（Simulated Instrumentation）**  
     - 引导LLM或基于规则的方法自动插入调试语句（如`System.out.println`），记录关键变量取值与控制流条件。  
     - 关键机制：
        - LLM插桩策略（context-aware insertion）；  
        - 编译与语义一致性检测（需保证移除打印语句后的代码逻辑不变）；  
        - 若LLM插桩失败，回退到**基于规则的插桩算法（Algorithm 2）**，确保鲁棒性。
     - 最终执行插桩后的程序，得到结构化的**运行时追踪（runtime trace）**。
  3. **调试驱动对话修复（Debugging-Driven Conversational Repair）**  
     - 以运行时轨迹为语义反馈，形成内外两层迭代机制：
       - 外层：多轮调试会话，每轮重新收集trace。
       - 内层：修复回合循环，LLM基于本轮trace与历史补丁反馈生成新补丁。
     - 每次失败的补丁都会生成对应的错误描述与执行日志，构成**对话式反馈链**。
     - 若达到多轮上限仍无成功补丁，则重启新一轮调试流程。
  4. **扩展：补丁增强（Patch Augmentation）**
     - 为防止测试集过拟合，在生成首个通过测试的“可疑补丁”后，指导LLM生成逻辑等价的变体，用于提升语义正确率。

  - 最多6个调试会话（N_session = 6）；
  - 每会话4轮修复迭代（K_round = 4）；

  - *检索式*：RepairAgent、ReinFix  
  - *混合式*：ThinkRepair

### 4. 计算资源
