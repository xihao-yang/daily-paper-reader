## 摘要
大型语言模型正在迅速演化为能够执行端到端网页编程的交互式编码代理，但现有的基准测试仅评估了这项能力的狭窄部分，通常局限于基于文本的生成及静态正确性指标，从而忽视了视觉保真度、交互质量和代码库级推理能力。我们提出了 WebCompass，一个多模态基准，用于对网页工程能力进行统一的全生命周期评估。鉴于现实世界的网页编程是生成、编辑与修复的迭代循环，WebCompass 覆盖了三种输入模态（文本、图像、视频）和三种任务类型（生成、编辑、修复），形成七种任务类别，以反映专业工作流程。我们通过一个多阶段、有人参与的流程，整理了包含 15 个生成领域、16 种编辑操作类型和 11 种修复缺陷类型的任务实例，并在容易/中等/困难三个级别上标注难度。在评估中，我们针对编辑与修复任务采用基于检查清单的 LLM-as-a-Judge 协议，并为生成任务提出一种新颖的 Agent-as-a-Judge 范式，该方法在真实浏览器中自主执行生成的网站，通过模型上下文协议（Model Context Protocol，MCP）探索交互行为，并迭代生成针对性的测试用例，从而逼近人工验收测试。我们评估了代表性的闭源与开源模型，观察到：（1）闭源模型仍显著更强且更加均衡；（2）编辑与修复展现出不同的难度特征，其中修复在保持交互性方面表现更好，但执行仍具挑战；（3）美学表现是最持久的瓶颈，尤以开源模型为甚；（4）框架选择对结果有实质性影响，其中 Vue 一贯更具挑战性，而 React 和 Vanilla/HTML 的表现则取决于任务类型。

## Abstract
Large language models are rapidly evolving into interactive coding agents capable of end-to-end web coding, yet existing benchmarks evaluate only narrow slices of this capability, typically text-conditioned generation with static-correctness metrics, leaving visual fidelity, interaction quality, and codebase-level reasoning largely unmeasured. We introduce WebCompass, a multimodal benchmark that provides unified lifecycle evaluation of web engineering capability. Recognizing that real-world web coding is an iterative cycle of generation, editing, and repair, WebCompass spans three input modalities (text, image, video) and three task types (generation, editing, repair), yielding seven task categories that mirror professional workflows. Through a multi-stage, human-in-the-loop pipeline, we curate instances covering 15 generation domains, 16 editing operation types, and 11 repair defect types, each annotated at Easy/Medium/Hard levels. For evaluation, we adopt a checklist-guided LLM-as-a-Judge protocol for editing and repair, and propose a novel Agent-as-a-Judge paradigm for generation that autonomously executes generated websites in a real browser, explores interactive behaviors via the Model Context Protocol (MCP), and iteratively synthesizes targeted test cases, closely approximating human acceptance testing. We evaluate representative closed-source and open-source models and observe that: (1) closed-source models remain substantially stronger and more balanced; (2) editing and repair exhibit distinct difficulty profiles, with repair preserving interactivity better but remaining execution-challenging; (3) aesthetics is the most persistent bottleneck, especially for open-source models; and (4) framework choice materially affects outcomes, with Vue consistently challenging while React and Vanilla/HTML perform more strongly depending on task type.

## 一、研究问题与背景

- **核心问题**：当前大语言模型（LLMs）已能执行网页端到端生成与开发任务，但现有评测基准仍以**纯文本输入和静态正确性指标**（如 HumanEval 的 pass@k）为主，无法反映：
  - **视觉保真度**（Web UI 美观与一致性）；
  - **交互性与可用性**（真实网站的点击、滚动、导航响应）；
  - **代码库级推理与演化能力**（如跨文件编辑与修复）。
- **研究动机**：构建一个能覆盖**“网页工程完整生命周期”**的评估体系，从生成、编辑到修复三个阶段，反映真实开发流程中的多模态应用能力。

## 二、方法论与技术方案

### 1. 核心思想
- 提出 **WebCompass**：一个统一的**多模态网页编程评测框架**。
- 支持三种输入模态：
  - 文本（Text）；
  - 图片（Image / Screenshot）；
  - 视频（Video / Interaction Recording）。
- 与三种任务类型对应：
  - **生成（Generation）**：从需求 / 画面 / 视频生成可运行网站；
  - **编辑（Editing）**：在现有项目中按指令修改；
  - **修复（Repair）**：识别并修正前端缺陷。

### 2. 七类任务组合
- Text-Guided Generation  
- Vision-Guided Generation  
- Video-Guided Generation  
- Text-Guided Editing  
- Vision-Guided Editing  
- Diagnostic Repair（文本描述缺陷）  
- Visual-Diagnostic Repair（图像描述缺陷）

### 3. 数据构建与采样流程
- 采用**人工参与与多阶段筛选**（human-in-the-loop）流程：
  - 从 WebGen-Bench、ArtifactsBench、V0、BigCode Arena 等源收集初始样本；
  - 使用 **BGE-M3 向量嵌入 + k-means 聚类** 去重；
  - 利用 LLM 执行类别与难度标注；
  - 人工最终审核，确保任务的**完整性、一致性与难度多样性**。
- 编辑与修复基于 50 个高质量网页原型，通过三阶段筛选（长度过滤 → GPT-4o 代码审查 → 人工扩展）。
- 修复任务使用**逆向注入法**：由干净页面引入 11 类真实缺陷，再要求模型修复。

### 4. 评价方法
两种互补的评审机制：

#### （1）LLM-as-a-Judge（用于编辑 / 修复）
- 基于**检查清单式评分**，输出三维指标：
  - 编辑：  
    - *Instruction Targeting*（指令定位）、  
    - *Feature Integrity*（功能正确性）、  
    - *Style Conformance*（视觉风格一致性）。
  - 修复：  
    - *Root-Cause Targeting*（根因定位）、  
    - *Interaction Integrity*（交互完整性）、  
    - *Reference Fidelity*（修正后视觉匹配度）。
- 自动收集**修改前后截图与日志**，并用头部模型（Claude-Opus-4.5）实现一致评分。

#### （2）Agent-as-a-Judge（用于生成）
- 通过**浏览器自动执行与检查表驱动脚本**实现：
  - 启动网页 → 交互操作（点击、滚动、填写表单等）；
  - 动态生成并执行测试用例；
  - 使用 MCP (Model Context Protocol) 捕捉 DOM、日志、截图；
  - 三维评分：  
    - *Runnability（可运行性）*、  
    - *Spec Implementation（功能实现度）*、  
    - *Design Quality（设计质量）*。
- 采用**谐波平均公式**综合评分，防止单维高分掩盖其他失败。

## 六、主要结论与发现

1. **总体性能差距显著**：  
   - 闭源模型平均高出开源模型约 25～30 分；
   - Claude-Opus-4.5、Gemini-3-Pro 在综合得分上领先（≈67 分）。

2. **任务间差异明显**：  
   - 生成任务在「可运行性」最好；
   - 编辑任务功能正确性下降；
   - 修复任务交互保持性高但定位难度大。

3. **视觉维度是长期瓶颈**：  
   - 即使顶级模型在视觉美观或保真度上仍低于 65 分。

4. **框架影响显著**：  
   - Vue 难度最高；
   - Vanilla HTML 在生成和编辑中更稳定；
   - React 在修复任务中略优。

5. **难度分级与一致性分析**：  
   - 模型随任务复杂度增加性能显著下降；
   - 高难度任务落差最大体现在功能实现维度。

6. **错误模式分析**：  
   - 主要错误为功能缺失（Feature Missing）和资源加载失败；
   - 修复任务中常见“未修复原缺陷”或“过度修改”问题；
   - 视觉错误集中于布局与配色不一致。
