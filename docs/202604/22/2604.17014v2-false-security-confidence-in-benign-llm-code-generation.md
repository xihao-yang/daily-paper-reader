## 摘要
以往研究表明，在以威胁为导向的环境中，当对代码生成代理和自动修补工作流施加对抗性或隐式通道时，会系统性地产生功能上正确但存在漏洞的输出。本文提出一种互补且不同的框架：虚假安全信心（False Security Confidence，FSC），从测量优先的视角在普通、非攻击场景的生成任务中研究相同的表面现象。我们的关注点不在于攻击是否能生成此类输出，而在于在没有显式攻击压力的情况下，它们出现的频率和形式，以及传统的功能性评估是否能够可靠地检测到它们。我们将FSC率形式化为功能正确输出集中安全失效的普遍性，并将其与先前的联合功能-安全度量（如SAFE）以及以结果驱动的评估框架（如CWEval）区分开来。同时，我们定义了一个三生态系统任务视图，用于研究FSC在通用编程、部署环境任务和安全显式编程中的表现，并识别出“FSC难”这一具有实际意义的精化层，在此层中静态分析器会遗漏那些仍可动态触发的漏洞。本技术报告的范围特意限定为框架阐述，而非完整的实证论文：其目的在于建立术语、测量边界以及研究设计承诺，以用于后续的大规模评估。

## Abstract
Prior work has demonstrated that functionally correct yet vulnerable outputs arise systematically in threat-oriented settings, where adversarial or implicit channels are used to induce security failures in code agents and automated patching workflows. This note introduces a complementary but distinct framing: False Security Confidence (FSC), which studies the same surface phenomenon from a measurement-first perspective in ordinary, non-attack-framed generation tasks. Our interest is not in whether attacks can produce such outputs, but in how frequently and in what forms they appear absent explicit attack pressure, and whether conventional functional evaluation reliably detects them. We formalize FSC rate as the prevalence of security failure within the set of functionally correct outputs, distinguish it from prior joint functional-security metrics such as SAFE and outcome-driven evaluation frameworks such as CWEval, define a three-ecosystem task view for studying how FSC manifests across general-purpose programming, deployment-context tasks, and security-explicit programming, and identify FSC-hard as a practically important refinement layer in which static analyzers miss vulnerabilities that remain dynamically triggerable. This technical report is intentionally scoped as a framework statement rather than a full empirical paper: its purpose is to establish terminology, measurement boundaries, and study design commitments for subsequent large-scale evaluation.

## 一、核心问题与整体含义

- **研究背景**：近年来，大型语言模型（LLMs）在代码生成任务中取得显著进展。研究者通常通过**功能正确性**来评估生成质量，即生成代码是否能通过单元测试或产生预期输出。  
- **问题提出**：然而，已有研究发现功能正确并不等于安全正确，LLMs 在无攻击场景下也可能生成带安全漏洞的代码，如不安全的输入处理、明文凭证存储等。  
- **研究动机**：作者指出这种现象易导致“**虚假安全信心**”（False Security Confidence，简称 FSC）——当生成代码显得功能正常时，人们误以为它安全可靠，从而过度信任其质量。  
- **论文定位**：本研究并非从攻击或漏洞挖掘的角度出发，而是**从测量和评估角度**系统定义并量化这种虚假安全信心现象，为后续大规模研究建立概念框架与术语体系。

## 二、方法论与核心思想

### 2.1 FSC 概念的定义
- **基本符号**：  
  - \( FCR(s) \in \{0, 1\} \)：样本 \( s \) 的功能正确性指标。  
  - \( SSR(s) \in \{0, 1\} \)：样本 \( s \) 的安全成功指标。  
- **判定条件**：当 \( FCR(s)=1 \) 且 \( SSR(s)=0 \) 时，即为 FSC 样本。  
  → 表示该代码在功能上正确，但存在安全漏洞。

### 2.2 FSC-rate：条件安全失效率
- 定义公式：

  \[
  \text{FSC-rate}(S) = \frac{| \{s \in S : FCR(s)=1 ∧ SSR(s)=0\} |} {| \{s \in S : FCR(s)=1\} |}
  \]
- **核心思想**：以功能正确样本为分母，度量其内部的安全失效比例。  
  - 区分于 SAFE 等联合功能-安全指标（将两者综合为单一分数）。  
  - FSC-rate 专注于**“看似成功但实则不安全”**的风险部分。

### 2.3 三生态系统任务视图
- 作者根据任务场景将 FSC 研究划分为三类生态系统：
  1. **通用编程任务**（General-purpose）：算法实现类问题，漏洞可能被功能测试掩盖。  
  2. **部署上下文任务**（Deployment-context）：涉及认证、密钥、配置等，漏洞受环境影响。  
  3. **安全显式编程**（Security-explicit）：任务文本中包含安全要求（如加密、验证），但模型仍可能生成潜在漏洞实现。

- **意义**：帮助研究者识别不同场景下“虚假安全信心”的成因和分布模式。

### 2.4 FSC-hard：难检漏洞层
- 定义：FSC-hard 是 FSC 子集，其中漏洞**静态分析无法发现但可通过动态或语义验证触发**。
- 目的：识别对安全工具最具欺骗性的情况，揭示现有评估体系在低可见漏洞上的盲区。

### 2.5 评估与验证原则
- 建议使用**分层验证栈**：
  - 功能验证：基于任务特定 oracle。  
  - 安全验证：结合静态分析、动态测试及人工审查。  
- 动态验证（例如触发真实漏洞）是 FSC-hard 样本的必要条件。
