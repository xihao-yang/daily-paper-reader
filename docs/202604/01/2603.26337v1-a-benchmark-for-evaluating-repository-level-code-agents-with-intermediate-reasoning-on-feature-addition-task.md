## 摘要
仓库级代码智能体在真实世界的功能添加任务中展现出强大潜力，因此，对其能力进行可靠评估变得愈发重要。然而，现有基准主要将这些智能体视为黑箱，根据最终测试正确性进行评估，无法深入了解其推理过程及失败原因。为解决这一局限，我们提出了 RACE-bench，这是一种用于在仓库级功能添加任务中评估代码智能体的推理增强型基准。RACE-bench 包含来自 12 个开源仓库的 528 个真实功能添加实例。每个实例都配有可执行的补丁验证，以及涵盖问题理解、文件定位、实现任务和步骤分解的结构化中间推理真值。基于此设计，我们提出了一个双轨评估框架，同时衡量补丁正确性与中间推理质量。我们在 RACE-bench 上评估了三种具有代表性的仓库级代码智能体。在完整基准上，不同智能体的解决率（Resolved Rate）介于 29% 至 70% 之间。推理层面的分析进一步表明，当前智能体在理解高层意图方面表现良好，但在将意图转化为具体实现步骤时其性能显著下降。我们还发现，补丁成功应用但测试失败的案例相较于成功案例展现出更低的推理召回率（下降 35.7%），以及更高的过度预测率（增加 94.1%）。这些发现突出了在评估仓库级代码智能体时，除最终补丁正确性之外，还应深入考察其推理过程质量的重要性。

## Abstract
Repository-level code agents have shown strong promise in real-world feature addition tasks, making reliable evaluation of their capabilities increasingly important. However, existing benchmarks primarily evaluate these agents as black boxes based on final test correctness, providing limited insight into how they reason and where failures arise. To address this limitation, we introduce RACE-bench, a reasoning-augmented benchmark for evaluating code agents on repository-level feature addition tasks. RACE-bench contains 528 real-world feature addition instances from 12 open-source repositories. Each instance is paired with executable patch verification and structured intermediate reasoning ground truth covering issue understanding, file localization, implementation tasks, and step decomposition. Based on this design, we introduce a dual-track evaluation framework that jointly measures patch correctness and intermediate reasoning quality. We evaluate three representative repository-level code agents on RACE-bench. On the full benchmark, Resolved Rates range from 29% to 70% across different agents. Our reasoning-level analysis further shows that while current agents perform well at understanding high-level intent, their performance degrades substantially when translating intent into concrete implementation steps. We also find that apply-success but test-fail cases exhibit lower reasoning recall (35.7% decrease) and higher over-prediction (94.1% increase) compared to successful cases. These findings highlight the importance of evaluating repository-level code agents beyond final patch correctness by examining the quality of their reasoning processes.

## 1. 核心问题与研究动机

### 背景
- 近年来，大型语言模型（LLMs）显著推动了自动化软件工程发展，催生了具备多步推理与工具交互能力的**仓库级代码智能体（Repository-level Code Agents）**。  
- 这些智能体能在真实项目中自动修复 Bug、增加新特性、生成测试用例。
  
### 研究问题
- 现有评测，如 **HumanEval、MBPP、SWE-bench、FEA-bench**，主要通过测试是否“**最终通过单元测试**”来判定智能体性能。
- 然而，这一“黑箱式”评估忽视了智能体在任务中间的推理过程，无法诊断：
  - 错误源自哪一步（如问题理解、文件定位、实现规划）；
  - 智能体是否只是**偶然“蒙对”**而通过测试。
  
### 研究动机
- **目标：** 建立一个可同时衡量**结果正确性**与**中间推理质量**的系统性基准。
- **意义：** 帮助社区识别代码智能体的真实推理能力与局限，为后续改进框架与模型提供可诊断信号。

## 2. 方法论：RACE-bench 的设计思想与关键技术

### 核心思路
构建一个“推理增强型”仓库级评测基准 **RACE-bench（Reasoning-Augmented Code Agent Evaluation）**，在真实软件开发语境下评估智能体的**从理解到实现全过程**。

### 关键组成

1. **数据构成**
   - 来源：12 个活跃维护的开源 Python 仓库。
   - 实例数量：共 **528 个有效功能添加实例**。
   - 每个实例包含：
     - 自然语言功能请求（对应 GitHub Issue）；
     - 可执行补丁验证（Fail-to-Pass / Pass-to-Pass 测试对）；
     - **结构化中间推理真值（Reasoning Ground Truth）**：
       - 问题理解（Concept Explanations, Goal Expectation）
       - 文件定位（File Localization）
       - 代码任务识别（Specific Tasks）
       - 步骤分解（Step Decomposition）

2. **推理真值构建流程**
   - **多阶段过滤 + LLM 辅助分析：**
     - 从 14,333 个 PR 中筛选出 2,931 个功能新增任务；
     - 经过环境验证后保留 580 个；
     - 再经 LLM 审核与人工复核去除有误实例（最终 528）。
   - **Ground Truth 生成工具链：**  
     使用 DeepSeek 模型与静态分析结合生成任务概念、文件、实体与操作步骤标注。  
     以 **测试驱动消融（test-driven ablation）** 判定哪些文件/实体是必需的（is_necessary）。

3. **双轨评测框架（Dual-Track Evaluation Framework）**
   - **轨一（Patch Verification）**：测试最终补丁在 FTP/PTP 用例下是否完全通过。
   - **轨二（Reasoning Assessment）**：利用总结型 LLM 将智能体交互过程转化为结构化推理链并与真值对齐，计算：
     - **召回率 Recall@X**
     - **过度预测率 OverPrediction@X**
     - **Goal Expectation 分数（10分制）**
   - 对齐算法结合**确定性匹配（文件名、实体名、位置）**与**语义匹配（LLM判断）**。

4. **轻量子集 RACE-bench Lite**
   - 100 个实例的代表性子集，用于快速测试和多模型对比。
   - 通过局部搜索算法保证子集的统计特征与全量集一致。

## 3. 实验设计

### 数据与环境
- **Benchmarks：**  
  - 主基准 **RACE-bench (528 实例)**  
  - 子集 **RACE-bench Lite (100 实例)**
- **环境统一：**  
  每个仓库构建独立的 Docker 镜像，包含固定依赖与版本，确保结果复现一致性。

### 参评系统
三种代表性仓库级代码智能体（来自 SWE-bench 排行榜体系）：
1. **AutoCodeRover**（结构化 AST 导航、保守编辑策略）  
2. **TraeAgent**（基于测试驱动迭代与扩增思维的多轮推理）  
3. **mini-SWE-Agent**（轻量版 SWE-Agent 框架，注重推理与执行结合）

### 基础语言模型（Base LLMs）
- **DeepSeek V3.2**（主实验模型）  
- 辅助测试中还使用 **GLM-4.7**、**GPT-5.2** 进行对比。

### 实验指标
- **补丁级**：Resolved Rate、Patch Apply Rate  
- **推理级**：Goal分数、Recall 与 OverPrediction（针对文件、任务、步骤等模块）

## 6. 主要结论与发现

1. **整体性能差异显著：**
   - mini-SWE-Agent → Resolved Rate **70.08%**
   - TraeAgent → **52.65%**
   - AutoCodeRover → **28.79%**
2. **推理层表现呈“瀑布式衰减”：**
   - 意图理解（Goal Score≈9.3/10）优秀；
   - 文件定位 Recall 约 0.8–0.9；
   - 实体任务 Recall 约 0.6–0.7；
   - 步骤分解 Recall 降至 0.3–0.5，OverPrediction 激增。
3. **错误特征：**
   - 应用成功但测试失败的补丁，相对成功案例：
     - 召回率降低 **35.7%**；
     - 过度预测增加 **94.1%**。
4. **推理风格差异：**
   - AutoCodeRover：保守、低过度预测但召回不足；
   - TraeAgent：探索性强、修改范围宽；
   - mini-SWE-Agent：平衡型，覆盖与精度兼优。
5. **模型差异性：**
   - Agent 架构比 Base LLM 对性能影响更大；
   - GPT 系列展现更精准的编辑控制；
   - GLM 目标一致性较弱。
6. **效率观察：**
   - DeepSeek 响应慢（平均 729 秒 / 实例）；
   - GLM 最高效（291 秒 / 实例）；
   - GPT 与 DeepSeek 性能相近但 token 消耗显著更低（降 80%）。

- **双轨评测框架**创新性地结合补丁功能正确性与推理质量；
- **可重现性高**：实例级 Docker 环境，确保完全一致的测试状态；
- **推理结构化设计系统化**：覆盖从意图到实现的四阶段五模块；
- **统计完整性**：528 实例大规模覆盖 + 精心构建轻量版子集；
- **分析深度**：揭示代码智能体“理解强–执行弱”的核心瓶颈；
- **中立客观的LLM-as-Judge方案**，结合语义与规则匹配，减少主观偏差。

## 8. 不足与局限

- **模型覆盖有限：** 仅评测三种 Agent 和三类 LLM；
- **未包含 GPU 或大规模并行评测**，实际环境开销较高；
- **评估依赖 LLM-as-Judge**，仍可能受语义偏差影响；
- **隐性推理难以观察**：智能体内部潜在思考过程无法完全显性化；

- **任务类型单一**（仅功能添加），未涵盖 Bug 修复、重构等其他软件演化任务。
