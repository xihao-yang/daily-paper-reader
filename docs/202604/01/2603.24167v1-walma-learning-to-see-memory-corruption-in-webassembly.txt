## Abstract 

WebAssembly’s (Wasm) monolithic linear memory model facilitates memory corruption attacks that can escalate to cross-site script-ing in browsers or go undetected when a malicious host tampers with a module’s state. Existing defenses rely on invasive binary instrumentation or custom runtimes, and do not address runtime integrity verification under an adversarial host model. We present 

Walma , a framework for WebAssembly Linear Memory Attestation that leverages machine learning to detect memory corruption and external tampering by classifying memory snapshots. We evaluate 

Walma on six real-world CVE-affected applications across three verification backends ( cpu-wasm , cpu-tch , gpu ) and three instru-mentation policies. Our results demonstrate that CNN-based classi-fication can effectively detect memory corruption in applications with structured memory layouts, with coarse-grained boundary checks incurring as low as 1.07 × overhead, while fine-grained mon-itoring introduces higher (1.5 ×–1.8 ×) but predictable costs. Our evaluation quantifies the accuracy and overhead trade-offs across deployment configurations, demonstrating the practical feasibility of ML-based memory attestation for WebAssembly.

This growth poses a significant security challenge because the ecosystem is dominated by code originally written in memory-unsafe languages like C and C++ [16, 19]. Recent studies demon-strate that exploits within Wasm modules can be escalated to attack the host environment [12, 24, 27]. Such attacks are possible because the host implicitly trusts data returned from the Wasm sandbox. A memory corruption bug can enable a data-only attack [20, 21], allowing an adversary to overwrite a trusted data structure with-out violating control-flow integrity (CFI). The Wasm sandbox only isolates the module from the host but does not enforce memory safety within the module’s own monolithic linear memory, which co-locates the stack, heap, and global data. Conversely, this pro-tection is unidirectional: it provides no guarantee for the module’s integrity against a malicious host. In cloud and edge deployments, a privileged adversary controlling the infrastructure can directly tamper with the module’s linear memory to subvert its internal logic, for instance bypassing authorization checks or manipulating computation results before they are returned to the client. Protecting Wasm modules in these scenarios is an open challenge. Mitigations such as ASLR [42] are incompatible with Wasm’s linear memory model, and Wasm’s built-in CFI does not cover data-only attacks [12, 24, 27]. Existing Wasm defenses require source-level modifications, custom compilers, or dedicated hardware [22, 23, 26, 30, 31], or use binary rewriting to detect pre-defined bug pat-terns [25, 52]; none address the malicious host threat model. Moreover, existing control flow attestation (CFA) schemes generate cryptographic logs of the execution path but, as noted in recent literature [5], struggle to detect data-oriented attacks where the logic is subverted but the path remains valid. In this paper, we present Walma , a framework for state-centric runtime verification via linear memory attestation (LMA) . Inspired by the principles of remote attestation [5], we shift the verification focus from static code properties to dynamic memory state. Walma 

operates on the insight that memory corruption events and external tampering leave distinct structural artifacts in the linear memory. We leverage this by treating memory snapshots as images and employing convolutional neural networks (CNNs) [39] to classify runtime integrity. Walma decouples execution from verification, a design that is runtime-agnostic and versatile across threat models. We address two deployment scenarios: (1) protecting sensitive com-putations from untrusted infrastructure by performing verification in a trusted execution environment (TEE), and (2) auditing poten-tially compromised modules in the browser to prevent privilege escalation via XSS. Our evaluation validates Walma as a first step towards con-tinuous memory attestation. We tested Walma on six real-world applications with known CVEs, including the audio encoder flac 

and other libraries representative of the performance-critical code that powers platforms like Google Earth and Twitch.tv. For work-loads with structured memory layouts, such as audio frames and pixel buffers, Walma achieves 100% verdict accuracy. We also demonstrate that a single generalized model can detect corrup-tion across different applications without program-specific train-ing. In terms of performance, we conducted an ablation study across three instrumentation policies: boundary defense (before host interactions), internal logic monitoring (at function entries), and fine-grained validation (every memory write). Boundary de-fense incurs only 7% overhead, making Walma practical for pro-duction deployment. To foster research, we open-source Walma at https://anonymous.4open.science/r/walma-5270. 

## 2 Background 2.1 Wasm Security 

Wasm’s security model [49] is built on three principles: software fault isolation (SFI) [45], control flow integrity (CFI) [2], and code im-mutability. These mechanisms create a strong sandbox that isolates the Wasm module from the host, preventing code-reuse [6, 9, 38] and code-injection attacks [32]. However, Wasm’s monolithic linear memory, a single, contiguous, and resizable array of bytes, remains a single point of failure. It co-locates the stack, heap, and global data without runtime-enforced separation. Unlike native architectures where program state is distributed across registers and memory, Wasm’s entire mutable state resides in linear memory, making it a comprehensive attestation target. This allows vulnerabilities such as buffer overflows to corrupt adjacent data structures, enabling cross-region attacks [12, 24, 27]. As Wasm adoption expands into cloud and edge environments, this architectural weakness becomes increasingly critical: the same memory model that enables high performance also creates a unified attack surface for both memory corruption and external tampering. ## 2.2 Remote Attestation 

Remote attestation enables a trusted verifier to determine the in-tegrity of a remote, untrusted attester , often also referred to as 

prover . While traditional schemes verify static properties like code identity [36, 37], detecting runtime exploits requires monitoring ex-ecution behavior. To this end, control-flow attestation (CFA) mech-anisms [3, 8, 11, 40, 53] cryptographically record the sequence of executed instructions. By comparing this trace against the ap-plication’s control-flow graph, CFA confirms that branches and function calls adhere to legitimate program logic. However, even these advanced methods do not fully cover data-only attacks [5, 20, 21]: an adversary can manipulate decision-making variables within Wasm’s linear memory to subvert logic without diverting the valid control-flow graph. This limitation motivates our state-centric ap-proach, which validates the integrity of the memory footprint itself rather than the execution path. 

## 3 Threat Model 

We consider two deployment scenarios, reflecting established ad-versarial models in cloud [28, 29, 47] and web security [12, 24, 27] Walma : Learning to See Memory Corruption in WebAssembly Conference’17, July 2017, Washington, DC, USA 

research. These scenarios capture the bidirectional nature of Wasm security: the module may be a victim of its environment, or a threat to it. 

## 3.1 Scenario A: Untrusted Cloud Host 

In this setting (Figure 1-A), a user deploys a trusted Wasm workload to a third-party cloud or edge infrastructure. The untrusted Host 

encompasses the OS, hypervisor, and physical administrators [28, 29, 47]. The adversary is a privileged actor (e.g., a malicious admin) with full control over the hardware. The attack goal is to manipu-late the workload’s internal state, such as bypassing access control checks, without crashing the application or violating CFI. Unlike software exploits [6, 9, 32, 38] that break out of a sandbox, this ad-versary attacks externally via the host interface: directly modifying linear memory via system-level privileges (e.g., ptrace or DMA), tampering with data exchanged at the host-module boundary, or controlling execution timing to facilitate race conditions. To de-fend against these attacks, the verifier operates within a trusted execution environment (TEE). Our approach establishes a hardware-rooted trust anchor that validates memory snapshots, ensuring that even a root-privileged host cannot tamper with the module’s state without being detected. This scenario is particularly relevant as enterprises increasingly deploy sensitive Wasm workloads to third-party edge infrastructure [4, 51]. For such high-value targets, the overhead of continuous attestation is negligible compared to the cost of undetected state manipulation. 

## 3.2 Scenario B: Compromised Web Module 

In the browser environment (Figure 1-B), the trust relationship is inverted [12, 24, 27]. The browser forms the trusted computing base, while the Wasm Module is potentially vulnerable or compromised by malicious inputs. The adversary exploits a vulnerability (e.g., memory corruption) within the module to corrupt its internal state, aiming to escalate privileges or hijack the embedded web applica-tion (e.g., via XSS on shared memory). Walma acts as a security monitor: the browser acts as the verifier , periodically checking the module’s memory state to detect corruption before it escapes the sandbox. Given that major web applications like Figma rely on Wasm for performance-critical operations [13], detecting corrup-tion before it can be exploited is essential for protection. In both scenarios, the goal of Walma is to provide continuous attestation of the computation state, detecting integrity violations regardless of whether they originate from within the module or from external tampering. 

## 4 System Design 

The design of Walma addresses the challenge of verifying exe-cution integrity in environments where either the module or the host cannot be trusted. As illustrated in Figure 2, the architecture operates in two phases: an offline training phase to construct a robust classifier from synthetically generated states, and an on-line inference phase where the attester and verifier cooperate to detect corruption at runtime. The decoupled design allows Walma 

to adapt to both threat scenarios described in Section 3 by placing the verifier in the appropriate trust domain. 

## 4.1 Offline Phase: Model Training 

To distinguish between benign and corrupted memory states, Walma 

relies on a supervised learning pipeline (Figure 2, top). 

Dataset Generation. Since valid Wasm memory layouts are highly application-specific, we cannot rely on generic heuristics. Instead, we employ binary instrumentation to inject snapshot hooks at deter-ministic points, followed by coverage-guided fuzzing . This allows us to explore the application’s state space systematically, generating a diverse corpus of memory snapshots that represent real execution paths. Crucially, we use sanitizers (e.g., AddressSanitizer [35]) as a ground-truth oracle during this phase: inputs that trigger san-itizer violations are automatically labeled corrupted , while valid executions are labeled benign .

Image Transformation & Learning. We interpret Wasm’s linear memory as a grayscale image, where byte values correspond to pixel intensities. This is feasible because Wasm’s entire mutable state resides in linear memory, unlike native architectures where state is distributed across registers and memory regions. This trans-formation allows us to leverage convolutional neural networks (CNNs) to detect structural anomalies that manifest as disruptions in the visual texture of the memory layout. Since the model learns the distribution of benign memory states, it can detect deviations regardless of their origin—whether from memory corruption or tampering by a malicious host. We employ two model architectures to balance detection capability against resource constraints: VGG-16 (134M parameters) as a high-capacity baseline, and a custom ResNet (1.9M parameters) optimized for deployment in TEEs and other constrained environments. 

## 4.2 Online Phase: Runtime Attestation 

The runtime architecture (Figure 2, bottom) consists of the attester 

and the verifier . Their placement depends on the threat model: in Scenario A, the verifier resides in a TEE to protect against a malicious host; in Scenario B, the verifier runs on the trusted host to monitor a potentially compromised module. 

4.2.1 The Attester. The attester operates on the untrusted host. It executes a Wasm binary instrumented to invoke a snapshot function at deterministic boundaries. When triggered, the attester captures the linear memory and performs a lightweight run-length encoding (RLE) compression. This step is critical to address the bandwidth bottleneck inherent in remote attestation [5]; by stripping zero-byte runs (common in sparse Wasm memory), we significantly reduce the data transfer cost before transmission to the verifier. 

4.2.2 The Verifier & Verification Architectures. The verifier is the root of trust, responsible for reconstructing the memory image, performing inference, and issuing a verdict. To support the diver-gent trust boundaries defined in Section 3, we developed three verification backends: 

In-Runtime ( cpu-wasm ). The verifier executes directly inside the Wasm runtime using WASI-NN. This configuration is designed for Scenario A, where the cloud host is untrusted. By moving verifica-tion into the TEE, we avoid transmitting large memory snapshots externally. ## 4.3 Verdict Aggregation 

Single-snapshot classification can be noisy due to transient valid states that visually resemble corruption. To mitigate false positives, 

Walma implements a verdict aggregation module which applies a sliding window mechanism. Sequential classifications are aggre-gated to form a final execution verdict. An execution is flagged as malicious only if a threshold of snapshots within the window are classified as corrupted , ensuring robust detection even in the presence of minor inferential noise. 

## 5 Implementation 

We implemented Walma as an end-to-end pipeline comprising 2088 

lines of Rust for runtime components and Wasm module instru-mentation, and 657 lines of Python for ML infrastructure. 

Instrumentation Policies: We use Walrus [46] to instrument Wasm binaries without modifying the source code. To balance per-formance and security visibility, we implemented three policies. 

Import Function triggers snapshots before host calls; this acts as a boundary defense, preventing the host from processing corrupted data residing in linear memory, which mitigates most attacks [24, 27]. Local Function instruments Wasm functions entries to approxi-mate control-flow monitoring. Finally, Memory Instruction triggers verification before every write operation (e.g., i32.store ), provid-ing fine-grained attestation of state mutations. 

Fuzzing & Data Generation: We integrated LibAFL [14] with the runtime to generate training data, using AddressSanitizer [35] as an oracle to label execution states as corrupted or benign . We constructed program-specific datasets, which allow the classifier to learn the unique memory layout and allocation patterns of a specific application, as well as a generalized dataset pooled from diverse applications to test cross-program transferability. 

Runtime & Verifier Backends: The attester extends the Wasm runtime Wasmtime with snapshotting and RLE compression. The verifier supports three backends. cpu-wasm runs directly inside the runtime via WASI-NN [48] (OpenVINO); by keeping data within the sandbox, it minimizes the TCB for Scenario A . cpu-tch uses 

LibTorch for native execution, optimizing performance for the trusted host in Scenario B . Finally, the GPU backend uses Torch-Script to offload inference; while providing high throughput, it incurs latency from PCIe transfers. For secure cloud deployment, we package the cpu-wasm pipeline within Gramine [1] to execute inside Intel SGX enclaves.

We evaluated Walma to answer two key questions: (1) Can visual analysis of linear memory reliably distinguish corrupted states from benign execution, even across different applications? (2) Is the runtime cost permissible for production deployment in latency-sensitive environments?

## 7 Discussion & Limitations 

Verdict Aggregation & Robustness. Individual snapshot classifi-cations can be noisy, but Walma mitigates this through temporal verdict aggregation. By analyzing sequences of memory states, the system absorbs transient misclassifications without false alarms, achieving 100% verdict accuracy on targets like flac even when individual snapshot accuracy is lower. 

Performance Trade-offs. The runtime overhead of Walma is highly dependent on the instrumentation granularity. Our ablation study demonstrates that for coarse-grained policies (e.g., import checks), the relative overhead is efficient at 1.07 × (7%). Even for fine-grained monitoring of internal function calls or memory instructions, the overhead remains acceptable for security-critical applications, rang-ing from 1.50 × (50%) to 1.82 × (82%). The tunability of Walma al-lows users to balance security visibility against performance. For integrity-critical deployments, this overhead is a reasonable price for continuous assurance. 

Deployment & Architecture. While Walma supports TEE-based verification for high assurance, our results highlight a critical ar-chitectural insight: for lightweight models like our Custom ResNet, data transfer costs dominate inference time. Consequently, in-runtime execution ( cpu-wasm ) proved significantly faster than external GPU offloading. This suggests that future secure runtime designs should prioritize maximizing locality, running verification logic as close to the execution context as possible, rather than offloading to acceler-ators. 

## 8 Related Work 

WebAssembly Security. Research in Wasm security has primarily focused on hardening the runtime or binary against exploitation. Defenses like MS-Wasm [23, 30] and library-level isolation frame-works like RLBox [31] enforce fine-grained compartmentalization, while hardware-based approaches like PKUWA [26] and Wave [22] provide memory protection. However, these approaches often re-quire custom runtimes, specialized compilers, or modifications to source code. Binary instrumentation approaches like WBSan [52], and others [12, 17, 18, 25] instrument binaries with runtime checks to detect memory errors, but are designed for bug discovery. In con-trast, Walma provides continuous runtime attestation of memory state integrity for deployed Wasm modules on untrusted hosts. 

Remote Attestation & CFA. Traditional remote attestation schemes such as SWATT [36] and Pioneer [37] verify static code identity, ensuring that the loaded binary has not been tampered with. To detect runtime attacks, CFA mechanisms like C-FLAT [3], OAT [40], and LiteHAX [11] generate cryptographic traces of the execution path. DIALED [10] extends this by attesting the integrity of data inputs. DMA’n’Play [41] uses hardware-based DMA to validate critical data in memory on embedded devices, demonstrating the value of state-centric attestation for detecting data-only attacks. In contrast, Walma is the first to enable state-centric attestation for complex software systems by attesting Wasm linear memory in the browser and on the edge. 

Memory Forensics & ML. Our work builds on offline memory forensics [33, 39], which demonstrated the utility of CNNs for de-tecting malware in static memory dumps. Walma is inspired by this concept to enable state-centric attestation, overcoming the latency Walma : Learning to See Memory Corruption in WebAssembly Conference’17, July 2017, Washington, DC, USA 

challenges of deep learning by employing lightweight ResNet archi-tectures and optimized snapshot transmission to enable continuous, online verification. 

## 9 Conclusion 

Walma presents a state-centric approach to WebAssembly security that detects data-oriented corruption and external tampering by classifying linear memory snapshots. Our evaluation demonstrates up to 100% verdict accuracy on structured targets while maintaining runtime overhead between 1.07 × and 1.82 × depending on attes-tation frequency. As Wasm adoption grows in security-sensitive contexts and existing defenses remain insufficient against data-only attacks and malicious hosts, this overhead is a necessary cost for continuous integrity assurance. Future work will focus on improv-ing detection for unstructured workloads and exploring temporal models to track state evolution.
