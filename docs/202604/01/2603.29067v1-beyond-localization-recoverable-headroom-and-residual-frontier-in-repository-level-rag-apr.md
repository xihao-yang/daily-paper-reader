## 摘要
在仓库级自动程序修复（APR）中，改进定位精度正越来越多地被视为实现更优修复的主要途径。本文提出一个更具针对性的问题：在定位能力增强之后，哪些定位后的机制仍然能够带来可恢复的性能提升？哪些提升受到我们协议的限制？以及还剩下什么残余潜力？我们在 SWE-bench Lite 上对三种具有代表性的仓库级 RAG-APR 范式——Agentless、KGCompass 和 ExpeRepair——进行了研究。我们的实验协议结合了 Oracle Localization、池内 Best-of-K、在固定接口下添加上下文探针（并设置同条件同字长填充控制）及同仓库硬负样本，以及统一封装的 oracle 检验。Oracle Localization 能提升三种系统的性能，但 Oracle 的成功率仍低于 50%。在采样的 10 个补丁池中，额外的候选多样性仍有帮助，但提升空间迅速饱和。在两个固定接口下，大多数高信息量的附加上下文条件仍优于其对应控制条件。统一封装检验显示不同系统响应差异：在相同封装下，KGCompass 与 ExpeRepair 仍保持较大增益，而 Agentless 对构建器选择更为敏感。提示层融合仍然存在显著的残余潜力：最佳固定探针仅比原生三系统在 Solved@10 并集上多解决 6 个实例。总体而言，更强的定位、有限的搜索、证据质量以及接口设计共同影响了仓库级修复的结果。

## Abstract
Repository-level automated program repair (APR) increasingly treats stronger localization as the main path to better repair. We ask a more targeted question: once localization is strengthened, which post-localization levers still provide recoverable gains, which are bounded within our protocol, and what residual frontier remains? We study this question on SWE-bench Lite with three representative repository-level RAG-APR paradigms, Agentless, KGCompass, and ExpeRepair. Our protocol combines Oracle Localization, within-pool Best-of-K, fixed-interface added context probes with per-condition same-token filler controls and same-repository hard negatives, and a common-wrapper oracle check. Oracle Localization improves all three systems, but Oracle success still stays below 50%. Extra candidate diversity still helps inside the sampled 10-patch pools, but that headroom saturates quickly. Under the two fixed interfaces, most informative added context conditions still outperform their own matched controls. The common-wrapper check shows different system responses: under a common wrapper, gains remain large for KGCompass and ExpeRepair, while Agentless changes more with builder choice. Prompt-level fusion still leaves a large residual frontier: the best fixed probe adds only 6 solved instances beyond the native three-system Solved@10 union. Overall, stronger localization, bounded search, evidence quality, and interface design all shape repository-level repair outcomes.

## 二、方法论与技术流程

### 核心思想
作者提出一种**分阶段控制实验框架**，通过一系列“oracle化”与固定接口操作，系统分析仓库级修复任务中各模块的极限作用。重点不在于提出新的模型，而是**揭示性能上限、瓶颈与错误来源**。

### 关键技术要点
1. **Oracle Localization（预言机定位）**  
   - 从金标准补丁中抽取实际修改的文件与行号作为“fault spans”，注入系统定位结果中，以消除定位误差。
   - 对比三个系统的性能变化，测量定位改进的直接贡献。

2. **Best-of-K 采样与选择饱和分析**  
   - 每个系统生成 10 个候选补丁，评估在理想“oracle选取者”下（即选取其中任意一个成功补丁）能恢复的最大性能。
   - 分析采样数量K对成功率的边际收益曲线。

3. **固定接口上下文探针（Fixed-Interface Probes）**  
   - 在系统生成最终补丁之前，统一插入额外上下文块（来自其他系统或额外RAG结构）。
   - 设计同字长填充控制（same-token filler）与同仓库硬负样本（hard negatives），用于区分真实信息增益与单纯的提示长度效应。

4. **统一封装验证（Common-wrapper Check）**  
   - 使用统一的封装器（wrapper）与提示构建器（builder），检测不同系统是否对提示接口敏感。

5. **残余前沿分析（Residual Frontier）**  
   - 对所有系统及探针的结果取并集，识别仍未被解决的实例，并对错误类型进行定性分析。

## 三、实验设计

### 数据集与场景
- **主要基准**：SWE-bench Lite（总计 300 个仓库级实例），提供仓库快照、issue描述及测试集。
- **任务**：每个实例需生成补丁并通过官方测试。
- **保持一致性**：固定任务划分与验证脚本以确保结果可复现。

### 对比系统
三种代表性 RAG-APR 范式：
1. **Agentless**：阶段式管线，层级定位 + 测试驱动验证。
2. **KGCompass**：基于知识图谱的结构化检索。
3. **ExpeRepair**：引入双记忆模块的经验增强式修复。

### 对比模型与算子

### 衡量指标
- **主要指标**：成功率（Success@300）、Solved@K、Recall@1/5等。  
- **统计显著性**：Wilson 置信区间 + McNemar 检验 + Bootstrap 置信带。

## 六、主要结论与发现

1. **定位仍有限**：  
   - Oracle Localization 提升显著（成功率由约30%升至40–43%），但**最高成功率仍不足50%**，说明下游过程仍是瓶颈。
   - 大部分提升来自执行通过率改善，而非仅提交率上升。

2. **搜索余量有限**：  
   - 多采样有效但**收益快速饱和**，K=5 已恢复约 87% 的最大增益。
   - 主要瓶颈在于**早期未能生成正确补丁系列（patch family）**。

3. **上下文信息是真正关键**：  
   - 多数信息性上下文探针显著优于同字长控制，证明效果来自信息结构而非长度。
   - 但**多源融合（UnionContext）并不必然更好**，甚至可能稀释单源稳定性。

4. **封装影响实际增益**：  
   - Agentless 的性能对封装器极度敏感，KGCompass 和 ExpeRepair 较稳定。

5. **残余前沿大且集中**：  
   - 在所有优化后，仍有 101/300 实例完全未能修复。其中 65 个集中在 *sympy* 与 *django* 仓库。
   - 常见错误类型包括：调用错误 API、补丁不完整、导入缺失、断言不匹配。
