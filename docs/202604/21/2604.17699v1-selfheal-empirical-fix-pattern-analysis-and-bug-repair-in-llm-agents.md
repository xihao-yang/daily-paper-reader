## 摘要
大型语言模型（LLMs）已经改变了软件开发和人工智能应用。虽然LLMs主要用于文本处理，但LLM代理（agents）通过支持自主行动、工具使用以及多步任务执行，扩展了这一能力。随着该领域的发展，开发者在调试这些复杂系统时面临新的挑战。为应对这一问题，我们首次对LLM代理中的错误修复模式进行了实证研究。我们收集并分析了来自三个平台的错误帖子与代码片段：Stack Overflow、GitHub 和 HuggingFace 论坛。我们研究了它们的修复模式、修复所应用的组件，以及涉及的编程语言和框架。此外，我们引入了 AgentDefect——首个针对LLM代理错误的基准数据集。该数据集包含37个运行时错误实例，以及对应的修复代码和测试文件。最后，我们提出了 SelfHeal，这是一种用于修复LLM代理错误的多代理系统。该系统使用两个独立的 ReAct 代理：修复代理和评审代理。这些代理利用包含内部知识（修复规则）和外部知识（网页搜索）的工具来提出并验证修复方案。我们的评估表明，以 Gemini 3 Pro 作为底层LLM的 SelfHeal，在性能上显著优于基线方法和当前最先进的方法。

## Abstract
Large Language Models (LLMs) have transformed software development and AI applications. While LLMs are designed for text processing, LLM agents extend this capability by enabling autonomous actions, tool use, and multi-step task completion. As this field grows, developers face new challenges in debugging these complex systems. To address this challenge, we present the first empirical study on bug fix patterns in LLM agents. We study buggy posts and code snippets from three platforms: Stack Overflow, GitHub, and HuggingFace Forums. We examine their fix patterns, the components where fixes are applied, and the programming languages and frameworks involved. Furthermore, we introduce AgentDefect, the first benchmark dataset for bugs in LLM agents. The dataset contains 37 runtime buggy instances along with fixed code and test files. Finally, we present SelfHeal, a multi-agent system designed to fix bugs in LLM agents. The system leverages two independent ReAct agents: the fix agent and the critic agent. These agents use tools that provide both internal knowledge (fix rules) and external knowledge (web search) to propose and validate fixes. Our evaluation shows that SelfHeal with Gemini 3 Pro as the backbone LLM outperforms both baseline and state-of-the-art approaches by a significant margin.

## 二、研究方法与技术路线

### 1. 核心思想

- 将 LLM 代理视为可出现故障的软件实体，而不仅仅是调试工具。
- 通过分析实际代码修复案例，生成可归纳的“修复模式”（fix pattern）规则，指导模型自动推理修复策略。
- 使用两个独立的 **ReAct** 智能体（fix 与 critic agent）协同完成自动化修复与验证。

### 2. 方法结构

#### （1）修复模式分析与规则生成
- 从 Stack Overflow、GitHub、HuggingFace 三个平台收集 **930 个错误实例**。  
- 识别 **23种修复模式**（如新增操作、修改参数、版本更新、修正语法等）。  
- 使用 LLM（GPT-5 Mini）提炼每个模式的摘要与修复规则，形成自然语言描述的“应用条件与修改方法”。

#### （2）SelfHeal 系统架构

**Fix Agent（修复代理）**
- 输入：原始错误代码与任务意图。
- 使用 4 个功能工具：
  1. `List Fix Patterns`：列出所有可用修复模式。  
  2. `Fix Pattern Rule`：按需提取具体模式规则。  
  3. `Web Search`：调用 Google API 检索额外知识（排除数据源网站防止泄漏）。  
  4. `Submit Fix Code`：输出候选修复版本。  

**Critic Agent（评审代理）**
- 接受修复代码，使用：
  1. `Code Compare`：分析差异。  
  2. `Validate API`：验证函数或库调用的有效性。  
  3. `Validate Format`：检查结构和格式可执行性。  
- 迭代 3 次评审-修改循环，最终收敛于最优版本。

### 3. 算法流程（文字描述）
1. 数据分析 → 生成修复模式 → 抽取规则；
2. Fix Agent 读取错误 → 查找规则 / 搜索外部信息；
3. 输出候选修复代码；
4. Critic Agent 比对与验证；
5. 若被拒绝 → Fix Agent 调整代码重新尝试，最多三次；
6. 输出最终通过评审的代码。

## 六、主要结论与发现

- **SelfHeal（Gemini 3 Pro）最高性能**：修复 22 / 37 实例。
- 相比 SWE-Agent，平均修复率提升 **16.21%**；相比零样本提升 **21.72%**。
- 两种信息源（内部规则与外部搜索）均明显提升性能：  
  - 去掉规则性能下降 **18.92%**；  
  - 去掉搜索下降 **13.51%**。
