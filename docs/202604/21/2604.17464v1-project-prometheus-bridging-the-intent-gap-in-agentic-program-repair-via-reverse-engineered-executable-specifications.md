## 摘要
从神经机器翻译到自主智能体的工作流转变，已经彻底革新了自动程序修复（Automated Program Repair, APR）。然而，尽管现有智能体具备先进的推理能力，它们仍常常遭受“意图鸿沟”（Intent Gap）——即生成的补丁与开发者原始意图之间的不一致。现有依赖自然语言摘要或对抗采样的解决方案，往往无法提供外科式修复所需的确定性约束。本文提出了一个名为\textsc{Prometheus}的新框架，通过优先考虑“规格推理”（Specification Inference）而非代码生成，以弥合这一鸿沟。我们采用行为驱动开发（Behavior-Driven Development, BDD）作为可执行的契约，利用多智能体架构，从运行时失败报告中逆向工程出 Gherkin 规格。为解决“意图幻觉”（Hallucination of Intent）问题，我们设计了一个\textbf{需求质量保证循环（Requirement Quality Assurance, RQA Loop）}，该机制利用真实代码作为代理预言机，以验证推理出的规格。我们在 Defects4J 基准集上的 680 个缺陷上对 \textsc{Prometheus} 进行了评估，结果具有突破性：本框架实现了\textbf{93.97\%}（639/680）的正确补丁率。更重要的是，它展现了\textbf{74.4\% 的救援率（Rescue Rate）}，成功修复了 119 个强盲智能体未能解决的复杂错误。定性分析表明，明确的意图使智能体避免了结构性过度工程，而趋向于精确、最小化的修复。我们的研究发现，APR 的未来不在于更大的模型，而在于能够使代码与经过验证的\textbf{可执行规格（Executable Specifications）}保持一致的能力——无论这些规格是预先存在的还是通过逆向工程获得的。

## Abstract
The transition from neural machine translation to agentic workflows has revolutionized Automated Program Repair (APR). However, existing agents, despite their advanced reasoning capabilities, frequently suffer from the ``Intent Gap'' -- the misalignment between the generated patch and the developer's original intent. Current solutions relying on natural language summaries or adversarial sampling often fail to provide the deterministic constraints required for surgical repairs.   In this paper, we introduce \textsc{Prometheus}, a novel framework that bridges this gap by prioritizing \textit{Specification Inference} over code generation. We employ Behavior-Driven Development (BDD) as an executable contract, utilizing a multi-agent architecture to reverse-engineer Gherkin specifications from runtime failure reports. To resolve the ``Hallucination of Intent,'' we propose a \textbf{Requirement Quality Assurance (RQA) Loop}, a mechanism that leverages ground-truth code as a proxy oracle to validate inferred specifications.   We evaluated \textsc{Prometheus} on 680 defects from the Defects4J benchmark. The results are transformative: our framework achieved a total correct patch rate of \textbf{93.97\%} (639/680). More significantly, it demonstrated a \textbf{Rescue Rate of 74.4\%}, successfully repairing 119 complex bugs that a strong blind agent failed to resolve. Qualitative analysis reveals that explicit intent guides agents away from structurally invasive over-engineering toward precise, minimal corrections. Our findings suggest that the future of APR lies not in larger models, but in the capability to align code with verified, \textbf{Executable Specifications} -- whether pre-existing or reverse-engineered.

## 一、研究背景与核心问题

### 1.1 研究动机  
自动程序修复（Automated Program Repair, APR）近年来从神经机器翻译式模型转向“多智能体”（Agentic）工作流，即使用具备推理与决策能力的智能体来自动修复代码。然而，尽管这些智能体能理解代码语义并执行复杂分析，它们在把握**开发者原始意图**方面依然存在显著缺陷，这一问题被作者定义为：

> **“意图鸿沟 (Intent Gap)”** —— 生成的补丁与原始需求之间存在错位。

### 1.2 问题本质  
现有方法常以自然语言描述或自生成测试用例作为“意图”的代理，但由于自然语言模糊、对抗采样不确定，缺乏可执行、确定性的约束，导致模型产生“幻觉式修复”（修错位置或破坏结构）。研究表明模型在缺乏精确定义的“意图”时容易过度修改代码（即“Berserker-style”修复）。

## 二、方法论：Prometheus 框架

### 2.1 核心思想  
**Prometheus** 的创新点在于：  
> **先推理“可执行规格”（Executable Specification），再执行修复。**  
也就是说，系统不直接让模型生成补丁，而是先让一个智能体“推理出缺失的需求”，再以此为约束指导修复流程。

### 2.2 框架结构与步骤  

论文提出一个 **三智能体架构（Three-Agent System）**：

1. **Architect（架构师）**  
   - 职责：从运行时失败报告（含堆栈信息和断言日志）中逆向工程出行为驱动开发（BDD）风格的 *Gherkin* 规格文件。  
   - 方法：通过两阶段推理（根因分析 + 规格综合）生成一种“给定-当-则(Given-When-Then)”格式的场景描述。  
   - 输出成果：`.feature` 文件，既可读又可执行。

2. **Engineer（工程师）**  
   - 职责：验证规格是否正确，即提出 **RQA Loop（需求质量保证循环）**。  
   - 机制：
     1. 在 **C_buggy**（原始有缺陷代码）上执行场景，期望**失败**；
     2. 在 **C_fixed**（开发者修复版本）上执行同一场景，期望**通过**。  
   - 若不符合条件，则循环再生规格。此过程称为“**夹心式验证（Sandwich Verification）**”。

3. **Fixer（修复者）**  
   - 职责：基于验证通过的 Gherkin 规格执行“外科式”修复（即仅在必要范围做最小更改）。  
   - 约束：实验阶段中 Fixer 仅可操作单一文件，以排除定位误差的干扰。

### 2.3 算法思想  
Prometheus 把传统随机搜索式修补转化为一种 **约束满足问题（Constraint Satisfaction Problem）**：  
- 目标函数由规格定义；  
- 搜索空间受“给定/当/则”条款约束；  
- 修复成功的定义：通过规格测试且语义等价于真实修复。

## 六、主要结论与发现

1. **Prometheus 显著提高修复率**：总体正确率 93.97%，救援率 74.4%，在复杂项目（Math、Compress）上表现尤为突出。  
2. **明确定义的可执行意图使修复更精准**：避免盲修中的过度修改和错误假设。  
3. **规格引导促进行为对齐**：
   - “意图镜像效应”（Intent-Behavior Mirroring Effect）：输入要求越精准，输出代码越局部、越正确。  
4. **成本结构合理**：更多计算被用于验证而非生成，避免无效补丁采样。  
5. **BDD 不仅为测试工具，更是认知支架（Cognitive Scaffold）**：即使无法执行，也能为模型限定逻辑空间。  
6. **未来方向**：
   - 引入“Detective”智能体用于自动定位；
   - 演化为具备自学习能力的“免疫系统”式维护框架。

## 七、优点与创新亮点

- **多智能体协作机制**：角色分工明确（Architect/Engineer/Fixer），体现“意图→验证→修复”的闭环。

- **高透明度与可解释性**：输出的 `.feature` 文件同时可人读可机检，有利于开发者信任与复现。  
- **强实验表现**：超越 RepairAgent 和 TSAPR 约 4.4 倍的复杂缺陷修复能力。  
- **认知发现**：提出“意图-行为镜像效应”，揭示模型行为与输入意图粒度的因果关系。
