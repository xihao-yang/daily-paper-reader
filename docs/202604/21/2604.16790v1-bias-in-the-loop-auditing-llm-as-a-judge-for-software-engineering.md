## 摘要
大型语言模型（LLM）越来越多地被用作评审者，用于在缺乏全面人工审查或可执行测试覆盖的情况下评估代码工件。在具备代理特性的软体工程工作流中，LLM评审的角色日益重要，可用于对候选解进行排序并指导补丁选择。尽管这种方法在扩展性上具有吸引力，但当前实践缺乏关于可靠性和偏差的系统性研究：对同一案例的多次评估可能存在不一致；轻微的提示修改可能导致结果变化；而在人类看来保持语义一致的等价扰动却可能引发不同的判断结果。本文从测量优先的视角研究了代码领域的LLM-as-a-Judge。我们分析了代码生成、代码修复与测试生成任务中的两种逐点评审机制，并系统性探查了提示诱发的偏差。研究考虑了重复运行的难度层级，以及通过控制提示干预以隔离单个呈现线索的实验设计，并通过一致性与偏差敏感性来评估评审表现。结果表明，即使底层代码片段保持不变，评审决策仍对提示偏差高度敏感。在三类任务中，多种偏差会系统地将偏好倾向于提示中偏爱的选项，当提示选项与标准答案一致时可提升准确率，但在其他情况下准确率显著下降。在某些情形下，这种影响足以改变任务层面的结论和模型的相对排名。这些发现表明，报告的评审性能可能反映的是提示伪影而非稳定的评估能力，从而直接威胁代码评估的有效性与可复现性。我们因此主张，LLM-as-a-Judge 相关研究应在准确率的同时报告偏差敏感性，并引入显式控制，以支持软件工程中更值得信赖的模型比较。

## Abstract
Large Language Models are increasingly used as judges to evaluate code artifacts when exhaustive human review or executable test coverage is unavailable. LLM-judge is increasingly relevant in agentic software engineering workflows, where it can help rank candidate solutions and guide patch selection. While attractive for scale, current practice lacks a principled account of reliability and bias: repeated evaluations of the same case can disagree; small prompt edits can swing outcomes; and seemingly semantics-preserving, human-equivalent perturbations may elicit divergent verdicts. This paper studies LLM-as-a-Judge for code through a measurement-first lens. We analyze two pointwise judging regimes across code generation, code repair task, and test generation, and we systematically probe prompt-induced biases. Our study considers difficulty levels for repeated runs and controlled prompt interventions that isolate one presentation cue at a time, and it evaluates judges using consistency and sensitivity to bias. We find that judge decisions are highly sensitive to prompt biases even when the underlying code snippet is unchanged. Across all three tasks, several biases systematically shift preferences toward the option favored by the prompt, improving accuracy when that option aligns with the gold answer but substantially reducing it otherwise. In some settings, these effects are large enough to change task-level conclusions and alter relative model rankings. These findings show that reported judge performance may reflect prompt artifacts rather than stable assessment ability, posing a direct threat to the validity and reproducibility of code evaluation. We therefore argue that LLM-as-a-Judge studies should report bias sensitivity alongside accuracy and incorporate explicit controls to support more trustworthy model comparison in software engineering.

## 一、核心问题与研究动机

- **研究背景**  
  大型语言模型（LLM）逐渐被用作“评审者”（LLM-as-a-Judge），用于代码生成、修复、测试生成等任务的自动评估，以替代人工审查或耗时的执行测试。  
  在“代理型软件工程（agentic SE）”工作流中，这种评审角色日益重要，可帮助排序补丁、筛选候选方案、决定何时自动合并或升级人工审核。

- **核心问题**  
  当前使用 LLM 评审存在严重可靠性与偏差问题：
  - 同一案例的重复评估结果不一致；
  - 微小提示（prompt）修改即可显著影响评审结论；
  - 语义保持一致的提示变化也会产生截然不同的判断；
  - 不同模型之间的排名可能因此发生变化，导致评估不可复现。
  
  本研究旨在**系统审计 LLM-as-a-Judge 在代码任务中的偏差敏感性与一致性**，评估其作为可靠评估工具的可行性。

---

## 二、方法论与技术框架

### 核心思想
- 从“**测量优先（measurement-first）**”的角度出发，设计系统化实验来量化两类维度：
  1. **提示偏差的影响**（Bias Sensitivity）；
  2. **测试重测一致性**（Test–Retest Reliability）。

### 关键要素与设计
- **评审模式**：基于 *pairwise comparison*（两选比较），让 LLM 从两个代码候选（A 与 B）中选择更优解。
- **对应公式**：
  - **微平均准确率**：
    \[
    Acc(b) = \frac{1}{|D||P|} \sum_{i \in D}\sum_{p \in P} \mathbf{I}[\hat{y}(i,p)_b = y(i,p)]
    \]
    表示模型在偏差条件 \( b \) 下的总体正确率。
  - **一致性率（CR）**：
    \[
    CR = \frac{1}{|D|} \sum_{i \in D} \mathbf{I}[\hat{y}(i,1) = \hat{y}(i,2)]
    \]
    衡量同一问题重复评判时的一致性。

### 偏差干预体系
- 设计并注入 **12 种提示偏差**（见论文表 1、2），包括：
  - 显式偏差：位置（Position）、权威（Authority）、连锁推理（CoT）、情感（Sentiment）、详细度（Verbosity）等；
  - 隐式偏差：风格多样性（Diversity）、来源名称（Model-name）、自增强（Self-enhance）等。
- 每次只改变一个提示因素，固定代码与评审步骤，以便隔离单一偏差效应。

## 三、实验设计与数据集

### 数据来源与任务
- 使用 **CodeJudgeBench** 基准测试（Jiang et al., 2025）：
  - 三类任务：
    1. **CodeGen**：代码生成对比；
    2. **CodeRepair**：代码修复对比；
    3. **TestGen**：单元测试生成对比。
  - 数据规模：总计 **5,352 个配对样本**（good/bad pairs），从 LiveCodeBench 提取。
  - 难度分层：Easy / Medium / Hard 三档。

### 评审模型
- 三种评审器（LLM-as-Judge）：
  1. **Qwen3-4B（开源通用模型）**；
  2. **Qwen2.5-Coder-3B（开源代码专用模型）**；
  3. **GPT（API 型闭源模型）**。
- 对比目的：观察偏差敏感性是否跨模型普适。

### 评估指标
- **Accuracy**（正确率）
- **Consistency Rate (CR)**（重复一致性）
- 在每个任务与每个偏差条件下进行成体系比较，并分析不同难度层的变化。

## 六、主要结论与发现

1. **高偏差敏感性**  
   所有模型的评审结果对提示细节变化高度敏感，微小的偏差注入即可改变正确率数十个百分点：

   - **Qwen 系列**：Position、CoT、Sentiment、Refined 等提示可系统性地偏向第一候选（A）。  
   - 当正确答案在位置 B，准确率显著下降，甚至跌至无偏时的一半。
   - **Verbosity**（长文本偏差）表现出相反效应：倾向选择第二候选（B）。

2. **重复性与稳定性（CR）**
   - 不同任务一致性不同：**CodeRepair 最稳定、TestGen 最不稳定。**
   - 某些偏差（Sentiment、Refined）能提升一致性，但这是“伪稳定”——模型在系统性偏差下重复犯同样错误。

3. **跨模型结论**
   - Qwen3 与 Qwen2.5-Coder 展现相同偏差模式，仅程度不同；
   - GPT 虽初始准确率高，但同样受提示影响，尤其在 TestGen 任务中下降明显。

4. **答案率（Answer Rate）差异**
   - 专门代码模型（Qwen2.5-Coder-3B）能在 99% 样本上输出合格判断；
   - 通用模型（Qwen3-4B）仅回答不到 45%，其余为非结构化自由生成。

5. **综合发现**
   - 报告的评审性能往往反映提示伪影，而非稳定的评估能力；
   - 现行 LLM-judge 评测存在重大可复现性风险，应增加提示敏感性报告与稳健性控制。

## 七、研究优点与创新

- **系统化的偏差审计框架**：首次将 NLP 评审偏差体系整合扩展到代码评审场景。
- **严格的实验控制**：固定代码与评分标准，仅改变提示变量，确保偏差来源可归因。
- **跨任务、跨模型验证**：同时分析开源与闭源模型的异同，增强结论的普适性。
- **新增一致性指标（CR）报告**：将稳定性视为评审可靠性核心指标。
- **提示设计可重复性**：公开 12 类提示模板，为后续研究提供可复用实验基线。

## 八、不足与局限

- **任务覆盖有限**：仅测试三类代码任务（生成、修复、测试），未涵盖如代码审查、摘要、翻译等更广应用。
- **语言与提示单一**：仅使用英文 prompt，可能无法泛化到多语言环境。
- **评审维度单一**：聚焦可选结果正确性，未评估解释质量或安全性。
- **实验重复次数较少**：一致性测试仅两次重复，无法反映全部随机性。
- **计算资源受限**：单 GPU 环境限制了模型规模与采样次数。
- **难度标签外源化风险**：采用 CodeJudgeBench 自带标签，其难度定义可能不完全与模型内部感知匹配。
