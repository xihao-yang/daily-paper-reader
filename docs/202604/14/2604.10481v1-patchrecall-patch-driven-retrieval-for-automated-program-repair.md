## 摘要
从大型代码库中检索正确的文件集是自动程序修复（Automated Program Repair，APR）的关键步骤。高召回率对于确保相关文件被包含至关重要，但简单地增加检索文件的数量会引入噪声并降低效率。为了解决这种权衡问题，我们提出了 PatchRecall，这是一种在召回率与简洁性之间取得平衡的混合检索方法。我们的方法结合了两种互补策略：(1) 代码库检索，通过将当前问题描述与代码库匹配，以发现潜在相关文件；(2) 基于历史的检索，利用相似的历史问题来识别被编辑的文件作为候选目标。来自这两种策略的候选文件将被合并并重新排序，以生成最终的检索集。在 SWE-Bench 上的实验表明，PatchRecall 能在不显著增加检索文件数量的情况下实现更高的召回率，从而提升 APR 的有效性。

## Abstract
Retrieving the correct set of files from a large codebase is a crucial step in Automated Program Repair (APR). High recall is necessary to ensure that the relevant files are included, but simply increasing the number of retrieved files introduces noise and degrades efficiency. To address this tradeoff, we propose PatchRecall, a hybrid retrieval approach that balances recall with conciseness. Our method combines two complementary strategies: (1) codebase retrieval, where the current issue description is matched against the codebase to surface potentially relevant files, and (2) history-based retrieval, where similar past issues are leveraged to identify edited files as candidate targets. Candidate files from both strategies are merged and reranked to produce the final retrieval set. Experiments on SWE-Bench demonstrate that PatchRecall achieves higher recall without significantly increasing retrieved file count, enabling more effective APR.

## 一、研究背景与核心问题

- **研究动机**：  
  自动程序修复（Automated Program Repair，APR）的关键前提是从庞大的代码仓库中准确定位需要修改的文件。然而，现有方法在召回率与检索效率之间存在权衡：
  - 提高召回率往往意味着检索更多文件，导致噪声增加；
  - 降低文件数量又可能遗漏目标文件，从而使修复失败。
- **研究意义**：  
  当前大型语言模型（LLMs）在代码修复任务中表现受限，尤其在真实项目（如 SWE-Bench 基准）上整体成功率不足 15%。  
  因此，为 APR 设计一种“高召回但不冗余”的文件检索机制成为亟需突破的方向。

## 二、方法论概述

### （1）核心思想

PatchRecall 引入一种**混合式文件检索机制**——同时融合同一仓库内的文本匹配信号与历史修复记录中的语义线索，通过**补丁引导的再排序机制**实现精简且高召回的文件候选集。

### （2）两条主要检索路径

1. **代码库检索（Codebase Retrieval）**  
   - 输入：当前问题（issue）文本。  
   - 方法：利用经典 BM25 稀疏检索算法计算问题与代码文件的文本相似度，选出 top‑k 文件。  
   - 优点：快速、可扩展。缺点：难以捕获语义关联。

2. **历史补丁检索（History-based Retrieval）**  
   - 基于 SWE-Bench 的“unverified”子集（1794 个问题），建立语义索引；
   - 使用 **Sentence Transformer（all‑mpnet‑base‑v2）** 模型从历史问题中找到前 10 个最相似的问题；
   - 提取这些历史问题对应的补丁中被修改的文件，并按照出现频率与相关性打分；
   - 该过程依据 SWE-Bench‑verified 数据的统计（80% 仅改 1 个文件）确定最多 10 个候选文件。

### （3）混合分数融合与重排序

- 令来自两条路径的文件得分分别为  
  \( s_{ST}(f) \) 与 \( s_{BM25}(f) \)；
- 进行 min–max 归一化，获得标准化得分 \( s'_{ST}(f) \) 与 \( s'_{BM25}(f) \)；
- 融合公式：

  \[
  H(f) = \alpha \cdot s'_{ST}(f) + (1-\alpha) \cdot s'_{BM25}(f)
  \]

  其中 α 为权重参数，控制语义与文本匹配信息的平衡（实验中 α≈0.4–0.6 表现最佳）。  
- 最后根据混合分数 H(f) 重新排序，选取前 k 个文件作为 APR 模型输入。

### （4）流程概览

1. 接收新问题文本；  
2. 并行执行补丁检索与代码库检索；  
3. 归一化并融合得分；  
4. 输出排名前 k 的文件；  
5. 将结果输入下游 LLM 进行修复。

## 三、实验设计

- **基准与数据集**：  
  - SWE-Bench (Jimenez et al., 2023)：源自 12 个实际 Python 开源项目的 GitHub 问题及修复补丁；
  - 包含两部分：  
    - SWE-Bench‑verified（500 条人工验证问题–补丁对）；  
    - SWE-Bench‑unverified（1794 条自动匯入对）。

- **任务**：评估检索阶段能否召回“真实被修改的文件”，以 Recall@k 为主要指标。

- **对比方法**：
  1. **BM25**（稀疏检索基线）；  
  2. **TF–IDF**（传统词频模型）；  
  3. **Sentence Transformer**（密集语义模型）；  
  4. **PatchRecall**（混合融合方案）。

- **实验参数**：
  - k ∈ [1,10]；
  - α ∈ [0,1]；
  - 比较不同检索深度与融合权重下的召回表现。

## 六、主要结论与发现

1. **Sentence Transformer 检索性能优于稀疏方法**（训练集外语义理解更强）；  
2. **PatchRecall 混合策略显著提高 recall@k**，在 α ≈ 0.4–0.6 时效果最优；  
3. 在不增加检索文件数量（k ≤ 10）的前提下提升召回率，实现“高召回 + 精简检索”；  
4. 对比 MAGIS 所用 BM25 基线，PatchRecall 的 recall 在各深度均显著更高；  
5. 呈现出强的通用性与轻量级特征，可与多智能体系统结合使用。
