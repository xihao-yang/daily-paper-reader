## Abstract 

Large language models (LLMs) are rapidly transforming software engineering by enabling developers to generate code ranging from small snippets to entire projects. As AI-generated code becomes increasingly integrated into real-world systems, understanding its characteristics and impact is critical. However, prior work primarily focuses on small-scale, controlled evaluations and lacks compre-hensive analysis in real-world settings. In this paper, we present a large-scale empirical study of AI-generated code in real-world repositories. We analyze both code-level metrics (e.g., complexity, structure, and defect-related indica-tors) and commit-level characteristics (e.g., commit size, frequency, and post-commit stability). To enable this study, we develop heuris-tic filter with LLM classification to identify AI-generated code and construct a large dataset. Our results provide new insights into how AI-generated code differs from human-written code and how AI assistance influences development practices. These findings contribute to a deeper under-standing of the practical implications of AI-assisted programming.

## 1 Introduction 

Recent advances in artificial intelligence, particularly large lan-guage models (LLMs), are profoundly reshaping modern software engineering practices. Systems such as OpenAI Codex, GPT-series models, and Claude, along with IDE-integrated assistants like GitHub Copilot and Cursor, have enabled developers to offload a wide spec-trum of programming tasks to AI. These tools are now routinely used not only for generating small code snippets or completing functions, but increasingly for synthesizing entire files, modules, and even full-fledged projects. As a result, AI-assisted programming

is transitioning from an experimental capability to a mainstream development paradigm, fundamentally altering how software is written and maintained. This paradigm shift raises critical questions about the charac-teristics and implications of AI-generated code in real-world soft-ware systems. While early studies have examined LLM-generated code in controlled environments—typically focusing on small, self-contained programming tasks or benchmark datasets—such settings fail to capture the complexity and dynamics of real-world devel-opment. In practice, AI-generated code is often intertwined with human-written code, iteratively refined, and deployed within large, evolving codebases. Understanding the properties of such code therefore requires moving beyond lab-based evaluations toward comprehensive empirical studies in real-world repositories. A key aspect of this understanding lies in analyzing code-level characteristics of AI-generated code. These include traditional soft-ware quality and complexity metrics, such as cyclomatic complex-ity, control-flow and data-flow complexity, loop nesting depth, and language-specific constructs (e.g., object-oriented metrics for Java or pointer-related complexity in C/C++). Additionally, defect-related indicators, such as bugs per KLOC, are essential for assessing reliability and maintainability. Comparing these metrics between AI-generated and human-written code can provide insights into whether AI assistance leads to simpler, more complex, or potentially more error-prone implementations. Beyond code-level properties, it is equally important to inves-tigate commit-level characteristics associated with AI-generated code. Modern software development is inherently collaborative and version-controlled, making commits a natural unit of analysis. Questions arise as to whether AI-assisted development changes developer behavior and workflow dynamics: Do AI-generated com-mits modify larger portions of code? Do they accelerate develop-ment by increasing commit frequency or reducing development time? How stable are such commits—do they require more frequent subsequent fixes, revisions, or reverts? Addressing these questions is crucial for understanding the broader impact of AI on software engineering processes. Despite the growing interest in AI-assisted programming, there remains a significant research gap. Existing work primarily focuses on limited sets of code-level metrics and evaluates LLM-generated outputs in controlled, small-scale settings. To the best of our knowl-edge, no prior study has conducted a comprehensive, large-scale em-pirical analysis that jointly examines both code-level and commit-level characteristics of AI-generated code in real-world repositories.

This gap limits our ability to draw reliable conclusions about the practical implications of adopting AI in software development. In this paper, we aim to bridge this gap by conducting a compre-hensive empirical study of AI-generated code in real-world projects. A central challenge in this endeavor is identifying and collecting AI-generated code from large-scale repositories, where such in-formation is not explicitly labeled. To address this, we develop multi-stage rule-based message filter and LLM-assisted classifier for Github commit messages to construct a large dataset of AI-generated code instances. Based on this dataset, we perform, to the best of our knowledge, the most extensive measurement study to date, covering a wide range of code-level and commit-level metrics. Our analysis yields a set of actionable insights into the current state of AI-assisted software development. These insights shed light on how AI-generated code differs from human-written code in terms of complexity, quality, and evolution, as well as how it influences development practices at the commit level. Such findings have important implications for researchers, practitioners, and tool designers seeking to better understand and improve the integration of AI into software engineering workflows. In summary, this paper makes the following contributions: 

• We build and release a large-scale (by far largest to our knowledge) dataset of AI-generated code extracted from real-world repositories, enabling various analytical study. 

• Based on the dataset, we perform comprehensive measurement study of both code-level and commit-level characteristics, covering an extensive set of metrics, comparing AI-generated and human-written code. 

• We distill multiple insights from our measurement findings re-garding AI-assisted programming in practice and discuss potential implications for skateholders including programmars and AI ven-dors. 

## 2 Related Works 

Existing research on LLM-generated code can be broadly grouped into three lines: controlled prompt-based evaluation of generated outputs, empirical studies of real-world AI-assisted development, and detection or measurement of LLM-generated code in the wild. Most prior work evaluates LLM-generated code in prompt-based or benchmark-style settings, where researchers design program-ming tasks, ask LLMs to generate code, and then analyze the re-sulting outputs. For example, [23] studied differences in maintain-ability and reliability between LLM-generated and human-written code, while [ 18 ] examined performance and efficiency, and [ 35 ]focused on call calibration and correctness of LLM-generated code. [ 41 ] compared code style differences between human-written and LLM-generated code and further tracked style trends in GitHub open-source projects. Other studies investigated code-level features such as line counts, cyclomatic complexity, nesting depth, and class counts [ 27 ], while [ 10 ] examined lexical properties such as lex-ical density, however these features were often used mainly for classification or detection purposes, rather than for a systematic characterization of how LLM-generated code differs from human-written code across multiple structural dimensions. Security is another major focus in prompt-based evaluations of LLM-generated code. Early work conducted exploratory assess-ments of the security of ChatGPT-generated code [ 16 ]. Subsequent studies expanded the evaluation to multiple models and further examined how model parameters and prompting strategies affect the security of generated code [ 3]. To enable more systematic as-sessment, [ 33 ] proposed a unified framework integrating code gen-eration and security analysis, while [ 22 ] provided a more detailed study of the impact of prompt engineering on code security. [ 29 ]studied LLM-based security patching, which also sheds light on the security implications of LLM-generated code. In addition, [ 38 ]and [ 42 ] developed datasets and benchmarks for more systematic evaluation of the quality and security of LLM-generated code. Some work has further explored mitigation strategies, such as prompt engineering for safer code generation [ 30 ] and model fine-tuning to improve the security of generated code [ 13 ]. Relatedly, several studies have investigated the use of LLMs themselves for vulner-ability and bug discovery in code [ 4, 19 ]. Overall, these studies provide important foundations for understanding the security of LLM-generated code, but they still predominantly evaluate model outputs under controlled settings. Beyond controlled prompting, a growing body of work has be-gun to examine LLM-generated code in real-world open-source development. [ 25 ] and [ 8 ] studied GitHub Copilot-generated code in real GitHub repositories, focusing on security issues and the ability of LLMs to repair vulnerabilities. [ 40 ] analyzed self-admitted LLM usage in GitHub repositories, examining how generative AI is used in practice and surveying developers about usage policies. Using the AIDev dataset [ 17 ] of LLM-agent pull requests, [ 34 ] inves-tigated security-related issues, review interactions, and ecosystem characteristics, while [ 12 ] examined cyclomatic complexity and code quality in the same setting. Other studies focused on specific aspects of AI-generated code in repositories, such as code smells in GitHub Copilot-generated code [ 43 ] and adoption patterns re-flected in modified lines and touched files at both commit and file levels for AI-agent-generated code [ 28 ]. These studies move beyond synthetic prompting scenarios and provide valuable evidence about how LLMs are actually used in practice. Another related line of work studies the detection and measure-ment of LLM-generated code in the wild. [ 7] measured the usage of LLMs for coding by training a classifier for LLM-generated code, while [ 36 ], [ 24 ], and [ 2] also explored methods for detecting LLM-generated code. These studies are closely related because they help identify AI-generated code in large-scale repositories, but their main goal is detection or usage estimation rather than a system-atic comparison of the characteristics of real-world LLM-generated code and human-written code. Despite this growing body of work, an important gap remains. Prompt-based studies mainly evaluate generated outputs on de-signed tasks, which do not fully reflect how LLMs are used in real software development. Real-world repository studies, in contrast, usually focus on isolated aspects such as security issues, code smells, review interactions, or adoption patterns. As a result, there is still limited understanding of how real-world LLM-generated code sys-tematically differs from human-written code, not only in code-level characteristics but also in development activity patterns. ## 3 Measurement Pipeline 

In this section we describe our measurement pipeline, from real-world dataset collection to the various metrics we consider and the methodology to measure them. The process of our pipeline is described in Figure 1. 

## 3.1 Dataset Collection and Alignment 

As we target real-world code repositories, the first challenge is how to collect AI-generated code within them. To that end, we got inspiration from recent work [ 31 ] that demonstrates the feasibility of identifying AI-generated code from GitHub by searching for explicit hints in code comments (e.g., “generated by ChatGPT”). However, such approaches suffer from several limitations. First, keyword search often introduces substantial noises due to sub-string matching (e.g., matching “use” in unrelated words), leading to false positives. Second, attribution comments are frequently am-biguous, as mentions of LLMs may refer to non-code tasks (e.g., documentation or data generation). Third, prior filtering strategies remain relatively coarse-grained and cannot reliably distinguish true AI-generated code from incidental references. To address these challenges, we design a refined data collection pipeline that im-proves both the precision and reliability: 

Phase 1. Initial Data Collection. Following prior work [ 31 ], we leverage the GitHub REST API to retrieve code files containing explicit mentions of popular LLM tools. We extend the scope to include recent models (e.g., Claude, Gemini, DeepSeek) and repos-itories up to 2025. After filtering for comments containing tool mentions, we obtain an initial dataset of 31,726 commits and 44,616 unique code files. 

Phase 2. Multi-Stage Filtering. To improve data quality, we de-velop a multi-stage filtering pipeline: 

(1) Keyword normalization. We eliminate false matches caused by substring-based search by ensuring that matched prefixes (e.g., “use”, “by”) appear as standalone words in comments. This step reduces noise introduced by the GitHub search mechanism. 

(2) Semantic attribution filtering. We identify explicit attribution patterns (e.g., “generated by”, “written by”, “enhanced by”) and analyze their surrounding context to ensure that the LLM is used for code generation . Mentions referring to non-code artifacts (e.g., text, media, or data generation) are excluded. 

(3) Usage-based exclusion. We remove cases where LLMs are men-tioned only as tools for auxiliary tasks (e.g., API calls, analysis, or communication), rather than directly contributing to code genera-tion. After rule-based filtering, we obtain 4,882 commits and 8,141 files. Manual validation on randomly sampled data shows high precision (TP=88/100, TN=95/100), indicating the effectiveness of our rules. 

Phase 3. LLM-assisted Classification. For samples that cannot be confidently classified by rules, we further employ an LLM-based classifier (ChatGPT 5-mini) to infer whether code is AI-generated. This step yields an additional 7,867 commits (10,713 files). Remain-ing uncertain cases are manually labeled, resulting in 686 additional positive samples. A secondary validation confirms the reliability of LLM-based classification (TP=85/100, TN=93/100). 

Final Dataset. After deduplication using repository, file path, and commit hash, our final dataset contains 12,749 commits and 19,816 code files with confirmed AI involvement. To our knowl-edge, this is the largest dataset of real-world AI-generated code by far, which we will make public. Fig. 2a and Fig. 2b illustrate its dis-tribution across LLM models/tools and programming languages, re-spectively. Consistent with prior observations [ 31 ], general-purpose models (e.g., ChatGPT) dominate real-world usage, while code spans a diverse set of languages with Python and JavaScript being the most prevalent. 

Human Code Alignment. To enable fair comparison between AI-generated and human-written code, we construct a matched human code dataset. Direct comparison without control may in-troduce confounding factors, such as repository-specific coding practices, project complexity, or code scale differences. For exam-ple, variations in defect density or complexity metrics may stem from repository characteristics rather than the effect of AI assis-tance. To mitigate such biases, we adopt an alignment strategy : for each AI-generated file, we randomly select two human-written files from the same repository and same programming language , with similar size (within 20% lines of code when possible). This design controls for repository-level and language-level factors, ensuring that observed differences more accurately reflect the impact of AI assistance rather than external confounders. ## 3.2 Study Code-Level Characteristics 

We aim at comprehensively understanding the differences between AI-generated and human-written code in real-world settings, to that end, we measaure an extensive and diverse set of code-level characteristics , surpassing prior works and providing a holistic comparison. 

Metric Taxonomy. Specifically, our scope includes the following categories of metrics: 

(1) Code Size and Basic Statistics. Basic properties such as lines of code (LOC), number of files, and token-level statistics, providing a baseline understanding of code scale and granularity. 

(2) Structural Complexity. We analyze fine-grained structural com-ponents extracted from abstract syntax trees (ASTs), including functions, parameters, classes, statements, conditionals, loops, and recursion. We further derive complexity-related metrics such as cyclomatic complexity, nesting depth, and decision counts. 

(3) Code Style and Duplication. Stylistic properties such as code duplication, reflecting maintainability and reuse patterns. 

(4) Object-Oriented Design Metrics. For object-oriented languages (e.g., Java, C++, Python), we measure classical OOP metrics (e.g., class-level complexity and coupling) to understand design quality. 

(5) Program Graphs. We analyze program characteristics from their control-flow, data-flow, and call graphs, capturing higher-level prop-erties regarding both syntax and semantic. 

(6) Code Defects and Security. We measure security-related proper-ties such as vulnerability patterns and counts of potential defects, gaining insights of comparative code quality. 

Metric Normalization. As the absolute volume of AI-generated and human-written code is different in our dataset (§3.1), direct metric comparison by raw counts is misleading. To address this, we normalize each metric using an appropriate measurement unit .For example, defect-related metrics are normalized by lines of code (e.g., bugs per KLOC), while structural metrics are measured at the function or file level (e.g., average number of loops per function). 

Statistical Significance Testing. To draw meaningful and reliable conclusions in code characteristic differences betwene AI and hu-man, we perform statistical hypothesis testing to determine whether observed AI/human differences are statistically significant. 

Implementation. We employ various tools to compute the afore-mentioned metrics (e.g., cloc for basic statistics, CK and Understand 

for OOP metrics, CodeQL for security analysis, etc.). More details can be found in Table 1, which summarizes all measured metrics, along with their analysis tools and normalization scopes. 

## 3.3 Study Commit-Level Patterns 

Beyond code-level characteristics (§3.2), it is equally important to understand how AI-generated code is produced, integrated, and evolved within real-world development workflows. Commit-level analysis provides a natural lens to study such dynamics, as commits capture both developer activity and project evolution. By examin-ing commit histories associated with AI-generated code, we aim to uncover how developers collaborate with AI tools in practice and how AI-assisted code influences development processes and outcomes. 

Research Questions. We specifically aim at answering the fol-lowing research questions during our commit-level study, which collectively characterize the real-world usage patterns and lifecycle dynamics of AI-assisted code. 

• RQ1: Temporal patterns. Do AI-assisted commits exhibit different temporal characteristics (e.g., working hours, release proximity) compared to human-written code? 

• RQ2: Change characteristics. Do AI-generated commits involve dif-ferent magnitudes or types of changes (e.g., number of modified files, lines of code, churn)? 

• RQ3: Post-commit evolution. Are AI-generated commits more likely to be revised, reverted, or modified after submission? 

• RQ4: Collaboration and workflow. How does AI-assisted code affect developer collaboration patterns, such as handoffs and multi-author contributions? 

• RQ5: Stability and maintenance. Does AI-generated code require more frequent fixes or exhibit different stabilization patterns compared to human-written code? 

Methodology. To answer these questions, we extract commit-level information from real-world repositories corresponding to both AI-generated and human-written code. Specifically, information is collected from the following dimensions: 

• Temporal Characteristics (for RQ1). Commit timestamps, including time of day, day of week, and distance to the nearest release. 

• Change Characteristics (for RQ2). The scale and scope of code changes, including the number of modified files, lines, and overall churn. 

• Post-Commit Evolution (for RQ3, RQ4, and RQ5). The frequency of post-commit modification and revert within fixed time win-dows (e.g., 7, 30, and 90 days), the number of subsequent edits, participating authors, and time to first modification. 

• Workflow and Maintenance Patterns (for RQ4, and RQ5). Statistical Hypothesis Testing. Similar to §3.2, we ensure that our comparative conclusions are statistically significant. Specifically, beyond hypothesis testing between AI-generated code and human written code, we also implemented the pairwise comparisons, that compare each AI-generated with one of the human written code in the same repository. we use Chi-squared tests [ 26 ] for categorical variables and Mann-Whitney U tests [ 20 ] for numerical variables in group-level analysis, and McNemar’s tests [ 21 ] and Wilcoxon signed-rank tests [39] for paired comparisons. 

Implementation. To collect the aforemntioned information and answer our research questions, we mainly use the git toolset with various corresponding commands (e.g., git diff for change char-acteristics and git log for some temporal metadata, etc.).

## 4.3 Code Style 

Code style reflects how developers organize, format, and reuse code. We examine two complementary aspects: duplication patterns and 

formatting structure .

## 6 Discussion 

Threats to Validity. We discuss some potential threats to validity of our work as follows: 

• Construct validity is challenged by the difficulty of identifying AI-generated code in the wild, as AI involvement is rarely explicitly labeled and there is no large-scale real-world dataset. We miti-gate this using a multi-stage filtering pipeline as described in §3.1, though false positive and negatives can still occur regardlessly, our filtering achieves high accuracy in sampled verification and ensures high-fidelity statistical study. 

• Internal validity may be affected by repository-level confounders. We reduce this risk by aligning AI and human samples (§3.1) within the same repository, language, and similar code size, though residual differences may remain. 

• Measurement validity depends on the accuracy and coverage of analysis tools. While we employ well-established tools and normal-ized metrics, tool limitations and language-specific behaviors may introduce noise. Therefore, we emphasize consistent cross-metric trends rather than conclusions from any single metric. 

• Statistical conclusion validity is influenced by the large sample size, which can make small effects statistically significant. We inter-pret results using both statistical significance and effect size, and complement group-level analysis with paired comparisons where applicable. 

• External validity is limited to public GitHub repositories with explicit AI-related signals, and is dominated by certain tools and languages (e.g., ChatGPT, Python, JavaScript). While this may limit generalization, it reflects real-world open-source usage and pro-vides an in-vivo perspective on AI-assisted development. 

Scope and Interpretation of the Results. Unlike prior work that evaluates LLM-generated code under controlled prompts, we study the observable outcome of real-world AI-assisted programming : code that has been selected, edited, and integrated into real repositories. Our goal is therefore not to characterize all possible model out-puts—which are inherently prompt-dependent—but to understand the typical properties of AI-involved code in practice .Under this scope, a consistent pattern emerges: AI-involved code tends to be more explicit, more verbose, and more layered. It is gen-erally larger, uses more structural constructs, and exhibits deeper nesting, while AI-associated commits are smaller, more localized, and more likely to incur follow-up modification over time. These findings reflect the combined effect of model behavior and human integration, rather than raw model outputs alone. 

Implications for Developers and AI Vendors. Our results sug-gest that AI assistance reshapes development effort rather than sim-ply reducing it. While AI-generated code is effective for localized edits and rapid drafting, it is associated with increased follow-up modification and longer stabilization over time. Developers should therefore anticipate a shift of effort from initial coding to downstream validation and maintenance , and allocate review and testing effort accordingly beyond the immediate post-commit window. More importantly, our findings indicate that AI-generated code tends to be more explicit, less compact, and more layered. This sug-gests that the primary risk is not correctness failure, but structural inefficiency —including unnecessary verbosity, reduced reuse, and fragmented logic. Developers may benefit from treating AI outputs as drafts to be consolidated , focusing on abstraction, deduplication, and structural simplification after generation. For AI tool vendors, these results highlight a gap between gener-ation quality and long-term code quality. Current systems optimize for immediate correctness and completion, but less for maintain-ability. Our findings suggest that future tools should better support 

post-generation refinement , such as encouraging reuse across files, reducing unnecessary expansion, and warning about overly deep or fragmented structures. In addition, the strong language-dependent differences observed in our study indicate that AI assistants should adopt more language-aware optimization strategies rather than relying on uniform generation behavior. 

## Conclusion 

In this paper, we presented a large-scale empirical study of AI-generated code in real-world repositories. Unlike prior work based on controlled prompts, we analyze code that has been selected, edited, and integrated into real projects, and compare it with matched human-written code. Our results show that AI-generated code is A Large-Scale Comprehensive Measurement of AI-Generated Code in Real-World Repositories Conference’17, July 2017, Washington, DC, USA 

generally more verbose, more explicit, and more structurally lay-ered, while human-written code is more compact and relies more on reuse. At the process level, AI-associated commits are smaller and more localized, yet incur more follow-up modification and longer-term maintenance. These findings suggest that AI assistance does not simply change who writes code, but reshapes how code is expressed and evolved. It acts as a local productivity aid while shifting effort toward downstream refinement. By releasing our measurement pipeline, we provide a foundation for future studies to track the evolving impact of AI on software engineering in practice.
