## 摘要
仓库级自动程序修复（APR）越来越将强化定位视为提升修复性能的主要途径。我们提出一个更具针对性的问题：一旦定位得到加强，定位之后的哪些杠杆仍能带来可恢复的收益、哪些受限于我们的实验协议、又剩下哪些残余前沿？我们在 SWE-bench Lite 上对三种具有代表性的仓库级 RAG-APR 范式（Agentless、KGCompass 和 ExpeRepair）进行了研究。我们的实验协议结合了 Oracle Localization、池内 Best-of-K 选择、带固定接口的上下文探针（针对每种条件设置相同 token 填充控制）与同仓库的强负样本，以及统一封装的 oracle 检查。Oracle Localization 提升了这三种系统的性能，但 Oracle 的成功率仍低于 50%。在采样的 10 个补丁池中，额外的候选多样性仍然有帮助，但其提升空间很快饱和。在两个固定接口下，大多数信息量较高的附加上下文条件仍优于其匹配的控制条件。统一封装检查显示不同系统响应差异：在相同封装下，KGCompass 和 ExpeRepair 的增益依然显著，而 Agentless 对构建器选择更为敏感。提示层融合仍留有较大的残余前沿：最佳固定探针相比三系统原生的 Solved@10 并集仅多解决 6 个实例。总体而言，更强的定位、受限的搜索空间、证据质量以及接口设计共同影响仓库级修复的结果。

## Abstract
Repository-level automated program repair (APR) increasingly treats stronger localization as the main path to better repair. We ask a more targeted question: once localization is strengthened, which post-localization levers still provide recoverable gains, which are bounded within our protocol, and what residual frontier remains? We study this question on SWE-bench Lite with three representative repository-level RAG-APR paradigms, Agentless, KGCompass, and ExpeRepair. Our protocol combines Oracle Localization, within-pool Best-of-K, fixed-interface added context probes with per-condition same-token filler controls and same-repository hard negatives, and a common-wrapper oracle check. Oracle Localization improves all three systems, but Oracle success still stays below 50%. Extra candidate diversity still helps inside the sampled 10-patch pools, but that headroom saturates quickly. Under the two fixed interfaces, most informative added context conditions still outperform their own matched controls. The common-wrapper check shows different system responses: under a common wrapper, gains remain large for KGCompass and ExpeRepair, while Agentless changes more with builder choice. Prompt-level fusion still leaves a large residual frontier: the best fixed probe adds only 6 solved instances beyond the native three-system Solved@10 union. Overall, stronger localization, bounded search, evidence quality, and interface design all shape repository-level repair outcomes.

## 一、核心问题与研究动机

- **背景**：  
  仓库级自动程序修复（Repository-level Automated Program Repair, APR）要求大语言模型（LLM）在完整项目代码库中定位并修复 bug。由于仓库规模庞大，直接输入整个代码库不可行，因此近年来主流方法以检索增强生成（Retrieval-Augmented Generation, RAG）为核心，通过检索相关代码片段以构建“有限上下文”修复任务。

- **核心问题**：  
  现有研究普遍认为“定位质量越强→修复效果越好”，但缺乏对定位之后系统仍能提升的空间的定量、结构化研究。作者提出问题：  
  > 当定位被强化后，定位之后的哪些改进手段仍可带来可恢复收益？哪些收益被协议限制？又剩下哪些“残余前沿”（无法进一步修复的部分）？

- **目标意义**：  
  通过在 SWE-bench Lite 基准上开展控制实验，研究定位强化后仍存在的性能瓶颈，明确未来仓库级 APR 应改进的重点（搜索策略、上下文构建、接口设计等）。

## 二、方法论与技术框架

- **总体框架**：  
  论文提出一个“四步控制研究协议”，以系统化分析仓库级 RAG-APR 的定位后阶段。

  1. **Oracle Localization（预言定位）**：  
     由金标准补丁提取故障行并注入到各系统中，以模拟“理想定位情境”，测试定位的上限影响。
  
  2. **Best-of-K Sampling**：  
     每个样本生成 K=10 个候选补丁，通过理想选择（即若任一能修复则计为成功）衡量候选多样性的上界。
  
  3. **Fixed-Interface Added Context Probes**：  
     在相同接口中添加外部上下文信息（来自其他系统或矩阵形式的知识），同时添加 “同 token 数填充控制”和 “同仓库硬负样本” 作为对照，以分离信息有效性与长度效应。
  
  4. **Common-Wrapper Oracle Check**：  
     统一补丁验证接口以消除构建器差异，检验不同系统对于接口封装的依赖程度。

- **对比系统与差异点**：  
  - **Agentless**：分阶段工作流 + 压缩仓库视图 + 测试驱动筛选。  
  - **KGCompass**：基于知识图谱的函数级检索，提供结构化实体路径。  
  - **ExpeRepair**：双重记忆机制，利用历史示例与语义洞察支持迭代修复。  

- **研究变量**：  
  - 定位强度（普通 vs Oracle）
  - 搜索深度（K 值）
  - 上下文结构（单源 / 多源 / 控制）
  - 封装接口类型（原生 / 统一）

## 六、主要结论与发现

1. **定位仍是必要但不足的提升手段**：  
   Oracle 定位使成功率上升至约 40–43%，但仍低于 50%，说明“找到正确位置”并不足以确保成功修复。  

2. **搜索多样性有界收益**：  
   Best-of-10 相比单次 Oracle 提升约 10–13%，但收益在 K=5 时已达 86–87% 饱和。  

3. **上下文增强确实有效但长度非关键**：  
   信息丰富的上下文条件显著优于控制，且性能与 token 数不完全正相关；有效结构比“更多文字”更重要。  

4. **提示融合（UnionContext）并不总更好**：  
   多源拼接可能稀释单源的稳定收益；个别情况下，长提示导致性能下降。  

5. **残余前沿仍广泛存在**：  
   即使整合所有增强提示，剩余 101 个任务未解决，覆盖 11/12 个仓库；主要失败类型为错误 API 调用、缺失导入、修复不完整。

## 七、优点与亮点

- **系统化的定位后因素分解**：首次将仓库级 APR 的提升源细分为定位、搜索、上下文、封装四层。
- **控制实验严密**：引入同 token 填充、硬负样本、共享封装等对照，分离长度效应与信息效应。
- **跨范式分析**：比较知识图谱、记忆增强、分阶段控制三类主流 RAG-APR 体系。
- **统计稳健性高**：多重置信区间与显著性检验保证结果客观。
- **残余分析深入**：定量识别“仍无法修复”的前沿问题，为今后研究指明方向。
