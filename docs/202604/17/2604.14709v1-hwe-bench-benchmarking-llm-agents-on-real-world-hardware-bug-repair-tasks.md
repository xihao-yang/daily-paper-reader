## 摘要
现有的硬件设计基准测试主要在独立的组件层面评估大语言模型（LLMs），例如根据规范生成HDL模块，而缺乏对代码库级任务的系统评估。我们提出了HWE-Bench，这是首个面向真实世界硬件缺陷修复任务的大规模代码库级基准测试，用于评估LLM智能体的性能。HWE-Bench由417个任务实例组成，这些实例来源于六个主要开源项目中的真实历史缺陷修复拉取请求，涵盖Verilog/SystemVerilog和Chisel语言，涉及RISC-V内核、SoC以及安全根信任模块。每个任务均在一个完全容器化的环境中进行，智能体需根据真实缺陷报告进行修复，正确性通过项目原生的仿真与回归流程验证。该基准通过高度自动化的流水线构建，便于高效扩展至新的代码库。我们使用四种智能体框架对七个LLM进行了评估，结果显示最佳智能体总体修复成功率为70.7%，在小型内核项目上表现超过90%，但在复杂的SoC级项目上下降至65%以下。我们发现不同模型间的性能差距比软件基准中普遍报道的更大，难度主要受项目规模和缺陷类型分布驱动，而非仅由代码规模决定。故障分析表明，智能体的失败可追溯至调试过程中的三个阶段：故障定位、硬件语义推理以及在RTL、配置和验证组件间的跨工件协作。这些发现为开发更强的硬件感知型智能体提供了具体方向。

## Abstract
Existing benchmarks for hardware design primarily evaluate Large Language Models (LLMs) on isolated, component-level tasks such as generating HDL modules from specifications, leaving repository-scale evaluation unaddressed. We introduce HWE-Bench, the first large-scale, repository-level benchmark for evaluating LLM agents on real-world hardware bug repair tasks. HWE-Bench comprises 417 task instances derived from real historical bug-fix pull requests across six major open-source projects spanning both Verilog/SystemVerilog and Chisel, covering RISC-V cores, SoCs, and security roots-of-trust. Each task is grounded in a fully containerized environment where the agent must resolve a real bug report, with correctness validated through the project's native simulation and regression flows. The benchmark is built through a largely automated pipeline that enables efficient expansion to new repositories. We evaluate seven LLMs with four agent frameworks and find that the best agent resolves 70.7% of tasks overall, with performance exceeding 90% on smaller cores but dropping below 65% on complex SoC-level projects. We observe larger performance gaps across models than commonly reported on software benchmarks, and difficulty is driven by project scope and bug-type distribution rather than code size alone. Our failure analysis traces agent failures to three stages of the debugging process: fault localization, hardware-semantic reasoning, and cross-artifact coordination across RTL, configuration, and verification components, providing concrete directions for developing more capable hardware-aware agents.

## 一、研究背景与核心问题

- **研究动机**：  
  随着大语言模型（LLM）在软件开发任务中表现卓越，研究者开始探索其在硬件设计与电子设计自动化（EDA）中的潜力。  
  现有硬件领域评测基准（如 VerilogEval、RTLLM、CVDP）主要集中在**组件级（component-level）代码生成任务**，例如从自然语言规范生成 Verilog 模块。  
  然而，这些任务**缺乏对真实项目级缺陷修复（repository-level bug repair）**的评估能力，无法衡量模型在复杂硬件工程环境下的综合调试能力。  

- **核心问题**：  
  > 能否构建一个“仓库级别（repository-level）”“执行验证（execution-grounded）”的统一评测体系，来测量LLM代理在真实硬件项目中的缺陷定位和修复能力？

- **研究意义**：  
  HWE-Bench 的出现填补了这一空白，使硬件设计中的 LLM 研究能像 SWE-bench 在软件领域那样实现可复现、可扩展的综合评估。

## 二、方法论与技术路线

### 1. 总体思路
- 提出 **HWE-Bench**：首个面向真实硬件缺陷修复任务的**仓库级基准测试（repository-level benchmark）**。
- 目标：评估LLM智能体在“阅读代码库—定位缺陷—生成修复补丁—通过仿真验证”全过程中的能力。
- 特点：任务来源真实历史Pull Request（PR），涵盖从RTL逻辑到验证脚本的完整上下文。

### 2. 自动化构建流水线
HWE-Bench通过一个**四阶段自动化流程**提取可验证任务（见文中 Figure 2）：

1. **仓库筛选**：
   - 从GitHub中筛选使用 Verilog/SystemVerilog 或 Chisel 的项目；
   - 筛选标准：>400 PR、>100 star，并有稳定开发历史；
   - 最终选取六个代表性项目。
2. **PR收集与过滤**：
   - 收集所有合并并关联 issue 的 PR；
   - 通过LLM分类代理执行语义过滤（排除文档更新、工具配置变更等非缺陷修复）；
   - 使用自动可行性评分（feasibility scoring）对候选PR进行排序，依据覆盖性、可复现度、仿真成本等指标。
3. **测试生成与验证**：
   - 自动生成测试脚本（test scripts）与环境准备脚本；
   - 使用 **Docker 容器**封装每个实例；
   - 基于“FAIL-to-PASS”模式验证真伪修复：在原始版本下应Fail，在应用补丁后应Pass。
4. **任务描述生成**：
   - 使用LLM生成自包含的问题描述文本，避免泄露补丁内容；
   - 通过自动与人工审查确保语义完整与评测公平性。

### 3. 容器化与执行验证
- 每个任务均在一个**完全容器化的EDA环境**中运行，复现真实硬件构建流程；
- 仿真工具覆盖：Synopsys VCS 和 Verilator；
- 验证标准：项目原生回归测试（regression flow）是否通过。

## 六、主要结论与发现

1. **总体性能**：
   - 最优模型 GPT-5.4 在 HWE-Bench 上解决率 **70.7%**；
   - 在 Ibex/CVA6/Caliptra 等小型核心项目上成功率 **>90%**；
   - 在复杂SoC（如 OpenTitan、Rocket Chip）上下降至 **<65%**。
2. **模型间差距显著扩大**：
   - 软件基准（SWE-bench）上模型仅差 <8%，而硬件任务上跨度达 >23%，表明硬件任务更能揭示模型能力差异。
3. **影响因素**：
   - 难度与**项目架构范围与缺陷类型分布**密切相关，而非单纯由代码行数决定；
   - 集成与配置类缺陷最具挑战。
4. **典型失败模式**（三类）：
   - **本地化错误**：误判缺陷位置；
   - **语义推理不足**：修补逻辑违反硬件时序或接口契约；
   - **跨工件协调失败**：修复RTL后未同步更新配置或验证文件。
5. **模型行为差异**：
   - Claude系列倾向静态分析，定位准但忽略运行时语义；
   - GPT-5.4 探索性强，能自发调整配置，但修改面广；
   - GLM 5.1 居开源模型首位，推理稳定；
   - DeepSeek与Qwen多次迭代但过度机械化；
   - Kimi局限于局部上下文，跨模块推理弱。
