## 摘要
Vibe编码本质上假定通过反馈循环对LLM生成的代码进行迭代式优化。虽然这种方法在传统软件任务中效果显著，但在运行时自适应系统中的可靠性仍不明确——尤其是在生成的代码未经过人工检查的情况下。本文研究了在集体自适应系统（Collective Adaptive Systems，CAS）中对LLM生成的自适应管理器进行基于反馈的自动化验证。我们关注验证循环中的关键挑战：如何在运行时检测生成代码的失效，以及如何以足够精确的方式报告问题，以便LLM能够修复它们。我们将自适应循环与vibe编码反馈循环相结合，通过以下两方面检查正确性：(i) 通用架构约束，以及 (ii) 形式化为功能约束逻辑（Functional Constraints Logic，FCL）的功能性约束——这是一种针对可能有限轨迹的、基于一阶时序逻辑的新形式逻辑。在“龙猎”CAS案例研究中，我们展示了细粒度的约束违背能提供可操作的反馈，通常在几次迭代内即可生成有效的自适应管理器，而简单的粗略度量式反馈往往停滞不前。我们的研究结果表明，反馈的精确度是确保vibe编码可靠性的主导因素，使得由无编程技能的领域专家设计的系统无需人工代码检查。

## Abstract
Vibe coding inherently assumes iterative refinement of LLM-generated code through feedback loops. While effective for conventional software tasks, its reliability in runtime-adaptive systems is unclear -- especially when generated code is not manually inspected. This paper studies feedback-based automated verification of LLM-generated adaptation managers in Collective Adaptive Systems (CAS). We focus on the key challenges of verification in the loop: how to detect failures of generated code at runtime and how to report them precisely enough for an LLM to fix them.   We combine the adaptation loop with a vibe-coding feedback loop where correctness is checked against (i) generic architectural constraints and (ii) functional constraints formalized in Functional Constraints Logic (FCL), a novel first-order temporal logic over potentially finite traces. Conducting the Dragon Hunt CAS case study, we show that fine-grained constraint violations provide actionable feedback that typically yields a valid adaptation manager within a few iterations, while simple coarse metric-based feedback often stalls. Our findings suggest that feedback precision is the dominant factor for reliable vibe coding in systems designed by domain experts with no programming skills, thereby obviating the need for human code inspection.

## 二、方法论：核心思想与关键技术

### 1. 整体框架

论文提出一种 **“适应循环 + 反馈循环”** 复合结构，以实现闭环自动验证与改进：
- **适应循环（Adaptation Loop）：** 由LLM生成的AM负责在CAS系统运行时，根据状态重新分配智能体组合（即“ensembles”）。  
- **反馈循环（Feedback Loop）：** 监控系统运行，检测约束违反，生成可供LLM理解的反馈报告，并据此促使LLM重写代码。

### 2. 多层约束验证机制

构建了三层约束层次，以实现 **逐层过滤不同类型错误**：
1. **通用约束（Generic Constraints）**：保证语法、接口、架构一致性（如函数缺失、重复分配）。  
2. **功能约束（Functional Constraints）**：使用新提出的 **功能约束逻辑（Functional Constraints Logic, FCL）** 表述时序行为要求。  
3. **结果度量（Metric Feedback）**：仅基于最终结果（如胜负、资源量）的粗粒度反馈。

### 3. FCL逻辑设计要点

- 一种基于**一阶时序逻辑**的扩展形式，用于描述**有限轨迹上的时序约束**。  
- 核心算子：  
  - `♦ⁿ_t φ`: 在当前步的窗口 [i, i+t) 中，命题 φ 至少成立 n 次。  
  - 支持有限轨迹边界操作符 **BEG** / **MAX**，避免边界误差。  
- 示例：  
  - “龙必须在前15步被攻击至少一次” → `♦¹₁₅(|Attack| ≥ 1)`  
  - “去洞穴必须最终发起攻击” → `(GoToCave ∧ MAX > 0) ⇒ ♦¹_MAX(Attack)`  
- 该逻辑在违约时可生成带时间窗口与变量取值的“可诊断反例”，从而转化为可供LLM修复的文本反馈。

### 4. 反馈生成与提示模板

每次迭代时：
1. 系统运行 → 验证器检测约束违反；  
2. 将违反项汇总为结构化报告（包括错误类型、时间片、关键变量状态）；  
3. 报告注入标准化提示模板 → 调用LLM → 生成修订版AM；  
4. 重复执行直至所有测试用例通过或达到最大迭代数。

## 三、实验设计

### 1. 实验场景

- **案例研究：Dragon Hunt（猎龙）CAS场景**  
  - 虚拟游戏体系，包含“村民”“龙”等智能体；村民为农夫或战士，需协调行动以在30步内击败巨龙。  
  - 包含多种动作与状态变量（Farm、Attack、GoToCave等）。  
  - 目的：验证系统能否自动合成正确的多步决策策略（经济建设→迁移→攻击）。

### 2. 对比方式

通过控制反馈粒度，比较三种反馈机制在收敛效率上的差异：
1. **Metrics-only:** 仅提供结果指标（如胜负、剩余HP等）；  
2. **Generic-only:** 提供语法与体系架构错误信息；  
3. **Full Constraint Feedback:** 包含架构约束与FCL功能约束的完整精细反馈。

### 3. 评价指标

- **收敛效率：** 达到“所有测试运行均通过验证”所需的反馈循环次数。  
- **稳定性与泛化：** 在多组初始状态与随机种子下是否能稳定实现目标行为。

## 六、主要结论与发现

1. **反馈精度决定收敛效率**
   - 完整约束反馈通常在少数迭代内收敛；
   - 粗略度量反馈易停滞或震荡，LLM难以识别具体问题。
2. **FCL约束提供明确修复信号**
   - 违反的逻辑公式能精确指示错误发生的时间区间和角色，从而引导LLM生成定向修复。  
3. **约束集相当于可执行需求**
   - 将领域专家的意图形式化为约束，能够在无需编程经验的前提下实现自动代码验证。  
4. **验证通过的AM能跨初始状态保持策略稳定**
   - 稳定表现为合理的阶段性规划（先种田积累经济，再集体攻击）。
