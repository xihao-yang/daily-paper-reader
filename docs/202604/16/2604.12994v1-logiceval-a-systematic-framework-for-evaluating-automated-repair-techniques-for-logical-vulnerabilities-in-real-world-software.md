## 摘要
软件中的逻辑漏洞源于程序逻辑缺陷，而非内存安全问题，这可能导致严重的安全失效。尽管现有的自动程序修复技术主要聚焦于修复内存破坏类漏洞，但由于其对漏洞代码及其预期行为的语义理解有限，因此在处理逻辑漏洞时表现不佳。另一方面，大型语言模型（LLMs）近年来在代码理解与修复方面的成功令人鼓舞。然而，目前还没有一个系统性框架用于分析此类技术在逻辑漏洞修复中的能力与局限。本文旨在系统评估传统方法与基于LLM的修复方法在应对现实世界逻辑漏洞中的表现。为支持评估工作，我们构建了首个包含86个具备实际安全影响并分配了CVE编号的逻辑漏洞数据集——LogicDS。我们还开发了一个系统性框架LogicEval，用于评估逻辑漏洞修复补丁。评估结果表明，编译与测试失败主要受提示敏感性、代码上下文丢失以及补丁定位困难等因素驱动。

## Abstract
Logical vulnerabilities in software stem from flaws in program logic rather than memory safety, which can lead to critical security failures. Although existing automated program repair techniques primarily focus on repairing memory corruption vulnerabilities, they struggle with logical vulnerabilities because of their limited semantic understanding of the vulnerable code and its expected behavior. On the other hand, recent successes of large language models (LLMs) in understanding and repairing code are promising. However, no framework currently exists to analyze the capabilities and limitations of such techniques for logical vulnerabilities. This paper aims to systematically evaluate both traditional and LLM-based repair approaches for addressing real-world logical vulnerabilities. To facilitate our assessment, we created the first ever dataset, LogicDS, of 86 logical vulnerabilities with assigned CVEs reflecting tangible security impact. We also developed a systematic framework, LogicEval, to evaluate patches for logical vulnerabilities. Evaluations suggest that compilation and testing failures are primarily driven by prompt sensitivity, loss of code context, and difficulty in patch localization.

## 二、方法与技术框架

### 2.1 核心思路
提出系统性框架 **LogicEval**，用于评估各种自动修复方法（包括传统、机器学习和 LLM 基线）在逻辑漏洞修复上的表现。  
该框架以**输入-修复-验证**三个阶段为核心，覆盖补丁生成、编译测试、语义推理等全过程。

### 2.2 输入组成
- **V、F**：漏洞代码与修复代码。  
- **D**：漏洞文本描述（通常来自 CVE）。  
- **VS**：行为规范（如 RFC 或项目文档）。  
- **Vctx**：上下文变量定义与全局声明。  
- **C、T**：编译与测试脚本。  

### 2.3 框架流程（文字化算法）
1. **补丁定位（Patch Localization）**：  
   - 通过手动或自动方法标注“核心修复块”（core fix），保证评测独立于定位误差。  
   - 仅评估单一修复点（single-hunk fix）以减少干扰。  
2. **补丁生成（Patch Generation）**：  
   - 将漏洞代码与辅助信息输入不同 AVR 模型（包括 LLM）。  
   - 支持多种提示配置：零样本（zero-shot）、少样本（few-shot）、链式思维（CoT）等。  
   - 输出修复代码及其自然语言解释。  
3. **补丁评估（Patch Evaluation）**：  
   - **编译与测试层**：检测补丁是否可编译、是否通过 PoC 测试。  
   - **语义推理层**：使用另一 LLM 作为“评审者”，比较补丁解释与真值修复说明（ground-truth reasoning）的相似度（含余弦相似度、LLM判定 J 值）。  
   - **手动复核层**：人工确认可行补丁与真实修复是否逻辑等价。

### 2.4 评估指标
- **C（Compilation Rate）**：可编译比例。  
- **T（Testing Rate）**：通过测试用例比例。  
- **CS（Cosine Similarity）/J（Judge Verdict）**：推理一致性。  
- **最终正确率（Correct）**：通过人工复核确认的正确修复比例。

## 六、主要结论与发现

1. **传统方法失效**：非学习与深度学习 AVR（SimFix、KNOD）在逻辑漏洞场景中生成的大部分补丁不合理、逻辑错误。  
2. **LLM 具潜力但仍不足**：  
   - LLM 修复能通过部分测试，但难生成语义一致补丁。  
   - 提供漏洞描述（D）或修复步骤（R）能显著提升质量。  
3. **Prompt 影响巨大**：  
   - 零样本提示效果最佳，而 CoT 提示常导致额外未定义变量（编译失败）。  
   - Prompt 设计需明确定位与上下文。  
4. **辅助信息至关重要**：  
   - 无任何文本信息时（P12）生成的补丁常错误地将漏洞误判为内存安全问题。  
5. **推理评估可行性**：  
   - LLM 作为“审查者”评估补丁解释的相似性，与人工判断的准确率可达 0.8。  
6. **整体成功率仍低**：  
   - 三大模型共修复 43 个真实漏洞，仅 5 个补丁达到完全正确。

## 八、不足与局限

- **数据规模有限**：LogicDS 仅含 43 真实漏洞（约 430 人工小时），尚不能涵盖更广泛漏洞类型。

- **生态限制**：未覆盖专有软件或闭源系统，评估主要基于开源 C/C++/Java 代码。  
- **算力与性能指标未披露**：不便于评估可扩展性和工业应用代价。
