## 摘要
大型语言模型（LLMs）能够根据自然语言生成代码，但它们在多大程度上能够捕捉到预期的程序行为仍不清楚。可执行的行为规范（Executable Behavioral Specifications）通过前置条件和后置条件定义，为评估这种理解提供了一种具体方式。然而，现有关于规范生成的研究在评估方法、任务设置以及规范的表达能力方面均存在局限。我们提出了 CodeSpecBench，一个基于执行评估协议的可执行行为规范生成基准。CodeSpecBench 支持函数级与代码库级任务，并将规范编码为可执行的 Python 函数。该基准数据集构建自多样化的真实代码库，使其能够现实地评估正确性（接受有效行为）与完整性（拒绝无效行为）。我们在 CodeSpecBench 上评估了 15 个最新的大型语言模型，结果显示在代码库级任务上的性能显著下降，其中表现最好的模型仅达到 20.2% 的通过率。我们进一步发现，规范生成比代码生成更具挑战性，这表明强大的代码生成能力并不一定反映对预期程序语义的深入理解。我们的数据与代码已开源于 https://github.com/SparksofAGI/CodeSpecBench。

## Abstract
Large language models (LLMs) can generate code from natural language, but the extent to which they capture intended program behavior remains unclear. Executable behavioral specifications, defined via preconditions and postconditions, provide a concrete means to assess such understanding. However, existing work on specification generation is constrained in evaluation methodology, task settings, and specification expressiveness. We introduce CodeSpecBench, a benchmark for executable behavioral specification generation under an execution-based evaluation protocol. CodeSpecBench supports both function-level and repository-level tasks and encodes specifications as executable Python functions. Constructed from diverse real-world codebases, it enables a realistic assessment of both correctness (accepting valid behaviors) and completeness (rejecting invalid behaviors). Evaluating 15 state-of-the-art LLMs on CodeSpecBench, we observe a sharp performance degradation on repository-level tasks, where the best model attains only a 20.2% pass rate. We further find that specification generation is substantially more challenging than code generation, indicating that strong coding performance does not necessarily reflect deep understanding of intended program semantics. Our data and code are available at https://github.com/SparksofAGI/CodeSpecBench.

## 一、研究背景与核心问题

- **研究动机**：当前大型语言模型（LLMs）在代码生成任务上表现出色，但尚不清楚它们是否真正理解程序的**预期行为语义**。  
- **核心问题**：如何量化和评估 LLM 是否能生成符合程序行为逻辑的**可执行行为规范（Executable Behavioral Specifications）**，即利用前置条件（preconditions）与后置条件（postconditions）来定义函数或程序的实际运行约束。  
- **研究缺口**：现有规范生成工作局限在：
  - 评估方法偏向**静态形式验证**；
  - 任务设置往往只覆盖**函数级场景**；
  - 规范形式缺乏**前后条件的可执行性与表达性**。  
- **论文目标**：构建一个统一的、执行型的评测框架，系统评估 LLM 对程序语义理解与行为约束的掌握程度，从而揭示代码生成能力与语义理解能力间的差距。

## 二、方法论与技术设计

### 1. 核心思想
- 提出 **CodeSpecBench** ——面向 LLM 的**可执行行为规范生成基准**；
- 以**执行式评估（Execution-based Evaluation）**替代传统的静态验证；
- 规范以**Python 可执行函数**（含 preconditions 和 postconditions）形式表示。

### 2. 功能结构
CodeSpecBench 分为两层：
- **Func-level（函数级）**：模型根据自然语言题意生成对应的前后条件函数；
- **Repo-level（仓库级）**：模型基于真实仓库问题报告及相应代码区，生成与函数交互的前后条件验证组件。

### 3. 评估逻辑
通过运行测试用例以验证两方面：
- **正确性（Correctness）**：规范是否接受有效输入及行为；
- **完整性（Completeness）**：规范是否能拒绝无效输入或错误输出；
- 最终指标为**Pass Rate**：需同时满足正确性和完整性。  

### 4. 执行机制说明（文字流程）
1. 模型生成 preconditions/postconditions；
2. 在真实测试环境中执行规范；
3. 对比“正确行为”与“错误行为”样例，判断规范能否准确区分；
4. 仓库级任务中通过插入 pytest 钩子 (`pytest_sessionstart`) 自动执行规范。

### 5. 数学定义（关键指标）
Pass Rate = 两维度同时满足：
> 若 `spec` 接受所有正确测试且拒绝所有错误测试 → 计为通过；否则为失败。

## 三、实验设计

### 1. 数据集与任务场景
- **函数级数据集（CodeSpecBench-Func）**  
  - 基于 **LeetCode-Dataset (2025)**，共 **2,494 个问题**；
  - 每个问题包含约 **217.8 个测试用例**（100 个正确 + 117.8 个错误）；
  - 测试样例来源：多模型生成 + 在线评测验证 + Buggy输出构造。

- **仓库级数据集（CodeSpecBench-Repo）**  
  - 基于 **SWE-bench Verified (Jimenez et al., 2024)**；
  - 包含 **12 个开源项目共 500 个问题**；
  - 任务来自真实仓库 issue（bug report / feature request），经过 UTBoost 扩展测试。

- 两层任务分别对应模型对**自包含函数理解**与**跨模块语义约束**的捕捉能力。

### 2. 评测对象
- 共评估 **15 个主流 LLM**：
  - 开源模型：Qwen3（0.6B–32B）、QWQ-32B、GPT-OSS(20B/120B)、DeepSeek-V3.2；
  - 商业模型：Gemini-2.5（Flash/Pro）、GPT-5（mini/full）、Claude-4.5-Sonnet；
- 部分模型启用了**推理模式（reasoning mode）**以增强逻辑性。

### 3. 对比与验证
- 与现有基准（如 FormalBench、VERINA、nl2postcond）进行对照；
- 不同模型规模与推理模式均有横向对比；
- 同时评估**规范生成**与**代码生成**的难度对比。

## 六、主要结论与发现

1. **模型理解力有限**：  
   生成可执行规范远难于生成代码，函数级最佳通过率仅 47%，仓库级仅 20.2%。  
2. **规模与推理帮助有限**：  
   模型规模扩大或启用 reasoning 能提升函数级表现，但在复杂仓库场景中难以延伸。  
3. **代码生成 ≠ 语义理解**：  
   高代码准确率不代表模型能形成正确的行为约束。  
4. **错误特征显著**：  
   - 函数级失败主要源于输入类型/范围约束错误（占 ~50%）；  
   - 仓库级失败集中于依赖解析与过度约束（占 ~30%）。  
5. **执行性瓶颈**：  
   弱模型生成的规范往往无法在仓库环境中成功执行（可执行率低于 10%）。  
6. **部分正向效果**：  
   使用模型生成的规范可用于指导代码重排序（re-ranking），提升代码生成精度约 2–3%。
