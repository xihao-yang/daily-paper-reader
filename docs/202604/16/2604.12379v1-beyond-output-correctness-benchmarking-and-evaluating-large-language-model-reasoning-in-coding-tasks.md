## 摘要
大型语言模型（LLMs）在解决编码任务时越来越依赖显式推理，但评估这种推理质量仍然具有挑战性。现有的推理评估器并非为编码任务设计，而当前的基准测试主要关注代码生成，使得其他编码任务在很大程度上未被探索。我们提出了 CodeRQ-Bench，这是首个用于评估 LLM 在三类编码任务——生成、摘要和分类——中推理质量的基准。利用该基准，我们分析了来自现有评估器的 1,069 个不匹配案例，识别出五个反复出现的局限性，并提出了四项针对编码任务推理评估的设计洞见。在这些洞见的指导下，我们提出了 VERA，这是一种结合基于证据的验证与考虑歧义的得分修正的两阶段评估器。CodeRQ-Bench 上的实验表明，VERA 在四个数据集上持续优于强基线模型，AUCROC 性能提升最高达 0.26，AUPRC 提升最高达 0.21。我们在 https://github.com/MrLYG/CodeRQ-Bench 发布了 CodeRQ-Bench，以支持未来的研究。

## Abstract
Large language models (LLMs) increasingly rely on explicit reasoning to solve coding tasks, yet evaluating the quality of this reasoning remains challenging. Existing reasoning evaluators are not designed for coding, and current benchmarks focus primarily on code generation, leaving other coding tasks largely unexplored. We introduce CodeRQ-Bench, the first benchmark for evaluating LLM reasoning quality across three coding task categories: generation, summarization, and classification. Using this benchmark, we analyze 1,069 mismatch cases from existing evaluators, identify five recurring limitations, and derive four design insights for reasoning evaluation in coding tasks. Guided by these insights, we propose VERA, a two-stage evaluator that combines evidence-grounded verification with ambiguity-aware score correction. Experiments on CodeRQ-Bench show that VERA consistently outperforms strong baselines across four datasets, improving AUCROC by up to 0.26 and AUPRC by up to 0.21. We release CodeRQ-Bench at https://github.com/MrLYG/CodeRQ-Bench, supporting future investigations.

## 二、方法论与核心设计

### 1. CodeRQ-Bench：推理质量评估基准

- **目的**：首个专为代码任务设计的推理质量评估基准，覆盖三种输出模态：

  - **代码分类 / 缺陷检测（Classification）**

### 2. 对现有评估器的系统性分析

- 对 ReCEval、SocREval、CaSE 在 CodeRQ-Bench 上的 1,069 条错判进行诊断分析，识别出五类系统性局限：
  1. **Evidence-Ungrounded Assessment**：未验证代码相关事实。  
  2. **Ambiguity Assessment Deficit**：无法识别任务的不确定性。  
  3. **Score Aggregation Failure**：聚合策略失衡导致结果失真。  
  4. **Self-Generated Reference Bias**：评估依赖自生成标准，形成循环偏差。  
  5. **Code-Unaware Assessment**：缺乏对代码语义和逻辑的理解。  
- 从分析中提炼出四项设计洞见：
  1. 评估应**基于证据进行验证**。  
  2. 应具备**歧义识别与不确定性惩罚机制**。  
  3. 应**避免自参照评分体系**。  
  4. 应**融入代码感知语义评估**。

---

### 3. VERA：两阶段推理质量评估器

- **核心思想**：  
  构造一个结合**证据验证（Evidence-grounded Verification）**与**歧义惩罚（Ambiguity-aware Correction）**的自动化评分系统。
- **两阶段流程**：

  **Stage 1：验证型推理评估**
  - 输入：任务描述、推理链 \(C = (s_1, s_2, …, s_n)\)、生成输出 \(o\)。  
  - 授予评估模型搜索能力，通过外部检索验证代码行为、API文档等事实。  
  - 根据技术正确性输出评分 \(r∈[1,10]\)，归一化为 \(p = (r-1)/9\)。

  **Stage 2：歧义感知得分修正**
  - 计算任务歧义度 \(a \in [0,1]\) 与推理处理质量 \(h \in [0,1]\)。  
  - 应用惩罚项公式：  
    \[
    δ = 
      \begin{cases} 
        a \cdot \min(2h - 1, 0), & a ≥ τ \\
        0, & \text{otherwise}
      \end{cases}, \quad τ = 0.4
    \]
  - 最终得分 \( \mathrm{VERA}(q,C,o) = \max(p + δ, 0) \)。  
  - 当推理对不确定问题表现过度自信时，会触发负向修正。

## 三、实验设计与比较对象

### 1. 基准与数据设置
- 使用 **CodeRQ-Bench** 四个子数据集，覆盖三类编码任务。  
- 样本由 GPT-4o 生成推理文本，由专家人工标注正确性。

### 2. 对比方法
共计六个基线方法：
1. **ReCEval** — 传统的逻辑一致性+信息量评估。  
2. **SocREval** — 苏格拉底式对话法推理评估。  
3. **CaSE** — 因果逐步推理一致性评分。  
4. **MAD\*** — 多代理辩论评估（改为连续分值）。  
5. **AutoRace\*** — 基于任务自适应规则的自动评估框架。  
6. **LLM-as-Judge** — 直接使用 LLM 打分法。

### 3. 评价指标
- **相关性指标**：Somers’ D、Spearman’s ρ  
- **区分性指标**：AUCROC、AUPRC  
- **结果统计**：均取三次实验平均值并报告显著性水平。

## 六、主要结论与发现

- **VERA 性能显著优于所有基线**：  
  - AUCROC 提升最高达 **0.26**，AUPRC 提升最高达 **0.21**。  
  - Somers’ D 与 Spearman’s ρ 均显著提高，说明排序一致性增强。  
- **Stage-1（证据验证）是主要改进来源**，对生成任务尤为关键；
- **Stage-2（歧义修正）在任务不确定时能防止过度评分**；
- **现有评估器普遍受限于语言逻辑判断而非代码语义判断**；
- **CodeRQ-Bench 有效揭示跨任务性能差异**，强调评估应超越代码生成任务本身。
