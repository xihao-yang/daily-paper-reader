## 摘要
最先进的代码生成框架依赖于“心理模拟”，即大型语言模型在内部追踪执行过程以验证代码正确性。我们揭示了一个根本性的局限：心理—现实鸿沟（Mental-Reality Gap）——模型会幻觉性地生成执行轨迹，并自信地验证存在缺陷的代码。这种鸿沟在两个相互独立的维度上体现：规格鸿沟（Specification Gap，规划阶段忽略边界情况）和验证鸿沟（Verification Gap，对有缺陷的代码幻觉出正确行为）。我们提出了基于一个简单原则的 SolidCoder：不要想象——要执行。S.O.L.I.D. 架构通过在算法设计前强制进行边界情况意识，并用基于属性的神谕（property-based oracles）在沙盒环境中替代想象的执行轨迹，从而解决这两个维度的问题。在 GPT-4o 上，SolidCoder 实现了最先进的 pass@1 表现：HumanEval 上 95.7%（+0.6%p），CodeContests 上 77.0%（+4.3%p），APPS 上 26.7%（+3.4%p）。消融实验表明，边界情况意识带来最大的单项提升，而执行验证能够捕获规格改进无法覆盖的不同类型错误。这些提升可推广至经过强化学习后训练的模型，验证了弥合这两个维度的鸿沟对于实现稳健的代码合成至关重要。我们开源了代码与框架，以促进未来研究。

## Abstract
State-of-the-art code generation frameworks rely on mental simulation, where LLMs internally trace execution to verify correctness. We expose a fundamental limitation: the Mental-Reality Gap -- where models hallucinate execution traces and confidently validate buggy code. This gap manifests along two orthogonal dimensions: the Specification Gap (overlooking edge cases during planning) and the Verification Gap (hallucinating correct behavior for flawed code). We propose SolidCoder with a simple principle: don't imagine -- execute. The S.O.L.I.D. architecture addresses both dimensions by forcing edge-case awareness before algorithm design and replacing imagined traces with sandboxed execution using property-based oracles. With GPT-4o, SolidCoder achieves state-of-the-art pass@1 performance: 95.7% on HumanEval (+0.6%p), 77.0% on CodeContests (+4.3%p), and 26.7% on APPS (+3.4%p). Ablation reveals that edge-case awareness provides the largest individual gain, while execution grounding catches categorically different errors that specification improvements cannot address. These gains generalize to RL post-trained models, validating that bridging both gap dimensions is essential for robust code synthesis. We release our code and framework to facilitate future research.

## 一、研究背景与问题定义

- **研究动机**  
  当前大规模语言模型（LLMs）在代码生成领域已展现出卓越能力，但其验证程序正确性的方式主要依赖“**心理模拟（mental simulation）**”——即模型在自身内部想象代码执行过程，凭“推理轨迹”判断结果是否正确。  
  然而，作者指出，这种方式存在典型的幻觉风险：模型会**自信地验证错误代码**，产生“心理—现实鸿沟（Mental-Reality Gap）”。

- **问题实质**  
  这种鸿沟包括两个独立维度：  
  1. **规格鸿沟（Specification Gap）**——规划阶段遗漏边界条件或特殊输入。  
  2. **验证鸿沟（Verification Gap）**——验证阶段模型“幻觉”出正确执行轨迹，错判错误逻辑为正确。  

- **研究目标**  
  本文旨在通过“具体执行（concrete execution）”替代想象演算，使代码生成过程形成**真实、可验证的闭环**，实现更稳健的代码合成。

## 二、方法论：S.O.L.I.D. 架构与核心技术

### 1. 核心理念  
> “不要想象 —— 要执行（Don’t imagine — Execute）”

LLM不再依靠文字推理模拟代码，而是**在沙盒中真正执行代码**，并用**属性驱动的验证预言机（property-based oracles）**取代对真值输出的依赖。这使模型可在缺乏标准答案时仍能检测逻辑错误。

### 2. S.O.L.I.D. 架构五大组件

| 组件 | 功能 | 作用维度 |
|------|------|-----------|
| **[S] Shift-left Planning** | 在算法设计前强制识别边界输入（如空输入、最大值、特殊模式），在方案设计阶段就纳入异常场景 | 缩小 *规格鸿沟* |
| **[O] Oracle-based Assertions** | 以属性或不变式形式构造测试（如排序保持长度、输出为输入的排列），无需确切输出 | 缩小 *验证鸿沟* |
| **[L] Live Execution** | 在受限沙盒中真实运行生成代码，并收集实际断言结果 | 消除幻觉验证 |
| **[I] Intermediate Simulation** | 在真实执行前进行一次轻量“心理追踪”，作为预筛选，降低无效执行的成本 | 提高效率 |
| **[D] Defensive Accumulation** | 持续积累已发现的失败测试用例，要求后续迭代全部通过，防止“回归错误” | 提升稳健性 |

### 3. 工作流程（Algorithm 1 概略）
1. 识别潜在边界输入（Shift-left Planning）
2. 生成算法计划并模拟验证
3. 将计划转译为代码
4. 执行中间模拟（Intermediate Simulation）
5. 生成属性测试（Oracle-based Assertions）
6. 在沙盒中执行（Live Execution）
7. 记录失败案例并迭代修改（Defensive Accumulation）
8. 若所有积累测试通过则终止；否则重复规划或调试循环。

该流程在推理阶段形成动态闭环，使计划设计和执行验证彼此反馈。

## 六、主要结论与发现

1. **总体性能提升显著**  
   - GPT-4o：HumanEval 95.7%(+0.6%p)、CodeContests 77.0%(+4.3%p)、APPS 26.7%(+3.4%p)  
   - 提升主要集中在中等难度任务（CodeContests），验证失败率显著下降。

2. **消融研究发现**
   - 删除 `[S]` 导致 –23.7%p 性能损失，说明边界识别影响最大。  
   - 删除 `[L]` 导致 –7.9%p，确认“真实执行”捕获类别上不同错误。  
   - 各组件协同效果明显。

3. **执行验证比心理模拟更可靠**  
   - 案例显示“心理模拟”可误判错误逻辑，真实执行能即时暴露 bug。

4. **方法具备通用性**  
   - 在不同模型（包括 RL 后模型）中仍保持性能提升。

## 八、不足与局限

- **语言与环境限制**：目前仅支持 **Python**，需要针对其他语言重新实现沙盒。  
- **模型依赖性**：Oracle 属性生成与代码验证均依赖同一 LLM，可能传播系统性偏差。  
- **评测范围**：仅涵盖函数级任务，未验证在仓库级（如 SWE-bench）上的效果。

- **数据覆盖与公平性**：虽保持条件一致，但仅使用三个模型，未包括 Gemini、Claude 等大型闭源模型。  
- **安全与伦理风险**：实际执行未信任代码存在潜在安全隐患，必须依赖沙盒隔离。
