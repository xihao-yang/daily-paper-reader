## 摘要
API 的自动化黑盒测试通常依赖于定义可用操作和数据模式的接口规范，但这些规范对行为语义的描述有限或不存在。这一语义差距加剧了测试判定问题（test-oracle problem），并限制了有效、有状态的调用序列的生成。我们提出了 IcePick，一个通过模型检验（model checking）实现系统性状态空间覆盖的 API 测试框架。IcePick 使用 TLA+ 对 API 的状态演化进行形式化建模，利用 TLC 模型检验器穷尽性地探索可达状态，并生成能够在行为模型上得到形式化覆盖保证的测试序列。为缓解状态空间爆炸并改进序列提取，我们提出了一种覆盖引导的广度优先遍历方法，用于探索 TLC 状态空间图。为了解决仅能依赖 HTTP 状态码的测试判定限制，我们提出 Glacier——一种一阶逻辑契约语言，用于在 API 规范中扩展可执行的语义契约，从而在测试执行过程中实现自动化行为验证。我们在 EvoMaster Benchmark 系统上评估了 IcePick，结果显示基于模型检验的探索能够实现完整的状态覆盖，并揭示多操作交互中的缺陷。我们还分析了其可扩展性，以刻画其实际应用中的限制与适用条件。总体而言，IcePick 为关键的基于 API 的系统提供了具有强覆盖保证的可重现测试套件。

## Abstract
Automated black-box testing of APIs typically relies on interface specifications that define available operations and data schemas, but offer limited or no behavioural semantics. This semantic gap amplifies the test-oracle problem and limits the generation of effective, stateful call sequences.   We introduce IcePick, a framework that achieves systematic state-space coverage for API testing by leveraging model checking. IcePick uses TLA+ to formally model API state evolution, employs the TLC model checker to exhaustively explore reachable states, and generates test sequences that provably cover the behavioural model. To mitigate state-space explosion and improve sequence extraction, we introduce a coverage-guided breadth-first traversal of the TLC state-space graph. To address oracle limitations beyond HTTP status codes, we propose Glacier, a first-order logic contract language that enriches API specifications with executable semantic contracts, enabling automated behavioural verification during test execution.   We evaluate IcePick on EvoMaster Benchmark systems, demonstrating that model-checking-guided exploration achieves complete state coverage and reveals faults in multi-operation interactions. We also analyse scalability to characterise practical limits and applicability requirements. Overall, IcePick provides reproducible test suites with strong coverage guarantees for critical API-based systems.

## 1. 研究问题与背景

### 核心问题
- 当前自动化 **黑盒 API 测试** 主要依赖接口规范（如 OpenAPI Specification, OAS），仅定义操作与数据模式，但缺乏**行为语义**。
- 此语义缺口导致：
  - **测试预言问题（Oracle Problem）**：难以判断输出是否符合期望；
  - 生成的测试序列多为无状态或随机性的调用，无法覆盖复杂系统状态变化。
- 因而迫切需要一种能够：
  - 形式化描述 API 状态演化，
  - 系统化探索可达状态空间，
  - 自动生成带有语义判断能力的测试用例的框架。

### 研究目标
- 融合 **模型检验（Model Checking）** 与 **契约式验证（Contract-based Testing）**，
- 提出一种可实现“**系统化 API 状态空间探索与行为验证**”的方法，
- 既解决测试覆盖性不足问题，也提升测试预言的精确性。

## 2. 方法论与关键技术

### 框架总览：ICE PICK
ICE PICK 是一个双阶段的黑盒测试框架，结合了 **形式建模、模型检验、语义契约生成与可执行验证**。

#### 核心组成
1. **GLACIER 契约语言**：
   - 一阶逻辑形式定义的 **可执行语义契约**。
   - 自动从 OAS 文件生成 CRUD（创建、读取、更新、删除）语义契约。
   - 包含预条件（requires）、后置条件（ensures）、系统不变式（invariants）。
   - 支持量化表达式与 `prev` 操作符，能引用操作执行前状态。

2. **TLA+ 模型建模与 TLC 检验**：
   - 将 RESTful API 状态转化为 **有限状态模型**。
   - 使用 TLC 模型检验器穷举可达状态并验证不变式。
   - 将 API 操作转化为 **TLA+ Actions**，并自动生成状态空间图（SSG）。

3. **呼叫序列生成算法（基于 BFS 覆盖）**：
   - 从 SSG 中通过 **覆盖引导的广度优先搜索（Coverage-guided BFS）** 生成最短路径。
   - 算法包括：
     - 阶段 1：收集从初始状态到终末状态的路径；
     - 阶段 2：反向收集从各状态到终态的最短路径；
     - 阶段 3：拼接不完整路径，保证完全状态覆盖。
   - 保证全状态覆盖（All-State Coverage），在无并行边时也保证全转换覆盖。

4. **自动化执行与判定模块：**
   - 自动生成输入数据（依据 JSON schema）。
   - 执行生成序列，对每次操作验证：
     - 预条件、后置条件、不变式；
     - HTTP 响应码一致性。
   - 测试结果分为 `OK / WARN / ERR / NOT_TESTED` 四类，能提供细粒度诊断信息。

## 3. 实验设计

### 实验平台与对象
- 测试环境：基于 **EvoMaster Benchmark (EMB)** 的多系统集合。
- 被测系统：

  - **Swagger Petstore**
  - **E-Commerce API**

### 实验组成
1. **适用性分析（Applicability Analysis）**：
   - 检查 OAS 规范的完整性与 REST/HTTP 合规性。
   - 识别无法从 OAS 正确生成契约的情况（如错误状态码、缺失 body 定义）。

2. **状态空间生成（SSG Generation）**：
   - 不同模型规模配置，如 `p4_t3_e4` 表示 4 玩家、3 比赛、4 报名实例。
   - 评估 TLC 的可扩展性与时间/存储开销。

3. **调用序列生成实验**：
   - 在不同 SSG 下运行 BFS 算法，检测状态与转换覆盖度。

4. **测试执行与故障检测**：
   - 注入三类逻辑错误（删除未生效、删错对象、参照完整性错误）。
   - 比较自动生成契约与人工扩展契约的检测差异。

## 6. 主要结论与发现

- **完整状态覆盖**：在多系统上实现 100% 状态覆盖，部分场景 74–84% 转换覆盖。
- **故障发现能力**：
  - 成功检测所有注入的 3 类缺陷；
  - 识别 Features-Service 中 97 个结构性错误（如缺失请求体定义）。
- **契约生成有效**：
  - 自动生成的 CRUD 契约即可发现核心错误；
  - 手动扩展契约并未显著提升检测率。
- **可扩展性结论**：
  - TLC 模型规模在 5–21 GB 范围内可在 1 小时内完成；
  - 超大模型（>400 GB）显著降低实用性。
- **REST 合规性的重要性**：
  - 若 API 设计不遵循 HTTP 语义及资源导向原则，测试不可行。
