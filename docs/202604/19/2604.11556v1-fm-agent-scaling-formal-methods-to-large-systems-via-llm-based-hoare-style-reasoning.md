## 摘要
由大型语言模型（LLM）辅助的软件开发已日益普遍，能够生成诸如编译器等大型系统。因此，增强生成代码的正确性变得至关重要。然而，由于代码复杂性，针对大型系统的自动化推理仍面临挑战。Hoare逻辑提供了一种将大型系统分解为较小组件并分别进行推理的方法（即组合推理）。但现有研究在扩展性方面仍然困难，因为Hoare逻辑要求为每个函数编写形式化规范，这对人工带来了沉重负担。当代码由LLM生成时，这一问题进一步加剧，因为开发者通常不了解每个函数的预期行为。本文提出了FM-Agent，这是首个实现针对大型系统的自动化组合推理的框架。借助LLM，FM-Agent引入了一种自顶向下的范式来自动生成函数级别规范。具体而言，FM-Agent根据调用者对函数行为的期望推导函数的规范，使得生成的规范能够反映开发者的意图，即使实现存在缺陷。开发者的意图通常以自然语言表达，而现有验证器仅支持公式。因此，FM-Agent将Hoare风格推理推广，用于针对自然语言规范进行函数推理。最后，为了确认漏洞的存在并解释其原因，FM-Agent会自动生成测试用例以触发潜在错误。在我们的评估中，FM-Agent成功在两天内推理了多达143k行代码的多个大型系统。这些系统已被其开发人员测试过，但FM-Agent仍然发现了522个新漏洞。这些漏洞可能导致严重后果，包括系统崩溃和错误的执行结果。

## Abstract
LLM-assisted software development has become increasingly prevalent, and can generate large-scale systems, such as compilers. It becomes crucial to strengthen the correctness of the generated code. However, automated reasoning for large-scale systems remains challenging due to code complexity. Hoare logic offers an approach to decomposing a large system into smaller components and reasoning about them separately (i.e., compositional reasoning). However, existing works still struggle to scale, because Hoare logic requires writing formal specifications for each function, imposing a heavy human burden. The problem is exacerbated when code is generated by LLMs, as developers lack a deep understanding of each function's expected behavior.   This paper presents FM-Agent, the first framework that realizes automated compositional reasoning for large-scale systems. Leveraging LLMs, FM-Agent introduces a top-down paradigm to automatically generate function-level specifications. Specifically, FM-Agent derives the specification of a function from how its callers expect the function to behave, so the generated specifications can reflect the developer's intent of a function even if the implementation is buggy. Developers' intent is usually expressed in natural language, while existing verifiers only support formulas. Therefore, FM-Agent generalizes Hoare-style inference to reason about functions against natural-language specifications. Finally, to confirm bug existence and explain bug causes, FM-Agent automatically generates test cases to trigger potential bugs. In our evaluation, FM-Agent successfully reasons about large-scale systems within 2 days, each of which has up to 143k LoC. These systems have already been tested by their developers, but FM-Agent still finds 522 newly discovered bugs. These bugs can cause serious consequences, including system crashes and incorrect execution results.

## 一、研究问题与背景  

- **核心问题**：在大型语言模型（LLM）生成代码的时代，自动化开发出的系统可达数十万行代码，其可靠性与正确性成为核心挑战。传统的形式化验证方法（如 Hoare 逻辑）虽具理论完备性，但受限于人工撰写规范的高成本，难以应用于规模庞大的系统。  
- **研究动机**：  
  - LLM 生成代码常包含隐性错误和逻辑瑕疵。  
  - 人工编写函数规格（pre/post conditions）对开发者而言负担巨大。  
  - 现有自动化验证工具（符号执行、定理证明器）在面对复杂系统时易出现路径爆炸和扩展性不足。  
- **总体目标**：提出一种可自动生成函数级自然语言规范并进行组合性推理（compositional reasoning）的框架，使得形式化方法可以在大型系统中可扩展地运行。

## 二、方法论与核心技术  

### 1. 整体框架结构

### 2. 核心思想  

- **自顶向下（Top-Down）规范生成**：  
  - 传统形式化验证通常自底向上，先为被调用函数写规格。  
  - FM-Agent改用自顶向下，由系统入口函数出发，通过分析各调用者如何使用其被调用函数，推导出调用者对被调用函数的“期望规格（Expected Specification）”。  
  - 相比基于实现生成规格的方式，更能体现开发者设计意图，避免由错误实现导致的错误传播。

- **自然语言化 Hoare 推理**：  
  - 扩展 Hoare 三元组 `{P} C {Q}` 至自然语言形式的前后条件。  
  - 使用 LLM 来在自然语言层面进行推理，即让模型推导每个语句块的后置条件（Post-condition）并验证其是否蕴涵函数规格的后置条件。  
  - 若推理路径无法满足规格，FM-Agent 将标记为潜在漏洞。

- **Bug Triggering 与验证机制**：  
  - 对未通过推理的函数，模型自动生成系统级输入测试用例（非仅函数单元测试）。  
  - 利用 LLM 理解系统整体数据流，在入口处构造触发路径，从而确认并解释漏洞。  

---

### 3. 关键算法流程  

- **规范生成顺序算法（Algorithm 1）**  
  - 构造函数调用图 `G(V,E)`  
  - 查找强连通分量（SCC）解决递归调用互依问题  
  - 压缩图为 DAG 并执行分层拓扑排序  
  - 每层函数集可 **并行** 生成规格，提高扩展性  

- **多调用者规格合成公式（基于 Definition 1）**：  
  ```
  P_F = (P1 ∨ P2 ∨ ... ∨ Pn)
  Q_F = (Q1 ∧ Q2 ∧ ... ∧ Qn)
  ```
  说明：若函数 F 被多个调用者调用，其前置条件取析取（任何调用环境均可满足），后置条件取合取（需同时满足各调用期望）。

- **推理与验证过程**：  
  - 对条件语句采用自然语言“分支推理”、对循环尝试生成循环不变式并验证一致性。  
  - 判断自然语言条件精确度：若翻译为公式后含有“Uninterpreted functions”，则放弃形式求解，仍采用自然语言推理。

## 八、不足与局限  

- **不保证形式化的完备性与严谨性**：FM-Agent不具备传统验证的*soundness*保证，可能有误报。  
- **依赖 LLM 推理质量**：易受模型幻觉（hallucination）影响，存在 false positives。  
- **目前仅处理顺序程序**：不支持并发与分布式系统推理，未来需扩展支持 rely-guarantee 或并发分离逻辑。  
- **计算成本高**：两天内消耗数十亿 token，成本较高。  
- **缺乏与传统形式化工具的深度对比**：未验证与 symbolic execution、SMT solver 联合效果。
