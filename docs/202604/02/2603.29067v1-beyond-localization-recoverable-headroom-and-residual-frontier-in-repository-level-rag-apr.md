## 摘要
在存储库级自动程序修复（APR）中，增强定位能力日益被视为改进修复性能的主要途径。本文提出一个更具针对性的问题：当定位得以强化后，哪些定位后的调节因素仍可带来可恢复的收益，哪些在我们的协议内受到限制，又存在哪些残余前沿？我们在 SWE-bench Lite 基准上，研究了三种具有代表性的存储库级 RAG-APR 范式：Agentless、KGCompass 和 ExpeRepair。我们的实验协议结合了 Oracle Localization、池内 Best-of-K 选择、固定接口的附加上下文探针（具备每种条件下的同词填充对照）与同一存储库的严格负样本，以及通用封装器的 oracle 检查。Oracle Localization 改善了三种系统的表现，但即便如此，Oracle 的成功率仍低于 50%。在抽样得到的 10 个补丁池中，额外的候选多样性仍然有助于性能提升，但该收益空间很快趋于饱和。在两个固定接口下，大多数信息量更高的附加上下文条件仍优于其对应的对照。通用封装器检查揭示了不同系统的响应差异：在相同封装器下，KGCompass 和 ExpeRepair 的增益仍然显著，而 Agentless 对构建器选择更为敏感。提示级融合仍保留较大残余前沿：最佳固定探针仅在三系统原生 Solved@10 联合结果之外新增 6 个已解决实例。总体而言，更强的定位能力、受限的搜索空间、证据质量以及接口设计共同决定了存储库级程序修复的结果。

## Abstract
Repository-level automated program repair (APR) increasingly treats stronger localization as the main path to better repair. We ask a more targeted question: once localization is strengthened, which post-localization levers still provide recoverable gains, which are bounded within our protocol, and what residual frontier remains? We study this question on SWE-bench Lite with three representative repository-level RAG-APR paradigms, Agentless, KGCompass, and ExpeRepair. Our protocol combines Oracle Localization, within-pool Best-of-K, fixed-interface added context probes with per-condition same-token filler controls and same-repository hard negatives, and a common-wrapper oracle check. Oracle Localization improves all three systems, but Oracle success still stays below 50%. Extra candidate diversity still helps inside the sampled 10-patch pools, but that headroom saturates quickly. Under the two fixed interfaces, most informative added context conditions still outperform their own matched controls. The common-wrapper check shows different system responses: under a common wrapper, gains remain large for KGCompass and ExpeRepair, while Agentless changes more with builder choice. Prompt-level fusion still leaves a large residual frontier: the best fixed probe adds only 6 solved instances beyond the native three-system Solved@10 union. Overall, stronger localization, bounded search, evidence quality, and interface design all shape repository-level repair outcomes.

## 二、方法论与技术核心

### 1. 整体思路
作者提出一个包含四个阶段的**“后定位控制协议”**，用于分解影响存储库级修复性能的因素：

1. **Oracle Localization（预言定位）**：基于真实故障位置（gold diff）注入准确信息，只暴露文件和行号，不泄露补丁内容。
2. **Best-of-K Sampling（池内最优选择）**：从每实例生成的10个候选补丁中，评估理想化选择带来的性能上限。
3. **Fixed-Interface Added Context（固定接口附加上下文探针）**：  
   在既定修复接口中添加跨系统的上下文块（来自其他APR系统证据）并设置对照：  
   - 同词填充控制（same-token filler）  
   - 同仓库的硬负样本（hard negative）  
4. **Common-Wrapper Oracle Check（统一封装器检查）**：  
   在共享的封装与构建环境中重放Oracle定位，分析系统差异是否源自接口或构建器。

### 2. 系统对比与流程
- 三种代表性 RAG-APR **范式对比**：  
  1. **Agentless**：分阶段工作流 + 压缩仓库视图 + 测试驱动选择。  
  2. **KGCompass**：基于知识图谱的结构化检索与函数级定位。  
  3. **ExpeRepair**：基于双记忆机制的“经验修复”(retrieval + demonstration)。

### 3. 关键技术机制（文字说明）
- **Oracle 注入流程**：解析目标补丁 → 提取预故障文件与行号 → 注入对应系统的候选集合。  
- **Best-of-K 测评**：对10个补丁样本进行贪婪排序，计算 Solved@K（前K个是否有至少一个通过测试）。  
- **上下文探针机制**：在修复提示阶段插入额外证据信息，验证“上下文内容”与“字数长度”对修复效果的差异影响。  
- **统计分析**：基于 Wilson 置信区间、精确 McNemar 检验、Holm 校正等方法确保结果显著性与配对比较公平性。

## 六、主要结论与发现

1. **Oracle Localization 带来显著改进**
   - 三系统成功率从约 28–33% 提升至 39–43%；增强定位显著减少“无补丁提交”与执行失败。
   - 即便如此，Oracle成功率仍低于 50%，表明后续阶段仍是主要瓶颈。

2. **候选多样性（Best-of-K）仍有有限收益**
   - 在每实例10候选池中，额外提升约 9–13 个百分点，但 86% 的增益已在K=5时饱和。

3. **附加上下文探针验证**
   - 有效上下文（如 KGCompass证据）明显优于等长填充及错误上下文；
   - 提示长度非单调，过长 prompt 反而可能削弱稳定性；
   - 简单拼接多个来源（UnionContext）不一定带来更高增益，反而可能稀释性能。

4. **残余前沿分析**
   - 即使集成全部6种上下文探针，仍仅能额外解决14/300实例；
   - 核心残余问题集中在大型仓库（如 sympy、django）；
   - 典型失败模式：错误API调用、缺失导入、修复不完整、测试断言不匹配。

## 七、优点与创新

- **多系统、控制变量式诊断实验**：首次系统性分离定位效应与后定位瓶颈。  
- **严格配对统计与显著性测试**：采用 Wilson、McNemar、Holm 校正确保比较公正。  
- **精心设计的上下文控制组**：利用同词填充与硬负样本，排除“提示长度”伪效应。  
- **残余前沿理念**：从实证角度刻画“仍无法通过提示融合解决”的任务边界，为后续研究提供方向。
