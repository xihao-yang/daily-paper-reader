## 摘要
大型语言模型（LLMs）如今已经能够解决真实世界的 GitHub 问题。然而，现有方法往往对代码上下文进行了过度近似，导致两个相互影响的问题：处理海量输入的成本高昂，以及由于噪声充斥上下文窗口而使模型偏离修复信号、效果低下。现有的压缩技术无法消除这种矛盾：通用压缩器会损害代码的语义完整性，而专用于代码的工具则缺乏对代码结构和任务上下文的意识，无法保留关键的修复要素。为此，我们提出一个包含两个组件的新框架。首先是 Oracle-guided Code Distillation（OCD），一种上下文蒸馏算法，结合了遗传搜索与增量调试，系统地将代码上下文缩减至其最小的充分子序列——仅保留实现成功修复所需的要素。我们使用这种精炼数据对 SWEzze 进行微调，这是一种轻量模型，能够在推理时学习压缩代码上下文，过滤噪声并避免分心，同时保留修复要素。在 SWE-bench Verified 上对三个前沿 LLM 进行评估时，SWEzze 能在不同模型间保持约 6 倍的稳定压缩率，相较于未压缩设置减少 51.8%-71.3% 的总体 token 成本，使问题解决率提升 5.0%-9.2%，并在效果、压缩比与延迟方面较当前最先进的上下文压缩基线实现最佳整体平衡。

## Abstract
Large Language Models (LLMs) are now capable of resolving real-world GitHub issues. However, current approaches overapproximate the code context and suffer from two compounding problems: the prohibitive cost of processing massive inputs, and low effectiveness as noise floods the context window and distracts the model from the bug-fixing signal. Existing compression techniques fail to resolve this tension: generic compressors compromise the semantic integrity of code, while code-specific tools lack awareness of code structure and task context to preserve essential patch ingredients. To address this, we propose a novel framework consisting of two components. First, Oracle-guided Code Distillation (OCD), a context distillation algorithm that combines genetic search and delta debugging to systematically reduce code contexts to their minimal sufficient subsequence - retaining only the ingredients required for a successful fix. We use this distilled data to fine-tune SWEzze, a lightweight model that learns to compress code context at inference time, filtering noise and combating distraction while preserving fix ingredients. Evaluated on SWE-bench Verified across three frontier LLMs, SWEzze maintains a stable compression rate of about 6 times across models, reduces the total token budget by 51.8%-71.3% relative to the uncompressed setting, improves issue resolution rates by 5.0%-9.2%, and delivers the best overall balance among effectiveness, compression ratio, and latency compared with state-of-the-art context compression baselines.

## 一、研究背景与核心问题

- **研究动机：**  
  当前大型语言模型（LLM）已经能够自动解决真实世界的 GitHub 问题（如 SWE-Agent、Agentless 等框架），但在实际应用中仍面临关键瓶颈：  
  1. **上下文过度近似（Overapproximation）**——代码上下文常被大量无关片段包围。  
  2. **高计算成本**——LLM 推理开销与上下文长度近似线性增长。  
  3. **性能退化**——噪声淹没修复信号，导致生成补丁偏离真正的 bug 修复逻辑。  

- **核心问题：**  
  现有压缩方案在保留“修复所需语义成分”（fix ingredients）方面不足：  
  - 通用压缩器会破坏代码的语义和结构完整性；  
  - 代码特定方法缺少任务感知（未理解哪部分代码对修复真正有用）。  

- **目标：**  
  构建一个既能**高效压缩上下文**，又能**保留实际修复所需片段**的压缩框架，从而提高 LLM 的自动修复成功率并降低推理成本。

## 二、方法论与技术方案

### （1）总体框架

论文提出一个两阶段的上下文压缩框架，包括：
1. **Oracle-guided Code Distillation（OCD）** —— 生成训练信号，用于提炼最小修复上下文；
2. **SWEzze 模型** —— 在推理时执行自动压缩，学习去除噪声而保留“补丁成分”。

### （2）Oracle-guided Code Distillation (OCD)

- **目标定义：**  
  将初始上下文 \( C_{init} \) 精简为最小充分子集 \( \hat{C} \subset C_{init} \)，满足：
  > 若去掉任意一个元素 \( e \in \hat{C} \)，则 LLM 无法生成通过验证测试的补丁。  

- **关键思想：**
  - 通过一个**“执行预言机”**判断上下文是否能生成成功修复；
  - 使用**遗传算法 (Genetic Algorithm, GA)** + **分层 Delta 调试 (Hierarchical Delta Debugging, HDD)** 搜索最小充分上下文；
  - 每轮搜索调用修复模型与单元测试，从而保证每个残留片段都“函数上必要”。  

- **搜索策略：**
  - **阶段一：GA 搜索足够上下文**
    - 个体表示为上下文片段包含向量（二进制基因）；
    - 适应度由片段优先级 Φ(𝑢) 与是否能修复成功共同决定；
  - **阶段二：HDD 层次化最小化**
    - 分为文件、函数、语句块三级；
    - 使用 `ddmin` 算法，优先删除低优先级单位，最终获得 1-最小充分上下文。

- **优先级评分 Φ(𝑢)：**  
  综合三个信号：
  1. **Patch Overlap**：该片段是否出现在人工修复修改的文件中；
  2. **Test Coverage**：单元测试覆盖该片段的比例；
  3. **Symbol Overlap**：代码符号与补丁中标识符的匹配度。  
  > Φ(𝑢) = 𝑤ₚ·I(Fpatch) + 𝑤𝑐·log(1+|Lcov|) + 𝑤ₛ·SymScore

- **输出数据：**  
  OCD 生成映射样本 (I, F, Cinit, ˆC)，作为后续压缩模型 SWEzze 的训练语料。

### （3）SWEzze 模型

- **模型架构：**
  - 基于 **Qwen3-Reranker-0.6B** 的交叉编码器结构；
  - 使用 **LoRA 参数高效微调**（rank=16, α=32）；
  - 输入：(问题描述 + 故障定位, 代码段) → 输出：保留概率分数 \( f(q, s; \theta) \in [0,1] \)。

- **训练策略：**
  - 类别不平衡：正样本（应保留）仅占 8.4%；采用类权重 + 语义角色加权（如 schema、definition、utility 等）；
  - 训练 3 个 epoch，使用 AdamW，学习率 2e−4，采用混合精度与梯度检查点以适配消费级 GPU。

- **推理阶段：**
  - 将代码解析为分段结构体；
  - 对每段打分并基于“预算感知贪心选择”组装压缩上下文；
  - 平均压缩率控制在约 6×。

## 三、实验设计

### （1）数据集与测试场景

- **训练集：**  
  来自 **SWE-Smith** 数据集（128 个 GitHub 仓库、约 50k 问题），从中采样 10k 问题用于 distillation。  
  OCD 过程生成了 3,157 个成功获得最小充分上下文的实例。

- **测试集 / Benchmark：**  
  **SWE-bench Verified**（500 个实际 GitHub Issue，12 个 Python 项目），为行业标准基准；包含问题描述、源代码快照、真实补丁及测试。  

### （2）对比方法

- **长文压缩 / 代码压缩基线：**

  - **SWE-Pruner**（轻量 reranker，基于 token 相关度）

- **被评估 LLM：**

## 六、主要结论与发现

- **压缩效果：**  
  SWEzze 在三种 LLM 上维持稳定 **≈6× 压缩率**。
- **效率提升：**  
  相较未压缩设置，**总 token 成本降低 51.8%–71.3%**，推理时间显著减少。
- **性能提升：**  
  问题修复成功率提高 **5.0%–9.2%**；  
  覆盖已解决实例总集合的 **93.8%–99.2%**。
- **延迟平衡：**  
  比 LongCodeZip 平均快 4×，虽较 SWE-Pruner 稍慢，但结果更稳定。  
- **错误分析：**  
  压缩失败主要源于：
  1. 残留干扰性上下文（误导模型）；
  2. 语义区分度不足（相似函数混淆）；
  3. 结构被破坏（过度剪枝）。
