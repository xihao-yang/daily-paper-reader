## 摘要
由于智能合约在部署后代码具有不可变性，其漏洞可能导致重大经济损失。现有工具虽然能够检测漏洞，但无法有效修复。本文提出了 SCPatcher 框架，将检索增强生成与知识图谱相结合，实现智能合约的自动修复。我们从 5,000 个经过验证的以太坊合约中构建知识图谱，提取函数级别的关系以形成语义网络。该图谱作为外部知识库，增强了大语言模型的推理能力，并实现精确的漏洞修补。我们提出了两阶段修复策略：首先进行基于知识引导的初步修复，然后利用 Chain-of-Thought 推理处理复杂漏洞。在包含不同类型漏洞的合约集上评估表明，SCPatcher 实现了 81.5% 的总体修复率和 91.0% 的编译通过率，显著优于现有方法。

## Abstract
Smart contract vulnerabilities can cause substantial financial losses due to the immutability of code after deployment. While existing tools detect vulnerabilities, they cannot effectively repair them. In this paper, we propose SCPatcher, a framework that combines retrieval-augmented generation with a knowledge graph for automated smart contract repair. We construct a knowledge graph from 5,000 verified Ethereum contracts, extracting function-level relationships to build a semantic network. This graph serves as an external knowledge base that enhances Large Language Model reasoning and enables precise vulnerability patching. We introduce a two-stage repair strategy, initial knowledge-guided repair followed by Chain-of-Thought reasoning for complex vulnerabilities. Evaluated on a diverse set of vulnerable contracts, SCPatcher achieves 81.5\% overall repair rate and 91.0\% compilation pass rate, substantially outperforming existing methods.

## 二、方法论：SCPatcher 框架

### 1. 核心思想

SCPatcher 将 **检索增强生成（Retrieval-Augmented Generation, RAG）** 与 **知识图谱（Knowledge Graph, KG）** 相结合，形成一个外部知识驱动的代码修复系统。  
修复过程分为两个阶段：

1. **知识引导修复阶段**：通过知识图谱检索相似代码模式，为 LLM（Deepseek-V3）提供具信任度的上下文；
2. **链式思维推理（Chain-of-Thought, CoT）阶段**：对初次失败的补丁进行逐步推理式修复，提高复杂漏洞处理能力。

### 2. 体系架构与主要技术模块

#### （1）知识图谱构建

- 来源：Etherscan 上 5,000 个已验证以太坊合约；
- 工具：使用 **Slither** 静态分析工具提取实体-关系三元组（如函数调用、变量依赖等）；
- 存储与向量化：
  - 结构化存储：NetworkX → property graph；
  - 向量嵌入：通过 **CodeBERT** 将函数节点转换为向量，用于高效检索与相似性计算。
- 增强信息：包括代码克隆关系、控制流依赖与使用模式。

#### （2）检索与多阶段重排序（Algorithm 1）

- 初始阶段：以目标函数向量为中心，采用 **k-NN（Top-50）** 相似搜索；
- 三阶段排序策略：
  1. **语法过滤**：仅保留函数签名兼容的候选；
  2. **信任评分（Trust Scoring）**：
     \[
     S_{final}(d,q) = S_{sem}(v_d, v_q) / \ln( f_{freq} + \epsilon )
     \]
     其中 \(S_{sem}\) 为语义距离，\(f_{freq}\) 为全局使用频率（GUF），提升常用安全模式权重；
  3. **多样性去重**：按语义克隆组（CloneID）挑选前 K 个样本，防止冗余。
  
- 输出：供 LLM 修复时使用的高置信参考。

#### （3）补丁生成与验证流程

- **Deepseek-V3 模型**：执行补丁生成；
- **双阶段机制**：
  - 阶段一：基于检索知识的初修（Standard Generation）；
  - 阶段二：若验证失败，则采用 CoT 推理生成更精细补丁；
- **验证工具**：Slither 自动检查合约的漏洞消解及语法正确性；
- **提示模板（Prompt）**：包含信任等级、引用频率、版本保持与依赖规则（如 SafeMath 处理）。

## 八、不足与局限

- **静态分析依赖性强**：过度依赖 Slither 检查，可能导致“检测器过拟合”；
- **行为正确性未验证**：仅做静态修复验证，未评估动态行为或业务逻辑一致性；
- **跨合约与复杂逻辑漏洞**：功能级依赖较弱，难以覆盖深层交互漏洞；
- **数据域偏倚**：知识图谱基于公开验证合约，可能不代表真实企业级代码；
- **推理成本**：双阶段生成策略提高推理时间与资源消耗。
