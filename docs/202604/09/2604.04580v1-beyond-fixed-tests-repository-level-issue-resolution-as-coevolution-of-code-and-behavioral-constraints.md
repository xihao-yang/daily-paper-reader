## 摘要
软件工程师在解决仓库级问题时，并不将现有测试视为不可更改的正确性预言机。相反，他们在新的修改暴露出缺失的假设或被误解的失败条件时，会迭代地改进代码以及表征期望行为的测试。相比之下，大多数基于大语言模型（LLM）的修复系统采用线性管道，其中测试或其他验证信号主要作为事后过滤器，将行为约束视为修复过程中固定不变的内容。这种表述将修复简化为在静态且可能不一致的约束下优化代码，导致搜索空间约束不足以及脆弱或过拟合的修复结果。我们认为，仓库级问题的解决本质上不是在固定测试下的优化，而是在不断演化的行为约束上进行搜索。为实现这一观点，我们提出了一个名为 Agent-CoEvo 的协同进化多智能体框架，在其中候选代码补丁与测试补丁被联合探索并迭代优化。我们的框架不把测试当作不可更改的预言机，而是将其建模为动态约束，既指导修复过程，又在修复过程中被修订。通过相互评估与语义重组，代码与测试候选逐步收敛到与问题描述一致的行为空间。我们在 SWE-bench Lite 和 SWT-bench Lite 上的评估表明，Agent-CoEvo 在修复成功率和测试重现质量方面均显著优于当前最先进的基于智能体和非智能体的基线系统。我们的研究结果表明，使修复智能体能够在搜索过程中修订行为约束对于可靠问题解决至关重要，这预示着从单纯的代码优化向实现与规范的协同进化转变。

## Abstract
Software engineers resolving repository-level issues do not treat existing tests as immutable correctness oracles. Instead, they iteratively refine both code and the tests used to characterize intended behavior, as new modifications expose missing assumptions or misinterpreted failure conditions. In contrast, most existing large language model (LLM)-based repair systems adopt a linear pipeline in which tests or other validation signals act mostly as post-hoc filters, treating behavioral constraints as fixed during repair. This formulation reduces repair to optimizing code under static and potentially misaligned constraints, leading to under-constrained search and brittle or overfitted fixes. We argue that repository-level issue resolution is fundamentally not optimization under fixed tests, but search over evolving behavioral constraints. To operationalize this view, we propose Agent-CoEvo, a coevolutionary multi-agent framework in which candidate code patches and test patches are jointly explored and iteratively refined. Rather than treating tests as immutable oracles, our framework models them as dynamic constraints that both guide and are revised by the repair process. Through mutual evaluation and semantic recombination, code and test candidates progressively narrow the space of behavior consistent with the issue description. Evaluated on SWE-bench Lite and SWT-bench Lite, Agent-CoEvo consistently outperforms state-of-the-art agent-based and agentless baselines in both repair success and test reproduction quality. Our findings suggest that enabling repair agents to revise behavioral constraints during search is critical for reliable issue resolution, pointing toward a shift from code-only optimization to coevolution of implementation and specification.

## 一、核心问题与研究动机

- **研究背景**：  
  当前基于大语言模型（LLM）的自动化代码修复系统，普遍假设现有测试是**静态且正确的行为约束**，并据此生成候选补丁。这种方法将修复任务简化为“在固定约束下优化代码”，实际操作中常导致：
  - 搜索空间受限；
  - 修复结果容易过拟合于不完整或错误的测试；
  - 无法像真实开发者那样动态调整测试以匹配真实行为。
- **核心问题**：  
  真实软件开发的“问题解决”过程不是静态的优化，而是**代码实现与行为约束的共同演化**。开发者往往在修复代码的同时修改测试、修订约束。因此，静态测试假设不能准确反映真实的软件维护模式。
- **研究目标**：  
  作者提出应将仓库级问题解决视为**在演化的行为约束下的联合搜索问题**，并构建一种让代码和测试共同进化的多智能体系统，称为 **Agent-CoEvo**。

## 二、方法论：Agent-CoEvo 协同进化框架

### 1. 核心思想
- 将修复任务建模为**代码补丁与测试补丁的协同进化过程**；
- 测试不再作为固定“预言机”，而是作为**动态约束（Dynamic Constraints）**；
- 通过互相评估与语义重组，使代码与测试在搜索的同时逐步趋向一致的行为描述。

### 2. 系统结构
Agent-CoEvo 由三个核心智能体组成：
1. **LocationAgent**：从自然语言问题描述中定位出相关文件与代码行；
2. **CodeAgent**：维护一组候选代码补丁；
3. **TestAgent**：维护一组候选测试补丁。

### 3. 协同进化过程（Algorithm 1）
系统通过若干迭代（通常为 5 次）执行以下阶段：
1. **初始化**：  
   - 由 LLM 生成初始代码与测试补丁群体；
   - 测试群体仅保留能在原始仓库中“失败”的样本；
2. **交叉评估（Cross Evaluation）**：  
   - 构建执行矩阵 `M[c, t]`，表示代码补丁 `c` 对测试补丁 `t` 是否通过；
   - 计算适应度：
     - 代码补丁适应度：  
       \[
       F_{\text{code}}[c] = |S_{\text{consensus}}| \times |S_{\text{pass}}|
       \]
       兼顾共识行为与测试覆盖；
     - 测试补丁适应度：  
       \[
       F_{\text{test}}[t] = \sum_{c} M[c, t] \cdot F_{\text{code}}[c]
       \]
       强调测试区分力；
3. **选择与保留**：采用精英保留策略，保留每轮中最佳个体；
4. **语义交叉（Semantic Crossover）**：  
   不采用简单拼接，而由 LLM 解析父代补丁的逻辑语义，综合两者优势生成新补丁；
5. **循环迭代直至收敛或达到最大轮次**。

### 4. 工程实现要点
- 在隔离的 **Docker 容器** 内运行各候选补丁，以防污染代码库；
- 使用 ASTSearch 和 RegexSearch 工具辅助定位上下文；
- 每次运行均自动恢复仓库初始状态，保证实验独立性。

## 三、实验设计

### 1. 数据集与 Benchmark
- **SWE-bench Lite**：  
  含 300 个真实 GitHub 仓库问题，用于评估修复正确性（代码补丁能否通过隐藏测试集）。
- **SWT-bench Lite**：  
  含 276 个实例，用于评估生成测试的有效性（测试能否复现问题并验证开发者的黄金修复）。
  - 额外度量指标：ΔC（测试覆盖提升量），反映测试是否覆盖修复区域。

### 2. 对比基线
分为三类共 10+ 方法：
- **代码中心修复模型**：Agentless、SpecRover、Moatless Tools、KGCompass、DARS；
- **测试生成模型**：AssertFlip、SWE-Agent+；
- **通用软件智能体**：SWE-Agent、AutoCodeRover、OpenHands。

所有基线均使用相当层级的 LLM（如 GPT-4o、Claude 3.5、DeepSeek-V3）以确保公平。

### 3. 模型设置
- 使用 **DeepSeek-V3-0324** 作为主要后端模型；
- 种群规模：10；
- 最大演化迭代次数：5；
- 温度：Localization 0.2、Patch Generation 0.5；
- 平均运行成本（通过 API 计费）约 1.11 美元/issue。
