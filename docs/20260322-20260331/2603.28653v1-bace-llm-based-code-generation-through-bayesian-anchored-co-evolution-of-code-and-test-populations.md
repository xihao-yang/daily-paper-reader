## 摘要
大型语言模型（LLMs）在代码生成方面展示了令人印象深刻的能力。尽管交互式反馈循环能够提升性能，但编写有效的测试并非易事。早期的多智能体框架（例如 AgentCoder）自动化了这一过程，但其将生成的测试视为绝对的真值。这种方法较为脆弱：错误的代码经常能通过有缺陷或过于简单的测试，而有效的解决方案反而会被破坏以满足不正确的断言。为解决这一局限，后续方法在很大程度上放弃了测试生成，转而依赖基于示例的计划与推理。然而，我们认为，如果将生成的测试建模为通过贝叶斯更新引导的噪声传感器，它们仍然是有价值的信号。基于此，我们提出了 BACE（Bayesian Anchored Co-Evolution，贝叶斯锚定协同进化），一个将合成过程重新表述为贝叶斯协同进化过程的框架，其中代码和测试种群在基于噪声交互证据的信念分布的相互更新下共同进化。通过在最小的公开示例上锚定搜索，BACE 防止了自验证循环中常见的协同进化漂移。在 LiveCodeBench v6（2025 年 3 月之后）的广泛评估中，BACE 在专有模型和开放权重的小型语言模型上均取得了更优的性能。

## Abstract
Large Language Models (LLMs) have demonstrated impressive capabilities in code generation. While an interactive feedback loop can improve performance, writing effective tests is a non-trivial task. Early multi-agent frameworks, such as AgentCoder, automated this process but relied on generated tests as absolute ground truth. This approach is fragile: incorrect code frequently passes faulty or trivial tests, while valid solutions are often degraded to satisfy incorrect assertions. Addressing this limitation, newer methods have largely abandoned test generation in favor of planning and reasoning based on examples. We argue, however, that generated tests remain a valuable signal if we model them as noisy sensors guided by bayesian updates. To this end, we introduce BACE (Bayesian Anchored Co-Evolution), a framework that reformulates synthesis as a Bayesian co-evolutionary process where code and test populations are evolved, guided by belief distributions that are reciprocally updated based on noisy interaction evidence. By anchoring this search on minimal public examples, BACE prevents the co-evolutionary drift typical of self-validating loops. Extensive evaluations on LiveCodeBench v6 (post-March 2025) reveal that BACE achieves superior performance across both proprietary models and open-weight small language models.

## 一、核心问题与研究动机

- **研究背景：**
  - 大型语言模型（LLMs）在从自然语言生成代码（NL-to-Code）任务上已取得显著进展，但生成代码往往存在潜在逻辑错误。
  - 以往方法（如 *AgentCoder*）尝试通过“程序员–测试者”闭环优化体系来改进逻辑正确性，即自动生成测试并基于执行结果修复代码。
  - 然而这些方法将**自动生成的测试视为绝对真值（ground truth）**，导致两个典型问题：
    1. **假阳性（False Positive）**：错误代码误通过不良测试；
    2. **假阴性（False Negative）**：正确解被错误断言拒绝。
  - 因此，生成测试信息虽然有用，却因其不确定性而被近期方法（如 *MapCoder*、*CodeSIM*）所放弃。

- **核心研究问题：**
  > 如何在测试集自身不可靠的条件下，仍能通过代码—测试交互过程收敛到正确的程序解？

- **研究目标：**
  - 提出一种稳健机制，使得生成的测试仍能作为有效信号指导代码优化；
  - 同时防止“自我验证”闭环中的进化漂移（Co-evolutionary drift）。

## 二、方法论与技术框架

### 1. 核心思想：贝叶斯锚定协同进化（BACE）

- 通过**贝叶斯概率建模**，将代码与测试视作两个**共演化的种群（co-evolving populations）**；
- 不再将测试执行结果视为确定信号，而是建模为**噪声传感器（noisy sensors）**；
- 代码与测试的“信念”分布会随着互相作用的执行证据不断更新。

### 2. 概念体系

- **代码种群 \(C = \{c_1,...,c_N\}\)** 与 **测试种群 \(T = \{t_1,...,t_M\}\)**；
- 每个个体都有一个隐藏变量：
  - \(X_i\)：代码正确性；
  - \(Y_j\)：测试有效性；
  - 信念表示为 \(b(c_i)=P(X_i=1)\), \(b(t_j)=P(Y_j=1)\)。

### 3. 噪声传感模型（Noisy Sensor Model）

- 执行结果 \(d_{ij} \in \{0,1\}\)：通过或失败；
- 通过三个噪声参数表示不确定性：
  - **α**：有效代码通过错误测试的概率；
  - **β**：错误代码通过有效测试的概率；
  - **γ**：错误代码通过错误测试的概率。

### 4. 贝叶斯信念更新机制

- 使用**对数几率空间 (log-odds space)** 累积证据，保持数值稳定；
- 更新规则（简述）：
  - 代码信念：综合所有通过/未通过的测试结果；
  - 测试信念：对称地通过高可信代码更新；
- 引入**可信度阈值不等式**，确保低可信度个体不能反向污染信念传播。

### 5. 锚定机制（Anchoring）

- 在每个任务中，利用公开的少量输入输出示例作为锚点 \(T_{anchor}\)；
- 锚点的信念固定为1，执行结果直接影响代码信念更新：
  - **正锚点（pass）**：迅速提升代码信念；
  - **负锚点（fail）**：信念降为0（不可逆惩罚）；
- 通过锚定防止模型陷入自肯定的错误循环。

### 6. 种群进化与多样性维护

- **交替进化（alternating evolution）**：代码与测试轮流进化，防止动态不稳定；
- **精英保留 (elitism)**：
  - 代码侧保留“行为向量”互异的代表；
  - 测试侧移除重复（冗余）传感器；
- **差分测试 (Differential Tests)**：用于发现功能等价类间的差异，促进分化与多样性。

### 7. 算法流程（简述）

1. 初始化代码与测试种群（由LLM生成多样解）。
2. 提取锚点样例作为高可信固定测试。
3. 在若干轮（10代）迭代中交替：
   - 执行所有代码–测试组合；
   - 计算观察矩阵并进行贝叶斯信念更新；
   - 进化相应种群（通过调试、交叉、差异检测等操作）。
4. 输出信念最高的最终代码和对应测试集。
