ZIBIN ZHENG, Sun Yat-sen University, China Translating natural language descriptions into viable code fixes remains a fundamental challenge in software engineering. While the proliferation of agentic large language models (LLMs) has vastly improved automated repository-level debugging, current frameworks hit a ceiling when dealing with sophisticated bugs like implicit type degradations and complex polymorphic control flows. Because these methods rely heavily on static analysis and superficial execution feedback, they lack visibility into intermediate runtime states. Consequently, agents are forced into costly, speculative trial-and-error loops, wasting computational tokens without successfully isolating the root cause. To bridge this gap, we propose DAIRA (Dynamic Analysis-enhanced Issue Resolution Agent), a pioneering automated repair framework that natively embeds dynamic analysis into the agent’s reasoning cycle. Driven by a Test Tracing-Driven methodology, DAIRA utilizes lightweight monitors to extract critical runtime data—such as variable mutations and call stacks—and synthesizes them into structured semantic reports. This mechanism fundamentally shifts the agent’s behavior from blind guesswork to evidence-based, deterministic deduction. When powered by Gemini 3 Flash Preview, DAIRA establishes a new state-of-the-art (SOTA) performance, achieving a 79.4% resolution rate on the SWE-bench Verified dataset. Compared to existing baselines, our framework not only conquers highly complex defects but also cuts overall inference expenses by roughly 10% and decreases input token consumption by approximately 25%. Resolving such issues requires developers to interpret informal problem descriptions, navigate large and tightly coupled codebases, identify fault locations, and construct correct patches under complex execution semantics [ 8 ]. Even for experienced developers, this process is time-consuming and cognitively demanding, often involving iterative debugging and validation [ 19 ]. Consequently, automating issue resolution has long been regarded as a high-impact yet highly challenging goal in automated software engineering [24]. Recent benchmarks such as SWE-bench have underscored the complexity of issue resolution, driving the proliferation of autonomous agent-based repair frameworks [ 2, 4 , 9, 10 , 15 , 20 , 22 , 24 ]. SWE-agent established the standard Agent-Computer Interface (ACI) paradigm [ 24 ]. By abstracting complex shell interactions into a streamlined set of commands—such as browsing, editing, and executing—ACI significantly optimizes the efficiency of LLM-environment interaction, serving as a robust baseline for automated issue resolution. Building upon this, recent works have explored diverse paradigms: planning-based approaches such as SWE-Search [ 2 ] and CodeTree [ 10 ] integrate MCTS to refine reasoning, while experience-driven and multi-agent methods like Live-SWE-agent [ 22 ], SWE-Exp [ 4 ], and SWE-Debate [ 9 ] utilize self-evolution or competitive dynamics to improve decision quality. However, these agents, predominantly relying on static analysis and superficial feedback, fail to achieve the fine-grained observability necessary for complex defects. This limitation is particularly evident in scenarios involving intricate control flows, implicit runtime state, and deeply nested dependencies. A growing body of evidence suggests that these limitations do not primarily stem from insufficient language or reasoning capability of LLMs, but from a fundamental lack of execution-level observability . Existing approaches predominantly rely on static analysis and code retrieval to construct the agent’s working context, selecting code snippets based on lexical or semantic similarity. Without constraints imposed by real execution trajectories, this static perspective often leads to speculative and non-deterministic exploration, where large amounts of irrelevant code crowd the limited context window while the true fault remains elusive. More critically, static code representations are inherently insufficient for modeling many behaviors that arise only at runtime . Real-world defects frequently depend on dynamic control flows, conditional execution paths, polymorphism, and implicit runtime state changes such as type mutation. Recent empirical studies corroborate this limitation. For example, Liu et al. [ 11 ]demonstrate that when evaluated on code reasoning tasks derived from mature real-world projects, LLMs experience substantial performance degradation—specifically, showing a 51.50% drop in input prediction accuracy and a 42.15% drop in output prediction when transitioning from easy to hard problems. These findings indicate that reasoning over static code alone is inadequate for capturing the intermediate execution states required in realistic software engineering scenarios. In contrast, human developers routinely rely on dynamic analysis to diagnose complex software defects. Dynamic analysis encompasses techniques that observe program behavior during execution, such as runtime tracing, variable state inspection, and call stack analysis [ 5]. By exposing concrete execution paths and intermediate states, dynamic analysis provides a causal and state-aware view of program behavior that complements static reasoning. Notably, dynamic analysis has already proven essential for faithfully characterizing real-world code complexity in evaluation settings [ 14 , 26 ], yet it remains largely absent from the decision-making loop of LLM-based issue resolution agents. Motivated by this gap, we argue that effective automated issue resolution fundamentally requires integrating dynamic execution information to transform speculative reasoning into deterministic inference . To this end, we propose DAIRA , a Dynamic Analysis–enhanced 

Issue Resolution Agent , which is the first automated issue resolution framework to deeply integrate dynamic analysis into the agent workflow. DAIRA adopts a Test Tracing–Driven paradigm, treating

execution traces as first-class evidence for agent reasoning. By tracing targeted execution scripts, DAIRA generates structured trace reports that expose concrete function call sequences, runtime states, and variable mutations, providing a panoramic view of program execution. Specifically, DAIRA consists of a lightweight dynamic tracing tool and an agent-oriented work-flow module. The tracing tool captures fine-grained execution information within a sandboxed environment, while the workflow module transforms raw traces into compact, semantically mean-ingful representations that are directly consumable by LLMs. This design enables precise fault localization and significantly reduces redundant retrieval and blind exploration. To evaluate the effectiveness of DAIRA, we conducted extensive experiments on the benchmark SWE-bench Verified. In addition, by providing precise fault localization through dynamic execution traces, DAIRA substantially reduces unnecessary code retrievals and speculative attempts, resulting in improved efficiency and an approximately 25% reduction in token consumption compared to baseline methods. The main contributions of this paper are summarized as follows: 

• A new paradigm for LLM-based issue resolution: We identify the lack of execution-level observability as a fundamental bottleneck and propose the first framework that systematically integrates dynamic analysis into the agent decision loop. 

• Agent-oriented dynamic analysis tooling: We design lightweight tracing and trace-semantic transformation mechanisms that expose causal runtime information while remain-ing compatible with LLM context constraints. 

• Comprehensive empirical validation: Extensive experiments on SWE-bench Verified demonstrate that DAIRA achieves SOTA performance in both repair effectiveness and cost efficiency, highlighting the indispensable role of dynamic execution information in resolving deep logical defects.

These examples collectively highlight a fundamental limitation of static code analysis: defects arising from dynamic control flow or implicit state changes are difficult to detect without observing intermediate execution processes. Consequently, just as human engineers rely on tracing tools, such as IDE-integrated debuggers, to inspect runtime states, equipping agents with mechanisms to visualize intermediate execution states is a critical strategy for augmenting their problem-solving capabilities. 

3 Approach 

Inspired by these practices, we aim to equip agents with a similar capability for dynamic tracing. To this end, we introduce DAIRA, an automated issue resolution framework that encapsulates dynamic analysis as a callable tool. This design empowers the agent to autonomously invoke tracing to reconstruct execution process. In the following part of this section, we detail the overall Test-Trace Driven Workflow, the design of the Dynamic Analysis Tool, and the Trace Log Semantic Analysis.

3.2 Dynamic Analysis Tool 

The Dynamic Analysis Tool module is designed to capture ground-truth execution behaviors during automated debugging. To address compatibility challenges in complex execution environments (e.g., legacy Python versions), DAIRA adopts a lightweight “trigger-and-collect” tracing strategy. The Dynamic Analysis Tool is built upon a customized version of the open-source Hunter engine [ 16 ]—a tracer designed to serialize execution behaviors into formatted logs—and is integrated into the agent’s action space via a standardized CLI. This allows the agent to dynamically configure log granularity via four parameters: Target File, Trace Scope, Target Function, and Trace Depth. Upon invocation, the tool executes the script in an isolated sandbox, capturing heterogeneous data (call stacks, variable states, exceptions) into structured logs for Trace Log Semantic Analysis. Figure 4 illustrates the Test Script generated during the problem-solving process and the Trace Log captured during its execution. 

Native Execution via Automatic Hooks . A critical challenge is format friction, stemming from the necessity to manually inject intrusive logging directives or proprietary hooks into the source code, which often impedes the agent’s ability to generate valid scripts. To mitigate this, we implemented a Native In-Process Execution strategy. Operating as a transparent wrapper, the tool executes agent-synthesized standard Python reproduction scripts directly within the sandbox via the runpy interface. This approach obviates the need for the agent to master specific debugging syntax or inject boilerplate code. Instead, the tool automatically injects low-level hooks leveraging 

sys.settrace into the runtime environment, guided by the specified parameters. This seamless integration minimizes barriers to dynamic observation, ensuring consistency with the agent’s standard workflow. Furthermore, this environment-agnostic design guarantees high portability, enabling the tool to accommodate diverse runtime environments characterized by legacy Python versions or intricate dependencies. 

Spatiotemporal Trace Filtering . To extract core insights and minimize noise, our tool employs strict spatiotemporal filtering. Temporally, we adopt an "on-demand" mechanism where the tracer activates only upon target function invocation, bypassing initialization noise. Spatially, a whitelist filters out standard libraries and irrelevant third-party dependencies. Regarding granularity, we capture only function calls, returns, and exceptions, relying on the LLM’s capacity to deduce intermediate logic directly from these call-return pairs. These designs achieve significant log compression without losing critical information. 

Adaptive Granularity Iteration . Static configurations struggle to adapt to the dynamic com-plexity of defects. To prevent deep call stacks from triggering context explosion, we introduce a dynamic adjustment mechanism. When trace logs exceed the valid context window, an overflow signal triggers iterative correction, prompting the agent to reduce trace depth and rerun. By dy-namically adjusting analysis granularity based on complexity, this mechanism completely avoids situations where excessive logs prevent dynamic analysis, enabling the module to dynamically adapt to different model context windows.

4.1 Experiment Setup 

4.1.1 Benchmark . To evaluate the practical software engineering capabilities of DAIRA, we utilize the SWE-bench Verified [ 8 ]. This dataset is a rigorously curated subset of the original SWE-bench, specifically designed to address the issues of solvability and environmental stability. It consists of 

500 human-validated instances collected from 12 popular open-source Python repositories (e.g., scikit-learn, Flask, Django). It also stands as the most authoritative benchmark for evaluating automated issue resolution systems. 

4.1.2 Baselines . To evaluate the effectiveness of our proposed method, we select several SOTA methods as baselines, representing the most advanced and widely recognized approaches in the field of automated software engineering: 

SWE-agent [24 ]: As a pioneering framework in automated software engineering, SWE-agent introduced the ACI to optimize the interaction between LLMs and the software environment. It utilizes a specialized set of commands to browse repositories, edit files, and execute tests, serving as a standard representative approach for issue resolution. 

Live-SWE-agent [22 ]: A self-evolving framework built upon Mini-SWE-agent. It allows the agent to synthesize custom tools at runtime to adapt to specific issues. Currently, it represents the SOTA performance among open-source agents on SWE-bench Verified Leaderboard. 

Mini-SWE-agent [ 24 ]: A lightweight variant of SWE-agent that balances performance and cost through a concise context window and streamlined reasoning. 

OpenHands [ 20 ]: An open-source platform for software development agents. OpenHands provides a flexible and comprehensive environment that supports various agentic workflows and interacts with the codebase through standard terminal commands and event streams. 

SWE-search [2 ]: An advanced agentic framework that integrates Monte Carlo Tree Search (MCTS) into the issue resolution process. By replacing traditional linear reasoning chains with tree-structured exploration, it empowers the agent to simulate future trajectories, execute backtracking from dead ends, and iteratively refine its repair strategies to avoid local optima. 

4.1.3 Implementation . We implemented our approach based on SWE-agent due to its stability and widespread adoption in the open-source community. In terms of tool usage, we only adopted the basic code search and editing tool ’edit_anthropic’ from its framework. Notably, our method is designed to be framework-agnostic and can be seamlessly adapted to any agent framework equipped with a configured runtime environment. Regarding resource constraints, we adhered to the standard configuration: a maximum limit of 250 steps and a budget cap of $1.00 per instance.

5 Related work 5.1 Static Analysis-based Approaches For Issue Resolution 

Static analysis-based approaches [ 12 , 17 , 21 , 23 , 25 ] emerge as a vital strategy to mitigate the constraints of LLM context windows in repository-scale issue resolution. Because ingesting an entire codebase is impractical, accurate patch generation requires a mechanism to precisely locate and retrieve the critical context—such as variable definitions and function dependencies—needed to understand the code logic. To achieve this, these methods leverage deterministic parsing for structured software representation. Agentless [ 21 ] proves that static retrieval can enable high-performance patching by decoupling fault localization from repair. To refine retrieval precision, AutoCodeRover [ 25 ] and SpecRover [ 17 ] employ ASTs and intent analysis, respectively, while KGCompass [ 23 ] utilizes knowledge graphs to enforce global dependency constraints. However, relying solely on static snapshots misses intermediate execution details—a gap our work bridges by integrating dynamic analysis for concrete execution evidence. 

5.2 Agentic Approaches For Issue Resolution 

Agentic approaches [ 2, 4 , 9 , 10 , 15 , 20 , 22 , 24 ] represent a paradigm shift towards closed-loop architectures capable of navigating the dynamic state space of software repositories. Unlike static workflows, these agents possess “agency”—the capacity to perceive environmental feedback, main-tain internal state, and dynamically adjust actions through standardized interfaces. Pioneering this domain, Yang et al. [ 24 ] introduce SWE-agent , utilizing a specialized ACI for autonomous issue resolution. To mitigate greedy reasoning errors, search-based approaches like SWE-search [ 2]and CodeTree [ 10 ] integrate tree-search algorithms (e.g., MCTS) to explore optimal repair tra-jectories. Addressing memory limitations, experience-driven methods such as SWE-Exp [4 ] and 

EXPEREPAIR [15 ] leverage historical repair data to guide decision-making. Additionally, Live-SWE-agent [ 22 ] enables self-evolution by updating strategies based on real-time feedback. However, these agents primarily rely on superficial input-output feedback, lacking the intermediate runtime observability required to diagnose the internal mechanisms of complex defects. 

6 Threats to validity 

Internal Validity . To mitigate data leakage, we utilize the rigorously curated SWE-bench Verified. Furthermore, by benchmarking DAIRA against SWE-agent using identical backbones (e.g., DeepSeek V3.2), we isolate the source of improvement. The significant performance gap confirms that gains stem from our dynamic analysis framework rather than the model’s memorized priors. 

External Validity . Our evaluation is currently confined to the Python ecosystem due to the reliance on specific hooks (e.g., sys.settrace ). However, the core Test Tracing-Driven Workflow is language-agnostic. By substituting the instrumentation engine, our work can be readily adapted to other runtimes (e.g., JVM or CLR) in future work. 

7 Conclusion 

In this paper, we introduced DAIRA, an agentic framework designed to overcome the “tunnel vision” of static analysis by integrating dynamic observability into the issue resolution process. DAIRA provides agents with high-fidelity execution reports, enabling them to pinpoint root causes through deterministic reasoning rather than speculative search. Extensive evaluations on SWE-bench Verified show that DAIRA achieves a SOTA resolution rate and reduces repair costs across foundation models. Notably, DAIRA exhibits robustness in handling high-difficulty tasks, outper-forming baselines by a large margin in complex scenarios. These results confirm that equipping agents with dynamic observability is a critical step toward automated software maintenance.
