## 摘要
代码代理能够自主解决 GitHub 问题，但当它们失败时，现有的评估方法无法揭示失败的原因或位置。诸如 Pass@1 之类的指标将整个执行过程简化为单一的二元结果，使得难以识别代理出错的阶段与原因。为克服这一限制，我们提出了 TRAJEVAL，一种将代理轨迹分解为三个可解释阶段的诊断框架：search（文件定位）、read（函数理解）和 edit（修改定位）。针对每个阶段，我们通过与参考补丁进行比较来计算精确率和召回率。我们分析了来自三种代理架构和七个模型的 16,758 条轨迹，发现了普遍存在的低效现象（所有代理平均查看的函数数量约为必要数量的 22 倍），但各自的失败模式不同：GPT‑5 能正确定位相关代码，但在确定修改目标时出错；而 Qwen‑32B 则完全无法正确发现目标文件。我们验证了这些诊断具有预测能力，可在 0.87–2.1% 的 MAE 范围内进行模型级 Pass@1 预测；同时具有可操作性：基于轨迹信号的实时反馈可使两个当前最先进的模型性能提升 2.2–4.6 个百分点，并减少 20–31% 的成本。这些结果表明，我们的框架不仅能更细粒度地分析代理行为，还能将诊断信号转化为实际的性能提升。更广泛地说，TRAJEVAL 将代理评估从基于结果的基准测试转变为基于机制驱动的成功与失败诊断。

## Abstract
Code agents can autonomously resolve GitHub issues, yet when they fail, current evaluation provides no visibility into where or why. Metrics such as Pass@1 collapse an entire execution into a single binary outcome, making it difficult to identify where and why the agent went wrong. To address this limitation, we introduce TRAJEVAL, a diagnostic framework that decomposes agent trajectories into three interpretable stages: search (file localization), read (function comprehension), and edit (modification targeting). For each stage, we compute precision and recall by comparing against reference patches. Analyzing 16,758 trajectories across three agent architectures and seven models, we find universal inefficiencies (all agents examine approximately 22x more functions than necessary) yet distinct failure modes: GPT-5 locates relevant code but targets edits incorrectly, while Qwen-32B fails at file discovery entirely. We validate that these diagnostics are predictive, achieving model-level Pass@1 prediction within 0.87-2.1% MAE, and actionable: real-time feedback based on trajectory signals improves two state-of-the-art models by 2.2-4.6 percentage points while reducing costs by 20-31%. These results demonstrate that our framework not only provides a more fine-grained analysis of agent behavior, but also translates diagnostic signals into tangible performance gains. More broadly, TRAJEVAL transforms agent evaluation beyond outcome-based benchmarking toward mechanism-driven diagnosis of agent success and failure.

## 2. 方法论：核心思想与关键技术细节
- **核心思想**
  将智能体的执行轨迹分解为三个可解释阶段：
  1. **Search（搜索）**：文件级定位。
  2. **Read（读取）**：函数级理解。
  3. **Edit（编辑）**：修改定位。
  
  每阶段通过与参考补丁的黄金上下文（golden context）对比计算 **精确率（Precision）** 与 **召回率（Recall）**：
  - **Precision**：衡量探索效率（查看目标元素的比例）。
  - **Recall**：衡量探索覆盖率（找到所有必要目标的比例）。
  
- **黄金上下文层级**
  - **Tier 0（核心）**：仅包含参考补丁中直接修改的文件和函数。
  - **Tier 1（扩展）**：Tier 0 基础上增加一跳依赖（导入模块、测试文件、父类等）。
  
- **公式描述**  
  对于文件集合 \( F \) 和函数集合 \( H \)：
  - Search Precision: \( P_s = \frac{|F^* \cap F_T|}{|F_T|} \)  
    Search Recall: \( R_s = \frac{|F^* \cap F_T|}{|F^*|} \)
  - Read Precision: \( P_r = \frac{|H^* \cap H_T|}{|H_T|} \)  
    Read Recall: \( R_r = \frac{|H^* \cap H_T|}{|H^*|}
  - Edit Precision: \( P_e = \frac{|H^* \cap H_{\hat{P}}|}{|H_{\hat{P}}|} \)  
    Edit Recall: \( R_e = \frac{|H^* \cap H_{\hat{P}}|}{|H^*|} \)

- **算法流程**
  1. 从参考补丁解析黄金文件和黄金函数。
  2. 遍历智能体轨迹，根据工具操作提取查看的文件、读取的函数、编辑的函数。
  3. 按公式计算六个指标（Ps, Rs, Pr, Rr, Pe, Re）。
  4. 将指标输入轻量逻辑回归模型预测 Pass@1 成功率。
  5. 可在执行过程中实时检测是否进入黄金上下文，并给智能体提供确认反馈。

## 6. 主要结论与发现
- **精度预测能力**：基于轨迹指标的逻辑回归预测 Pass@1，MAE 在 0.87–2.1% 之间，排名相关系数最高达 1.00。
- **普遍行为模式**：
  - 所有智能体精确率低（过度探索：函数查看数是必要数的 ~22 倍）。
  - 成功与失败的关键在于召回率**而非精确率**。
- **不同模型的失败模式**：
  - GPT-5：成功发现代码文件但编辑定位错误（低 Edit Recall）。
  - Qwen3-32B：文件发现失败（低 Search Recall）。
- **干预有效性**：
  - 实时黄金上下文确认反馈可使 GPT‑5 提升 4.6%，Qwen‑480B 提升 2.2%，同时降低成本 20–31%。
- **跨分布泛化**：
  - 在不同代码库、语言中排名保持稳定相关性（ρ ≥ 0.886）。
