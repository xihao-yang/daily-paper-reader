# MobileDev-Bench: A Comprehensive Benchmark for Evaluating Language Models on Mobile Application Development 

Moshood A. Fakorede 1, Krishna Upadhyay 1, A.B. Siddique 2, Umar Farooq 11Louisiana State University, 2University of Kentucky

Abstract 

Large language models (LLMs) have shown strong performance on automated software en-gineering tasks, yet existing benchmarks focus primarily on general-purpose libraries or web applications, leaving mobile application devel-opment largely unexplored despite its strict plat-form constraints, framework-driven lifecycles, and complex platform API interactions. We in-troduce MobileDev-Bench, a benchmark com-prising 384 real-world issue-resolution tasks collected from 18 production mobile applica-tions spanning Android Native (Java/Kotlin), React Native (TypeScript), and Flutter (Dart). Each task pairs an authentic developer-reported issue with executable test patches, enabling fully automated validation of model-generated fixes within mobile build environments. The benchmark exhibits substantial patch complex-ity: fixes modify 12.5 files and 324.9 lines on average, and 35.7% of instances require coor-dinated changes across multiple artifact types, such as source and manifest files. Evaluation of four state-of-the-art code-capable LLMs, GPT-5.2, Claude Sonnet 4.5, Gemini Flash 2.5, and Qwen3-Coder, yields low end-to-end resolution rates of 3.39%–5.21%, revealing significant performance gaps compared to prior bench-marks. Further analysis reveals systematic fail-ure modes, with fault localization across multi-file and multi-artifact changes emerging as the primary bottleneck. We release MobileDev-Bench with task instances, an evaluation har-ness, and containerized environments to sup-port reproducible research on AI-assisted mo-bile application development. 

1 Introduction 

Large language models (LLMs) such as GPT-4 (OpenAI, 2023), Claude (Anthropic, 2023), and Code Llama (Roziere et al., 2023) have demon-strated remarkable progress in automated soft-ware engineering, particularly in repository-level issue resolution. Benchmarks such as SWE-bench (Jimenez et al., 2024a) establish a realistic evaluation paradigm by requiring models to resolve real-world GitHub issues through patch generation, where correctness is validated by executing the rel-evant test suite. Subsequent extensions (Yang et al., 2024; Zhang et al., 2024; Kabir, 2025; Mhatre et al., 2025; Zan et al., 2024a, 2025) broaden coverage across programming languages and evaluation set-tings, revealing persistent performance gaps even for frontier code-capable LLMs. Despite these advances, existing benchmarks pri-marily target issue resolution in general-purpose libraries or web applications. However, mobile application (app) development remains largely un-derexplored in benchmark design, despite intro-ducing distinct engineering constraints. Mobile apps are built within platform-defined architec-tures with lifecycle-managed execution (Android, 2020). They frequently employ declarative and event-driven programming (Flutter, 2026) and rely on asynchronous state management across user in-terface and service components. Compilation and testing are mediated by framework-specific build systems (Android, 2025), and correctness often depends on coordinated changes across heteroge-neous artifacts, including source files, manifests, re-source definitions, and configuration metadata (An-droid, 2024; Flutter, 2025). These characteristics make mobile app repositories structurally differ-ent from the library-centric settings that dominate existing issue-resolution benchmarks. We introduce MobileDev-Bench , a benchmark of 384 manually verified issue-resolution tasks col-lected from 18 production mobile app repositories spanning Android Native (Java/Kotlin), React Na-tive (TypeScript), and Flutter (Dart). Each instance pairs an authentic developer-reported issue with its merged pull request and executable test patches, enabling automated validation of model-generated fixes. Across models, fault localization emerges as the primary bottleneck, with file-level recall ranging from 13.8% to 19.5%. This work makes the following contributions: • We introduce MobileDev-Bench, a benchmark of 384 manually verified mobile app issue-resolution tasks drawn from 18 production repositories across Android Native, React Na-tive, and Flutter. • We develop a compilation-aware, execution-based evaluation framework for issue resolu-tion across Java, Kotlin, Dart, and TypeScript mobile app projects and release MobileDev-Bench with task instances, an evaluation har-ness, and containerized environments 1.• We provide a comprehensive evaluation of four frontier code-capable LLMs, showing that resolution rates drop to 3.39%–5.21% and that fault localization emerges as the primary bottleneck for mobile app issue resolution. 

2 Related Work 

Early benchmarks such as HumanEval (Chen et al., 2021) and MBPP (Austin et al., 2021) established a foundation for assessing the programming ca-pabilities of LLMs. These benchmarks primarily focused on small, self-contained problems: 95.8% target Python and over 83% evaluate function- or statement-level tasks (Cao et al., 2024), overlook-ing the repository-level complexity of real-world software development.

SWE-bench (Jimenez et al., 2024a) introduced repository-scale evaluation with 2,294 instances derived from real GitHub issues in Python projects, requiring models to generate patches validated through test execution. Subsequent work extended this direction to additional languages. Multi-SWE-bench (Zan et al., 2025) spans seven programming languages with 1,632 instances, while SWE-bench Java (Zan et al., 2024b) provides 91 Java-specific tasks. SWE-PolyBench (Rashid et al., 2025) fur-ther shows that realistic repair tasks often require edits across multiple files. Despite this progress, ex-isting benchmarks primarily focus on library-style repositories and do not capture platform-specific app development environments. Mobile app development introduces additional challenges not represented in current benchmarks, including framework-specific architectures, com-plex build systems, and coordination across hetero-geneous artifacts such as source code, resources, and configuration files. Mobile app development spans Android, iOS, and cross-platform frame-works such as React Native (Facebook, 2015) and Flutter (Google, 2018). Prior research in mobile apps has focused on analysis and testing tools, such as FlowDroid (Arzt et al., 2014), CarCompat (Fako-rede and Farooq, 2025), and DroidBot (Li et al., 2017), while benchmarks for evaluating AI-assisted issue resolution remain largely unexplored .MobileDev-Bench addresses this gap by intro-ducing a benchmark of 384 issue-resolution tasks from 18 production mobile repositories spanning Android (Java/Kotlin), React-Native, and Flut-ter. By focusing on mobile apps and their het-erogeneous project artifacts and build environ-ments, MobileDev-Bench complements existing repository-level benchmarks and broadens evalua-tion for AI-assisted software engineering systems. 

3 MobileDev-Bench Construction 

MobileDev-Bench is constructed from real issue-pull request (PR) pairs in open-source mobile app repositories. The construction process follows a five-phase pipeline as shown in Figure 1, designed to ensure that each instance corresponds to a real software engineering task and supports reliable execution-based validation. 

3.1 Phase 1: App Selection 

We begin by constructing a pool of candidate mo-bile app repositories spanning three major frame-works: Android Native (Java/Kotlin), React-Native (TypeScript/JavaScript), and Flutter (Dart). Repositories are collected from F-Droid 2 , a cata-log of free and open-source Android apps, and from GitHub searches filtered by mobile framework tags. We prioritize production apps distributed through app stores, as they typically represent mature code-bases with active development and testing practices. This process yields an initial pool of 433 candidate repositories. To ensure quality and benchmark viability, we filter repositories based on community adoption (≥400 GitHub stars), recent maintenance, an open-source license, a GitHub-hosted issue tracker, and verified buildability in a clean environment. The detailed filter criteria are provided in Appendix A. 

3.2 Phase 2: PR Collection and Filtering 

From the selected repositories, we retrieve merged pull requests (PR) using the GitHub API, collecting metadata such as issue descriptions, pull request text, commit history, and patch information. To ensure stable repository states, we only consider PRs merged into the repository’s default branch.

This ensures that each task has a clear problem description derived from the issue report. (ii) PRs must modify or introduce test files so that the correctness of the fix can be verified through automated execution. 

(iii) For each retained PR, we extract two patches: a test patch containing the test changes and a fix patch containing the implementation changes. These patches are anchored to the PR’s base commit to enable controlled replay of the de-velopment history. Thus, each candidate instance contains four components: (i) the base commit, 

(ii) the issue description, (iii) the test patch, and

To ensure reproducibility, we automatically construct build environments by analyzing repository meta-data and generating framework-specific Docker-files. 4.1 Benchmark Statistics 

MobileDev-Bench comprises 384 manually veri-fied instances drawn from 18 production mobile repositories spanning three ecosystems: Flutter (Dart), Android Native (Java/Kotlin), and React-Native (TypeScript). Per-repository statistics are reported in Table 3. Repository sizes range from 337 to 10,944 source files (440K lines on aver-age), reflecting the diversity of mature open-source codebases included in the benchmark. Instance counts vary from 1 to 95 per repository, with the four largest contributors being thunderbird-android (95), wordpress-android (73), element-x-android (63), and zulip-flutter (51), together ac-counting for 73.4% of the dataset. 4.2 Benchmark Features 

MobileDev-Bench captures characteristics of mo-bile app development that are absent from existing repository-level benchmarks. In particular, several fixes span multiple artifact types and may involve more than one language within a single patch. 

Multi-Artifact Fix Patterns. Mobile app fixes often require multi-artifact fixes. We categorize modified files in each patch into the eight artifact types defined in Table 4. Source files appear in nearly all instances (99.5%). However, many fixes also require non-source changes: 20.6% modify resources, 17.2% modify localization files, 19.5% modify build files, and 2.9% modify manifest files. As summarized in Table 5, 35.7% of instances mod-ify two or more artifact types and 17.7% modify three or more, demonstrating that cross-artifact co-ordination is a routine property of mobile app fixes.

Intra-Patch Language Diversity. Fix patches also frequently span multiple languages. We iden-tify cross-language instances by examining the set of source-language file extensions modified within each fix patch. In total, 21.9% of instances modify source files written in more than one lan-guage. The most common pattern is Kotlin–Java co-modification, representing 20.6% of all instances and 26.5% of Kotlin-based fixes. This reflects the ongoing migration of Android projects from Java to Kotlin, with co-modification rates vary-ing widely across repositories depending on their stage of migration: projects retaining residual Java code exhibit rates above 40%, while repositories that have fully transitioned to Kotlin show none. A second cross-language pattern occurs in React-Native repositories, where TypeScript–JavaScript co-modification appears in 40.0% of React-Native instances, reflecting gradual TypeScript adoption in existing JavaScript codebases. 

4.3 Benchmark Comparison 

Table 1 situates MobileDev-Bench among exist-ing issue-resolution benchmarks across four di-mensions: domain coverage, verification, difficulty stratification, and patch complexity. Most prior benchmarks focus on library-style repositories, of-ten in a single language or a small set of languages. Their gold patches are typically small, averaging between 1.2 and 4.9 modified files and between 14 and 163 changed lines. In contrast, MobileDev-Bench targets production mobile apps and exhibits substantially larger fixes, averaging 12.5 modified files and 324.9 changed lines. 

Patch Complexity. Figure 3 compares per-repository median patch size across five bench-marks using three metrics: files modified, hunks, and changed lines. SWE-PolyBench, SWE-bench Multimodal, and Multi-SWE-bench occupy a com-pact low-to-moderate range, with median file counts of 2.6, 3.6, and 3.0, and median changed lines of 54, 128, and 73, respectively. Rust-SWE-bench is an outlier on the files axis (median 10.4 files) owing to the broad cross-module scope of typical Rust fixes, yet its median changed lines (92) remain well below those of MobileDev-Bench. MobileDev-Bench presents the largest median hunk count (18.6) and changed line count (264.8) across all five benchmarks. Compared with SWE-PolyBench, the closest match on repository count, MobileDev-Bench fixes are 3.0 × larger in files, 2.6 × in hunks, and 4.9 × in changed lines. Relative to Multi-SWE-bench, the ratios are 2.6 ×, 2.4 ×,and 3.7 ×, whereas relative to SWE-bench Multi-modal, they are 2.2 ×, 2.3 ×, and 2.1 ×. While Rust-SWE-bench has a slightly higher median file count than MobileDev-Bench (10.4 vs. 7.8), MobileDev-Bench exceeds it by 1.8 × in hunks and 2.9 × in changed lines, indicating that mobile app fixes in-volve more densely modified changes rather than touching more files. This pattern reflects the coor-dinated nature of mobile app fixes, where a single logical change often requires updates across source code, resources, and build configuration files. 

Design Dimensions. Beyond patch size, MobileDev-Bench differs from prior benchmarks in several design aspects. First, it targets production mobile apps spanning three frameworks (Android Native, React Native, and Flutter) and four pro-gramming languages. Second, it incorporates both human verification and difficulty stratification, en-abling controlled evaluation across task complexity levels. Third, it explicitly captures multi-artifact fixes, where resolving an issue requires coordinated changes across heterogeneous project components rather than isolated source edits.

5 Evaluation 

5.1 Evaluation Framework 

We adopt the localization–repair pipeline from Agentless (Xia et al., 2024). Given an issue descrip-tion and repository context, the model first iden-tifies potentially faulty files and functions (fault localization), then generates a patch as a unified diff. Generated patches are applied to the base com-mit and validated by executing the repository test suite within our Docker-based environment. To support the languages in MobileDev-Bench, we replace the Python-specific parsing in Agent-less with tree-sitter–based parsing, enabling robust structural analysis for Java, Kotlin, TypeScript, and Dart. This allows consistent localization and patch generation across mobile frameworks. 

Models. We evaluate four models (proprietary and open-weight): Claude Sonnet 4.5, GPT-5.2, Gem-ini 2.5 Flash, and Qwen-3-Coder. 

Metrics. We report two complementary metrics. 

(i) Resolution Rate: the percentage of tasks where the generated patch passes the full test suite. (ii) Lo-calization Metrics: file-level Precision, Recall, and F1, measuring how accurately models identify the ground-truth modified files. We provide formal definitions of metrics in Appendix D.

6 Conclusion 

We introduced MobileDev-Bench, a benchmark for evaluating LLM-based program repair on mobile app development tasks. It contains 384 verified instances from 18 production repositories across Android Native, React Native, and Flutter, captur-ing key aspects of mobile development such as multi-artifact coordination and framework-driven execution. Evaluation of four frontier LLMs shows low resolution rates, with fault localization as the primary bottleneck. File-level recall remains lim-ited and declines further for complex tasks, par-ticularly multi-file and multi-artifact fixes. These results highlight the need for improved mobile app framework-level understanding to handle multi-file dependency reasoning and artifact-aware localiza-tion. MobileDev-Bench provides a realistic bench-mark and baseline results for future research in AI-assisted mobile app development. 7 Limitations 

MobileDev-Bench focuses on Android Native and cross-platform apps, including React-Native and Flutter repositories, and does not include iOS-native projects written in Swift or Objective-C, lim-iting coverage of the full mobile ecosystem. Eval-uation relies on execution-based validation using repository test suites, which may not fully capture issues with insufficient or incomplete test coverage. Additionally, the benchmark is constructed from issue–pull request pairs in open-source repositories, which may not represent all mobile development ac-tivities, such as UI design iterations or refactoring tasks without associated issues. Finally, our eval-uation uses a single-pass repair pipeline, whereas app development often involves iterative debugging and interactive workflows.

D.1 Evaluation Framework 

We evaluate MobileDev-Bench using the localiza-tion and repair pipeline from Agentless (Xia et al., 2024), a state-of-the-art autonomous code repair framework that has demonstrated strong perfor-mance on Python-centric benchmarks. Agentless decomposes the repair task into two sequential phases: 1. Fault Localization: Given an issue descrip-tion and repository context, the model identi-fies potentially buggy files and functions. This phase employs retrieval-augmented genera-tion to narrow the search space from the en-tire codebase to a manageable set of candidate locations. 2. Patch Generation: The model generates a patch as a unified diff format targeting the localized code elements. The generated patch must be syntactically valid and applicable to the base commit. This two-phase decomposition enables us to sep-arately measure localization capability (can the model identify where to fix?) and repair capability (can the model generate the correct fix?), providing diagnostic insights into failure modes. We validate generated patches using a Docker-based test harness. For each task instance, we: (1) clone the repository at the base commit, (2) apply the model-generated patch, (3) execute the repository test suite in an isolated Docker container, and (4) compare test outcomes against the ground truth. A patch is considered successful if it passes all tests that pass with the developer’s ground truth patch, including previously failing tests that should now pass. This evaluation approach mirrors the work-flow used in SWE-bench (Jimenez et al., 2024a) and Multi-SWE-bench (Zan et al., 2025), en-abling direct comparison with existing benchmarks while avoiding the compounding errors and non-determinism of full agentic systems that include web browsing, terminal access, and multi-turn in-teractions. 

D.2 Adaptation for Mobile Development Languages 

The original Agentless implementation relies on Python’s Abstract Syntax Tree (AST) module for parsing and extracting code structures. To support the diverse programming languages in MobileDev-Bench (Java, Kotlin, TypeScript, and Dart), we incorporate tree-sitter 4, a parsing library that pro-vides unified syntax tree generation across multiple languages. Similar to the MagentLess 5 adaptation for Multi-SWE-bench, we replace Python-specific parsing calls with tree-sitter-based parsing while maintain-ing the same localization and patch generation al-gorithms. This ensures that: • Code structure extraction (classes, methods, functions) works consistently across Android (Java/Kotlin), Flutter (Dart), and React Native (TypeScript/JavaScript) codebases. • Function-level localization operates with the same granularity across all four languages. • The core Agentless algorithm remains un-changed, ensuring fair comparison with re-sults on Python-based benchmarks. For non-code artifacts (XML layouts, JSON i18n files, Gradle build scripts), which lack traditional function boundaries, we perform file-level localiza-tion only.

D.5 Evaluation Infrastructure 

Each task instance is executed in an isolated Docker container with repository-specific dependencies in-stalled from the original project’s build configura-tion (Gradle for Android, pub for Flutter, npm/yarn for React Native). Test execution timeouts are set to 30 minutes per task to accommodate large test suites in produc-tion repositories. Tasks that exceed this limit are marked as timeouts and counted as failures.

This section presents comprehensive evaluation results for four frontier language models on MobileDev-Bench. We analyze resolution rates, fault localization performance, and the relationship between task characteristics and repair success. 

E.1 Resolution Rate Analysis 

Table 12 shows resolution rates ranging from 3.39% to 5.21% across the four models, with Qwen3 Coder achieving the highest rate (20/384 tasks). Tasks in MobileDev-Bench require modifying an average of 12.5 files (median: 5), reflecting the multi-file nature of mobile development. 

Multi-File Modifications Show Lower Resolu-tion Rates. 

Figure 8 reveals a dramatic degradation in res-olution rates as the number of required file modi-fications increases. Single-file tasks achieve 12.3– 15.1% resolution rates across models, substantially higher than the 3.4–5.2% overall performance (a 2.5–4.5x difference). However, this success rapidly diminishes with task complexity. Tasks requiring 2–3 file modifications show a steep drop to 2.5–5.0% resolution, less than half the single-file rate. Mid-complexity tasks requiring 4–10 files prove nearly unsolvable, with resolution rates approaching zero (0.0–4.1%). This difficulty cliff highlights models’ fundamental limitations in coordinating changes across multiple files. Interestingly, tasks requiring 11+ files show a modest recovery (0.0–4.2%), with Qwen3 Coder resolving 5 of 119 such instances (4.2%). How-ever, detailed analysis (Section E.3) reveals this recovery stems from repository-specific architec-tural patterns rather than improved multi-file rea-soning capabilities. Specifically, 83.3% of re-solved 11+ file instances come from a single reposi-tory ( element-hq/element-x-android ), suggest-ing that certain codebases facilitate large-scale au-tomated changes through consistent structure and comprehensive test coverage. Across all models, only 39 unique instances were successfully resolved, accumulating to 64 total res-olutions due to overlap. E.4 Fault Localization Performance 

Table 14 shows that fault localization is the pri-mary bottleneck. Overall retrieval performance ranges from 13.8–19.5% recall, 42.1–59.0% preci-sion, and 17.4–24.3% F1, with Claude Sonnet 4.5 achieving the highest scores. The large precision-recall gap (2.5–4.3x) indicates conservative file selection: models identify relevant files but miss many necessary modifications. 

Recall Degradation by File Count. Recall de-grades severely with file count: from 34–53% on single-file tasks to 2–3% on 11+ file tasks, rep-resenting a 32–51 percentage point decline. For tasks requiring 6–10 files, models identify fewer than 10% of necessary files. This catastrophic degradation reflects models’ inability to trace cross-layer dependencies in mobile architectures, where changes cascade across network, data, presentation, view, and test layers. 

Precision Trends. Precision increases with file count (34–53% for single-file tasks to 43–68% for 6–10 file tasks), indicating conservative behavior on complex changes. Localization errors stem pri-marily from missed files rather than false positives: models identify a core subset correctly but fail to expand to all necessary modifications. 

Localization Patterns. Three distinct patterns emerge: • Perfect localization (5–8% of tasks): Almost exclusively single-file changes with strong se-mantic signals in issue descriptions. • Zero localization (46–59% of tasks): Models edit entirely incorrect files, common when ground truth modifies non-code artifacts or peripheral modules. 162 tasks (42.2%) show zero recall across all models. • Partial localization (36–47% of tasks): Mod-els find some relevant files but miss cascading modifications across architectural layers.

F Discussion 

Our evaluation reveals resolution rates of 3.39– 5.21% on MobileDev-Bench. Three architectural properties of mobile development explain this low performance. First, multi-artifact coordination :models achieve 1.36x success on single-artifact tasks but only 0.36x on multi-artifact tasks re-quiring synchronization across source code, re-sources, build configurations, and manifests. Sec-ond, framework-mediated execution : platform life-cycles obscure control flow from static analysis. Third, cross-module dependencies : recall drops from 34–53% on single-file tasks to 7–10% on multi-file tasks (6–10 files), and further to 2–3% on large-scale tasks (11+ files). Localization is the primary bottleneck. File-level recall of 13.8–19.5% imposes a hard ceiling on res-olution rates. For mobile development, the chal-lenge lies not in generating correct patches but in identifying which files need modification. This finding suggests research should prioritize multi-file dependency tracing, artifact-aware retrieval, and framework-specific reasoning over patch gen-eration improvements. The non-monotonic relationship between file count and resolution rate provides insight: single-file tasks achieve 12 to 15% resolution, mid-complexity tasks (4 to 10 files) drop to 0 to 2%, while 11+ file tasks show modest recovery (0.0 to 4.2%). This pattern suggests models struggle with 

architectural complexity (interconnected changes across heterogeneous components) rather than raw file count. Repository-specific phenomena hint that certain architectural patterns may facilitate auto-mated repair. G Future Work 

Our findings suggest three high-priority research directions. First, given that localization is the primary bottleneck, techniques combining seman-tic search with static analysis for multi-file de-pendency tracing warrant immediate investigation. Second, repository-specific success patterns hint that certain architectural characteristics aid auto-mated repair; systematically analyzing what makes codebases "LLM-friendly" could inform both tool design and best practices. Third, mobile issues of-ten include screenshots and crash logs; evaluating multi-modal models’ ability to ground repairs in visual artifacts may improve UI-related bug local-ization. Extending to iOS, developing framework-aware representations encoding lifecycle semantics, and testing iterative agentic workflows provide ad-ditional promising directions. 

H Potential Societal Impact 

MobileDev-Bench provides an execution-validated benchmark for mobile-specific program repair, ex-posing failure modes invisible to library-centric evaluations, guiding practitioners on where LLM assistants are and are not reliable in mobile CI pipelines, and highlighting the need for human oversight when deploying model-generated patches in accessibility-, localisation-, or security-sensitive components.
