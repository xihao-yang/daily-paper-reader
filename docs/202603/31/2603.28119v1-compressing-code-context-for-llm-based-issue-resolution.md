## 摘要
大型语言模型（LLMs）现已能够解决现实世界中的 GitHub 问题。然而，当前方法往往过度近似代码上下文，因此面临两个相互叠加的问题：处理大规模输入的高昂成本，以及由于噪声充斥上下文窗口而导致模型难以聚焦于修复信号的低效性。现有的压缩技术无法有效解决这一矛盾：通用压缩器会损害代码的语义完整性，而代码专用工具则缺乏对代码结构和任务上下文的理解，难以保留关键的补丁成分。为了解决这一问题，我们提出了一个由两个组成部分构成的新框架。首先是 Oracle-guided Code Distillation（OCD），一种上下文蒸馏算法，结合遗传搜索和增量调试方法，系统地将代码上下文缩减至最小的充分子序列——仅保留实现成功修复所需的关键成分。我们利用这些蒸馏后的数据对 SWEzze 进行微调，这是一种轻量化模型，能够在推理时学习压缩代码上下文，过滤噪声并减轻干扰，同时保留修复所需的关键信息。在 SWE-bench Verified 基准上针对三种前沿 LLM 的评估结果显示，SWEzze 在不同模型下保持约 6 倍的稳定压缩率，相比未压缩设置可减少 51.8%–71.3% 的总 token 成本，将问题解决率提升 5.0%–9.2%，并在效果、压缩率与延迟之间实现了优于最先进上下文压缩基线的整体平衡。

## Abstract
Large Language Models (LLMs) are now capable of resolving real-world GitHub issues. However, current approaches overapproximate the code context and suffer from two compounding problems: the prohibitive cost of processing massive inputs, and low effectiveness as noise floods the context window and distracts the model from the bug-fixing signal. Existing compression techniques fail to resolve this tension: generic compressors compromise the semantic integrity of code, while code-specific tools lack awareness of code structure and task context to preserve essential patch ingredients. To address this, we propose a novel framework consisting of two components. First, Oracle-guided Code Distillation (OCD), a context distillation algorithm that combines genetic search and delta debugging to systematically reduce code contexts to their minimal sufficient subsequence - retaining only the ingredients required for a successful fix. We use this distilled data to fine-tune SWEzze, a lightweight model that learns to compress code context at inference time, filtering noise and combating distraction while preserving fix ingredients. Evaluated on SWE-bench Verified across three frontier LLMs, SWEzze maintains a stable compression rate of about 6 times across models, reduces the total token budget by 51.8%-71.3% relative to the uncompressed setting, improves issue resolution rates by 5.0%-9.2%, and delivers the best overall balance among effectiveness, compression ratio, and latency compared with state-of-the-art context compression baselines.

- **研究动机**  
  当前大型语言模型（LLM）在软件工程任务中已能解决真实 GitHub 上的问题（issue fixing），但在“构建输入上下文”阶段存在固有瓶颈：

- **挑战**  
  通用文本压缩技术会破坏代码语义；代码专用压缩或剪枝工具又缺乏上下文理解，易丢失补丁所需的“修复成分（fix ingredients）”。

- **研究目标**  
  寻找一种**能够保留修复关键语义信息、同时显著降低输入长度**的上下文压缩策略，从而平衡“修复效果”与“计算效率”。

## 二、方法论与技术框架

### 1. 整体框架

论文提出两阶段框架：

1. **Oracle-guided Code Distillation (OCD)** — *训练阶段的数据蒸馏器*  
   - 利用“预言机（oracle）”搜索，找到每个问题的**最小充分上下文（Minimal Sufficient Context, MSC）**，即能让 LLM 产出正确补丁的最小代码子集。  
   - 结合遗传算法（Genetic Algorithm, GA）和分层差分调试（Hierarchical Delta Debugging, HDD）来自动探索这一最小上下文。  

2. **SWEzze 模型** — *推理阶段的上下文压缩器*  
   - 以 LoRA 方式微调自 Qwen3-Reranker-0.6B，学习在推理时自动过滤冗余代码，只保留 fix ingredients。  
   - 推理时无需预言机，基于得分的贪心选择，动态控制压缩比例。

### 2. 核心思想

- 将上下文压缩视为一个“保留补丁成分的判别学习任务”，而非单纯句法压缩。
- 利用数据级蒸馏生成训练样本：(I, F, C_init, Ĉ)，其中：
  - I：问题描述；
  - F：故障位置；
  - C_init：初始检索上下文；
  - Ĉ：OCD 搜索出的最小充分子集。
- 最小充分条件定义：若移除任何单位代码段后修复失败，则该段是必需的。

### 3. OCD 算法细节

- **搜索空间**  
  对源代码进行三层结构化分解：文件 → 函数 → 语句块。所有候选组合均保持语法合法。  
- **优先度打分 Φ(u)**：  
  综合三类信号：
  1. **Patch overlap**：单位 u 是否位于真实补丁修改文件；
  2. **Test coverage**：失败测试覆盖的行数；
  3. **Symbol overlap**：补丁标识符与单位标识符的词汇匹配。
  组合公式：Φ(u) = wₚ·I + w_c·log(1+|L_cov|) + wₛ·SymScore  
- **两阶段搜索**
  - *Phase I: Genetic Search*  
    随机群体进化以找到能产出通过测试补丁的上下文子集；
  - *Phase II: Hierarchical Delta Debugging*  
    自顶向下删除冗余文件、函数、代码块，实现 1-minimality。  

- **评估准则**  
  若 LLM 基于候选上下文生成的补丁在多数验证运行中通过测试，即视为该上下文“充分”。

### 4. SWEzze 模型设计

- **输入–输出形式**  
  将每个 (issue, fault, segment) 视为分类任务，预测该代码段是否应保留（标签 y ∈ {0,1}）。  
- **训练机制**
  - 采用 LoRA（rank=16, α=32）微调；
  - 采用 role-aware 权重平衡样本不均衡；
  - 使用 bfloat16 混合精度、梯度检查点、AdamW 优化。
- **推理过程**  
  模型为每个分段打分→贪心选取→满足用户设定的 token 压缩预算。

## 三、实验设计

### 1. 数据集与环境

- **训练数据集：SWE-Smith**  
  - 来源：128 个 GitHub 仓库；从中抽取 10K 个 issue；
  - OCD 成功构建的 3,157 个最小上下文样本；
  - 总 segment 数：156,545（其中 13,102 为正样本，相关性密度 8.4%）。

- **测试基准：SWE-bench Verified (500 实例)**  
  - 含实际 issue 描述、修复补丁和回归测试；

### 2. 对比方法（Baselines）

| 方法名称 | 特点 |
|-----------|------|
| **LLMLingua-2** | 通用 token 级压缩器 |
| **LongCodeZip** | 层级式代码压缩（基于困惑度） |
| **SWE-Pruner** | 任务感知剪枝 reranker |
| **No Compression** | 不进行压缩 |
| **No Context** | 仅提供问题描述与故障位置 |

### 3. 下游模型（用于生成补丁）

- **Resolution Rate**：通过全部测试用例的补丁比例；

## 六、主要结论与发现

1. **压缩率稳定**：约 6×，在不同模型间波动小；  
2. **成本大幅下降**：相对未压缩减少 51.8%–71.3% 的 token 开销；  
3. **修复效果提升**：提高 5.0%–9.2% 的问题解决率；  
4. **覆盖性强**：SWEzze 解决的问题实例占所有成功案例 93.8%–99.2%；  
5. **运行时合理**：平均延迟 6.1 秒/实例，比 LongCodeZip 快约 4×；  
6. **失败原因**主要有：  
   - 保留了干扰性片段；  
   - 无法区分语义近似但功能不同的代码段；  
   - 过度压缩导致语义破碎。
